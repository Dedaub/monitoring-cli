# Compound V3 (Comet) — Topics, Selectors, Addresses (Ethereum, Base, Arbitrum, Optimism, Polygon)

**Status:** verified against Ethereum, Base, Arbitrum, Optimism, and Polygon mainnet RPC and the canonical `compound-finance/comet` repo (`deployments/<net>/<market>/roots.json`) on 2026-05-29. Every topic0/selector computed locally with keccak and cross-checked against live `eth_getLogs` and deployed bytecode; every market address verified via `eth_getCode` + live `baseToken()`.
**Scope:** Compound III ("Comet") is the modern isolated-base-asset lending protocol. **Of the seven requested chains, Comet is deployed on five: Ethereum, Base, Arbitrum, Optimism, Polygon PoS. It is NOT deployed on BNB Smart Chain or Avalanche C-Chain** (`eth_getCode` empty on both — confirmed). Comet also lives on Mantle, Scroll, Unichain, Linea, and Ronin (out of scope here). Topics + selectors are chain-agnostic; addresses are network-specific. For the legacy cToken/Comptroller system see `compound/v2.md`.

**Architecture.** Each Comet market is a **single contract per base asset** (e.g. `cUSDCv3`, `cWETHv3`). You supply/withdraw/borrow/repay the base asset *directly on the Comet* — there is no cToken. Collateral assets are held as non-yielding balances; only the base asset earns/pays interest, which accrues **per second** (V2 was per block). Balances can be queried with `balanceOf` (base supply) and `borrowBalanceOf` (base debt) — an account is in exactly one of those two states. Liquidation is **two-step and non-atomic**: a keeper `absorb()`s an underwater account (protocol takes the collateral, emits `AbsorbDebt`/`AbsorbCollateral`), then anyone `buyCollateral()`s that seized collateral from the protocol at a discount. Every market is a **TransparentUpgradeableProxy**; its logic is split between the main **Comet** implementation and an immutable **CometExt** extension delegate (which serves the ERC-20-ish/view surface). A per-chain **Configurator** (also a proxy) stores each market's parameters and, via **CometProxyAdmin**, deploys new Comet implementations and upgrades the market proxies.

| `0x4232cd63` | `supplyTo(address dst, address asset, uint256 amount)` | Supply on behalf of `dst`. |

Patterns to internalize:

1. **Addresses recur across chains by coincidence.** `0x9c4ec768…58bf` is the Base cUSDbCv3 AND the Arbitrum cUSDCv3 proxy. `0xbdE8F31D…372d` is the Base CometProxyAdmin AND the Arbitrum Bulker. `0x45939657…e581` is the Base Configurator AND the Polygon CometRewards. `0x18281dfC…468A` is the bridgeReceiver on both Base and Polygon. These are deterministic-deployer / nonce collisions — **always condition on `(chainId, address)`, never address alone.**
2. **One Configurator + one CometProxyAdmin + one CometRewards per chain**, shared by all that chain's markets. Each market is a separate Comet proxy.
3. **L2/sidechain markets are governed cross-chain.** Ethereum governance sends messages → per-chain `bridgeReceiver` (via OP messenger / Arbitrum retryable / Polygon FxPortal) → executes the config/upgrade locally.
4. **Base-token names lie.** Polygon `cUSDCv3` is bridged **USDC.e**; Arbitrum has both native `cUSDCv3` and bridged `cUSDCev3`; Base has native `cUSDCv3` and legacy `cUSDbCv3`. Always read `baseToken()`.

## 9. Proxies (old and new)

Comet uses **OpenZeppelin TransparentUpgradeableProxy** (EIP-1967) for both markets and the Configurator — a clean change from V2's bespoke Unitroller/Delegator pattern. There is also a **second, non-EIP-1967 delegation layer inside Comet**: the immutable `extensionDelegate` (CometExt).

| Layer | Pattern | How to read it |
|-------|---------|----------------|
| **Comet market** | EIP-1967 TransparentUpgradeableProxy. | Impl: storage slot `0x360894…bbc`. Admin: slot `0xb53127…6103` → the chain's CometProxyAdmin. |
| **Configurator** | EIP-1967 TransparentUpgradeableProxy (same CometProxyAdmin). | Same slots. |
| **CometExt (extension delegate)** | NOT a proxy slot — an immutable address baked into the Comet implementation; Comet's `fallback` delegatecalls it for the ERC-20/view surface (`name`, `symbol`, `balanceOf`, `collateralBalanceOf`, `allow`, …). | `extensionDelegate()` → `0x44ff241d`. |
| **CometProxyAdmin** | Plain OZ `ProxyAdmin` (owner = the chain's Timelock/bridgeReceiver). | `getProxyImplementation(proxy)`, `getProxyAdmin(proxy)`. |

**Live implementations rotate.** Reading the EIP-1967 impl slot on 2026-05-29 (these change on every governance `CometDeployed`):

## 10. Detection invariants & gotchas

1. **Borrowing = `Withdraw` of the base asset.** There is no separate Borrow event. An account `withdraw`-ing base beyond its supply goes negative (`userBasic.principal < 0`) — that's a borrow. Repaying = `Supply` of base. Distinguish supply-vs-debt via `principal` sign, not event name.
2. **Liquidation is two events, two txs, two actors.** `absorb()` (→ `AbsorbDebt` + `AbsorbCollateral`, by a keeper) seizes collateral into the protocol; `buyCollateral()` (→ `BuyCollateral`, possibly later, by anyone) sells it. They are NOT atomic and usually NOT the same address. Don't pair them by tx.
3. **`balanceOf` ≠ collateral.** `balanceOf` is base **supply** only; pure-collateral suppliers read 0. Use `collateralBalanceOf(account, asset)` and `borrowBalanceOf(account)`.
4. **View/ERC-20 calls hit CometExt, core actions hit Comet.** A selector absent from the main Comet impl bytecode (e.g. `collateralBalanceOf` `0x5c2549ee`, `balanceOf`, `name`) is served by the `extensionDelegate`. Probe via the proxy, not the impl.
5. **Per-second interest.** `getSupplyRate`/`getBorrowRate` are per-second (V2 was per-block). Don't reuse V2 annualization.
6. **`getReserves()` selector collides with Uniswap V2** (`0x0902f1ac`) but returns a single `int` — different ABI.
7. **Not on BNB or Avalanche.** Any "Compound III"-looking contract there is not Compound. (Compound *is* additionally on Mantle/Scroll/Unichain/Linea/Ronin — out of scope here.)
8. **Base-token name vs reality.** Read `baseToken()`; Polygon `cUSDCv3` is bridged USDC.e, Arbitrum/Base have separate bridged-vs-native markets.
9. **`max uint` semantics.** `supply(base, type(uint).max)` repays the entire debt; `withdraw(base, type(uint).max)` withdraws the entire supply. The on-chain `amount` in the event is the actual resolved amount.
10. **Rewards are a separate contract.** COMP is not emitted by Comet on supply/borrow; accrual is tracked internally and realized via `CometRewards.claim` → `RewardClaimed`. The reward token per market comes from `rewardConfig(comet)` (COMP, but a different address per chain).
11. **Cross-chain addresses collide** (§8.1) — always key on `(chainId, address)`.
