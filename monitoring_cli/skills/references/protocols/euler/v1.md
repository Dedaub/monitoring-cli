# Euler V1 — Topics, Selectors, Addresses (Ethereum only)

**Status:** verified 2026-05-29. Event topic0s and function selectors computed locally with keccak from `euler-xyz/euler-contracts` (`master`) source; addresses from that repo's `addresses/euler-addresses-mainnet.json`; code sizes and historical logs verified against `https://ethereum-rpc.publicnode.com`.
**Scope:** Euler V1 was deployed **only on Ethereum mainnet** (chain ID 1). It launched Dec 2021 and was drained on **2023-03-13** (~block 16,817,996) in a $197M donation/self-liquidation exploit. Most funds were returned by the attacker; the protocol was never restarted and governance migrated to **Euler V2** (see [v2.md](v2.md)). These contracts are still on-chain and queryable, but markets are effectively dormant. Useful for historical indexing / post-mortem analysis.

## Architecture — module + dispatch proxy (read this first; it dictates event addresses)

Euler V1 is a **single logical deployment** built from one storage/dispatch contract plus pluggable module-logic contracts, fronted by minimal forwarding proxies:

- **`Euler` main contract** (`0x27182842…`, 907 B): holds **all protocol storage** and the module dispatch table. Not called directly by users.
- **Module logic contracts** (the `modules.*` addresses, 18 KB each): the actual code for Markets, Liquidation, Governance, Exec, EToken, DToken, RiskManager, Swap, SwapHub, IRMs, Installer. Registered by `moduleId` via the Installer. Never called directly — never hold storage.
- **Single-proxy module proxies** (519 B each): exactly one public proxy per single-instance module — `markets`, `liquidation`, `governance`, `exec`, `swap`, `swapHub`, `flashLoan`. A call to a proxy forwards calldata (plus appended `msg.sender`) to the `Euler` main contract, which `delegatecall`s the module logic in **main-contract storage context**.
- **Multi-proxy modules — EToken & DToken**: one proxy is `CREATE`-deployed **per activated market** by `Markets.activateMarket()`. Every eToken proxy shares the single `modules.eToken` logic; every dToken proxy shares `modules.dToken`. eTokens are interest-bearing ERC-20 deposit receipts; dTokens are ERC-20-style debt tokens.
- **Internal modules — RiskManager, IRMs**: invoked internally by other modules via `delegatecall`; **no public proxy**. Do not filter on these addresses for user activity.

### Where each event is emitted (critical for monitoring)

Because module logic runs via `delegatecall` inside the **main `Euler` contract**, protocol events are logged with **`address = 0x27182842…` (the main contract)**, not the proxy. Markets are distinguished by the **indexed `underlying`** token field, not by log address. The exception is the ERC-20 `Transfer`/`Approval` pair: modules re-emit those through the token proxy via `emitViaProxy_Transfer/Approval`, so **`Transfer`/`Approval` log address = the per-market eToken/dToken proxy**.

| Event class | Examples | Log `address` |
|-------------|----------|---------------|
| Protocol / accounting | `Deposit`, `Withdraw`, `Borrow`, `Repay`, `Liquidation`, `EnterMarket`, `ExitMarket`, `MarketActivated`, `AssetStatus`, `Request*`, `Gov*` | **main `Euler` `0x27182842…`** (keyed by indexed `underlying`) |
| ERC-20 token | `Transfer`, `Approval` | **per-market eToken / dToken proxy** |

Verified live: `eth_getLogs` on `0x27182842…` for the `Deposit` topic returns 106 hits in a 30k-block 2022 window; `Liquidation` returns 1. (publicnode getLogs with a topic-only filter and no `address` returns `[]`, so always pin the address.)

## 4. Proxies & upgradeability

- **Not** EIP-1967/Transparent/UUPS. Euler V1 uses a **custom module-dispatch pattern**: minimal forwarding proxies (519 B) → `Euler.dispatch()` (`call`) → `delegatecall` into module logic.
- Upgrades happen by the Installer's `installModules([...])`, which rebinds a `moduleId` to a new logic address — verified on-chain by `InstallerInstallModule` logs. There is no per-proxy implementation slot to read; the live logic address for a module is the latest `moduleImpl` for that `moduleId` in `InstallerInstallModule`.
- eToken/dToken **multi-proxies** are created by `Markets.activateMarket` (`ProxyCreated` event, `moduleId` = eToken/dToken id) — one pair per underlying.
- `upgradeAdmin` / `governorAdmin` are managed by the Installer module (set via `InstallerSetUpgradeAdmin` / `InstallerSetGovernorAdmin`).

## 5. Detection invariants & gotchas

- **All accounting events log under the single main contract `0x27182842…`** — do **not** expect Deposit/Borrow/Liquidation under per-market addresses. Disambiguate markets via the indexed `underlying`.
- **`Transfer`/`Approval` are the only events under eToken/dToken proxy addresses** (re-emitted via `emitViaProxy`). An eToken `Transfer(addr, 0x0, amount)` accompanies a `Withdraw`; `Transfer(0x0, addr, amount)` accompanies a `Deposit`.
- **`Request*` events pair 1:1 with their accounting events** — useful as a cross-check but redundant for balance tracking.
- **`batchDispatch` (`0xeb937aeb`) is the dominant entrypoint** — most user activity arrives as a batch routed through the Exec proxy; decode the inner `items[].data` (each targets a module proxy) to attribute actions.
- **The protocol is dormant post-2023-03-13.** Treat any *new* large flows as anomalous. The exploit abused `donateToReserves` (`0x36f022aa`) + self-liquidation; `donateToReserves` calls are a historical red flag.
- Sub-accounts: an account's 256 sub-accounts are `owner XOR subAccountId` in the low byte; events report the resolved sub-account address.
