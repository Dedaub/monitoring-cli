# Ekubo Protocol V3 (EVM) — Topics, Selectors, Addresses (Ethereum, Base, BNB, Arbitrum, Optimism, Polygon — NOT Avalanche)

**Status:** verified against live RPC on every listed chain and the canonical `EkuboProtocol/evm-contracts` repo (release **v3.1.1**, 2026-05-03) on 2026-06-05. All 5 Core event `topic0`s confirmed against live Ethereum `Core` logs; every contract confirmed via `eth_getCode`; the 116-byte anonymous swap log decoded from a live sample.
**Scope:** Ekubo Protocol's **V3** EVM deployment. Topics + selectors are chain-agnostic; addresses are network-specific but **byte-identical across all chains** (CREATE2 vanity). For the deprecated **V2** (Ethereum + Sepolia only, different event layout), see [`v2.md`](./v2.md).

Ekubo Protocol is a **singleton, ownerless, immutable** concentrated-liquidity AMM by Moody Salem (the former Uniswap V3 lead). It is the EVM port of Ekubo-on-Starknet and is architecturally closest to **Uniswap V4** (one `Core` contract holds every pool's state and all tokens; pools are storage, not contracts). Six mechanics to internalize before indexing:

1. **No `Swap` event — swaps emit an ANONYMOUS `log0` (zero topics).** The hottest path (`Core.swap_6269342730`) emits `log0(ptr, 116)` — 116 bytes of data with **no `topic0`** (Core.sol:846). On Ethereum this is ~97% of all `Core` logs (4541 of 4701 in a 2 500-block sample). You **cannot** filter swaps by `topic0`; you detect them by `address == Core AND topics == [] AND data.length == 116`, then decode the data (§1.2). Extensions (Oracle/TWAMM/BoostedFees) emit their own compact anonymous logs too.
2. **Flash accounting via `lock()`.** All state-changing ops happen inside a `lock()` callback (`ILocker.locked_6416899205(uint256 id)`); debts must net to zero before it returns. Tokens move via `startPayments`/`completePayments`/`withdraw`. There are no per-swap ERC-20 `Transfer`s from the pool — read the swap log's `balanceUpdate`.
3. **Mined zero/low selectors.** `Core.swap_6269342730()` and the locker callback `locked_6416899205(uint256)` both have selector **`0x00000000`** (mined function names, for gas). `forwarded_2374103877(bytes32)` = `0x00000001`. Calldata to `Core` beginning `0x00000000` is a swap.
4. **Extensions, not hooks.** A pool's behaviour is attached via an **extension** address embedded in the `PoolConfig`. The extension's **high byte** (`uint8(uint160(extension) >> 152)`) encodes which call points fire (§7) — Ekubo's analogue of V4 hook-flag bits, but in the HIGH byte of the extension address. The same token pair with a different extension/config is a different pool with a different `PoolId`.
5. **Native token is `address(0)`, first-class.** No WETH wrapping needed. `token0 < token1` is enforced and `address(0)` sorts first.
6. **Three pool types in one `config`:** concentrated, stableswap, full-range — distinguished by bits inside the `PoolConfig` (§2.0). `fee` is a `uint64` fraction of 2⁶⁴ (not a tier).

**`PoolKey` / `PoolId`:** `PoolKey = (address token0, address token1, bytes32 config)`. `PoolId = keccak256(abi.encode(token0, token1, config))` — a 32-byte hash over the 96-byte key (no init-code hash, no CREATE2 pool address; a pool is created by `initializePool`, a state write). Index by `(chainId, poolId)`.

> **Chain availability is uneven (§4 verified).** `Core`, `MEVCapture`, `Oracle`, `MEVCaptureRouter`, `Incentives`, `BoostedFees`, `ManualPoolBooster` + data-fetchers are on **Ethereum, Base, BNB, Arbitrum, Optimism, Polygon** (and MegaETH/Monad/Ink/Unichain). **TWAMM, the `Positions`/`Orders` NFT managers, `Auctions`, and `TWAMMDataFetcher` are deployed ONLY on Ethereum, Base, Arbitrum** (of the requested set). **Avalanche has NO Ekubo deployment at all.** On BNB/OP/Polygon there is no first-party LP-NFT manager (LP via `Core` directly) and no DCA/TWAMM.

> **Deployed ≠ active.** V3 requires the **EIP-7939 (`CLZ`) opcode** live on the chain to function (§3). As of the v3.1.1 release (≈2026-05-15) **OP-Stack chains (Base, Optimism, Ink, Unichain) had not yet shipped EIP-7939**, so their Ekubo deployments are present-but-dormant until the chain activates it. Confirmed empirically (2026-06-05): **Ethereum carries essentially all current V3 volume** (~5 400 swaps / 3 000 blocks); Base, BNB, Arbitrum, Optimism, Polygon `Core` returned **0 logs** in recent sample windows. Treat the §4 `✓` as "bytecode deployed," not "actively trading" — anchor live monitoring on Ethereum today, and re-probe the others as CLZ rolls out.

## 2. Function signatures (chain-agnostic)

Selectors are `keccak256(canonical signature)[0:4]` with all user-defined value types resolved (`SqrtRatio`→`uint96`; `PoolId`/`PositionId`/`PoolConfig`/`SwapParameters`/`PoolBalanceUpdate`/`PoolState`/`Locker`/`OrderConfig`→`bytes32`; `PoolKey`/`OrderKey`/`AuctionKey`/`DropKey`→`(address,address,bytes32)`).

### 2.0 `PoolConfig` (bytes32) bit layout — needed to read any PoolKey

| Bits | Field | Notes |
|------|-------|-------|
| `[96:256]` (high 160) | `extension` (address) | `address(0)` = base pool, no extension. High byte = call points (§7). |
| `[32:96]` (uint64) | `fee` | Fraction of 2⁶⁴ (e.g. 0.3% ≈ `0.003 · 2⁶⁴`). **Not** a discrete tier. |
| bit `31` (`0x80000000`) | pool-type flag | set ⇒ **concentrated**; clear ⇒ **stableswap**. |
| `[0:31]` (low 31) | `concentratedTickSpacing` (concentrated) | For stableswap these bits re-encode amplification/center-tick/width instead. |

### 2.1 FlashAccountant (inherited by `Core`; entrypoint for ALL pool ops)

## 3. Deployment model — CREATE2 vanity, identical address every chain

Every V3 contract is deployed via **CREATE2** with a fixed salt, so each contract has the **same address on every chain** and **byte-identical runtime** (verified §4). Two salts are used:

- `DEPLOYMENT_SALT = 0x28f4114b40904ad1cfbb42175a55ad64187c1b299773bd6318baa292375cf0dd` (all contracts except the router)
- `MEV_CAPTURE_ROUTER_DEPLOYMENT_SALT = 0x38e2e731c4e6738213b17b239fe56c423f6bdc5b5969897c260d464c35a63982`

Leading-zero/patterned addresses (`Core 0x0000…d701`, `MEVCapture 0x5555…41Be`, `MEVCaptureRouter 0xd26f2000…ce00`) are mined for cheap calldata. Toolchain: solc `0.8.33`, `via_ir`, `evm_version = 'osaka'` — **EIP-7939 (`CLZ`, count-leading-zeros opcode; shipped in Ethereum's Fusaka/Osaka hardfork, ~2025-12) support on the target chain is required for the contracts to function**, which is why V3 only targets Osaka-ready chains and is absent on Avalanche. The v3.1.1 release notes that **as of ≈2026-05-15 OP-Stack chains lacked EIP-7939**, so contracts deployed on Base/Optimism/Ink/Unichain "will become functional once implemented." Deployment (bytecode present) therefore precedes functionality on those chains.

Three patterns to internalize:

1. **Addresses are chain-agnostic; availability is not.** Every contract that exists on a chain has the *same* address as on every other chain (CREATE2). So a single allowlist works — but you must gate by **which** contracts a chain actually has (`Core` everywhere except Avalanche; TWAMM/Positions/Orders only on ETH/Base/Arb of the six).
2. **`Core` is the only address you need for swap/LP/fee detection.** All of init/swap/LP/fee flows through it; `PoolId` discriminates pools. Routers/Positions are just lockers that appear in the log's `[0:20]` locker field.
3. **`PoolId` is chain-local.** Same `PoolKey` math → same `PoolId`, but a `PoolId` only means something paired with its chain's `Core`. Index by `(chainId, poolId)`.

## 6. Proxies

**There are none.** Every Ekubo V3 contract is a plain immutable CREATE2 deployment — `eth_getStorageAt(addr, 0x360894…bbc)` (EIP-1967 impl slot) returns `0x0` (confirmed on Ethereum `Core` and verified non-proxy by design: the docs state "there is no upgrade mechanism for EVM contracts"). There are no per-pool contracts to proxy: a pool is storage inside `Core`, created by `initializePool` (a state write), never deployed — no pool init-code hash, no CREATE2 pool address.

"Stateful vs stateless" (Ekubo's own taxonomy) is **not** an upgrade axis — both are immutable. *Stateful* (`Core`, `Positions`, `Orders`, extensions) hold state and are never redeployed at a new address; *stateless* helpers (data-fetchers, router) may be **re-deployed at a NEW address** in a future release (a new contract, not an upgrade) — so for routers/fetchers maintain a per-version allowlist. Across V3's life, `TWAMM`, `Orders`, and `TWAMMDataFetcher` already have a prior address each (`0xd4f1060cB9C1A13e1d2d20379b8AA2Cf7541ED9b`, `0xfF6Cf0Ca6d7a30A60539ACd4Bb20B3DF84Ea0644`, `0xc07e5B80750247c8b5D7234a9C79DFc58785392B`); the §4 set is the current (v3.1.1) one. **To upgrade behaviour, Ekubo deploys a NEW version (V2 → V3), not a proxy swap** — so "old vs new" lives at the *version* level: see [`v2.md`](./v2.md) for the deprecated Ethereum V2 set (different addresses AND different event signatures).

## 7. Detection invariants & gotchas

1. **Swaps have NO `topic0`.** The single most important fact: a swap emits an anonymous `log0` (§1.2). Topic-only log filters miss 100% of swaps. Detect: `address == Core AND topics == [] AND length(data) == 116`, then `poolId = data[20:52]`, `delta0 = int128(data[52:68])`, `delta1 = int128(data[68:84])`. (publicnode silently returns `[]` for topic-only filters anyway — always filter by `address`.)
2. **Everything funnels through ONE `Core` address per chain.** No pool addresses exist. Discriminate by `PoolId`. `PoolId = keccak256(abi.encode(token0, token1, config))`.
3. **No fields are indexed.** Unlike Uniswap, Ekubo's named events put `poolId` and all args in `data`, not topics. You cannot server-side filter by `poolId` — fetch all `Core` logs for a topic0 and decode `data[0:32]`.
4. **`PositionUpdated` = add + remove + poke.** Sign of `liquidityDelta`; `0` = fee poke. `balanceUpdate` packs `int128 delta0` (high) ‖ `int128 delta1` (low). `sender`/`locker` is the `Positions` manager or a custom locker, never the end user directly.
5. **`swap` and `locked_…` both have selector `0x00000000`; `forwarded_…` is `0x00000001`.** They live on different contracts (Core vs lockers), so no collision. Calldata to `Core` starting `0x00000000` = swap; calldata to a locker starting `0x00000000` = the lock callback. Don't treat a `0x00000000` selector as "fallback/ETH transfer."
6. **Extension call-points are in the extension address's HIGH byte** (`uint8(uint160(extension) >> 152)`), bit map below — Ekubo's analogue of V4 hook flags. `extension == address(0)` ⇒ plain pool, no callbacks. Addresses are mined to land on the required bits.
7. **Native token = `address(0)`.** A pool's `token0` is frequently `0x0000…0000` (native ETH/BNB/POL). `token0 < token1` always; native sorts first. Code assuming an ERC-20 `Transfer` per leg misses native legs (and Ekubo emits no per-swap `Transfer` at all — flash accounting).
8. **Chain capability differs (§5).** On BNB/OP/Polygon: no `Positions`/`Orders`/TWAMM/Auctions. LP shows up only as `Core.PositionUpdated` with `locker` = some integrator, not a first-party NFT mint. On Avalanche: nothing.
9. **18-decimal stablecoins on BNB.** USDC/USDT are 18 decimals on BNB (Binance-Peg). Read `decimals()`; never hard-code `/1e6`.
10. **`Positions`/`Orders`/`Auctions` NFTs are Solady ERC-721.** Mint = `Transfer(from=0)`, burn = `Transfer(to=0)`. Position economics still live in `Core`.
11. **Stateless helpers can move addresses across releases** (§6) — anchor swap/LP detection on `Core`, never on a router/fetcher address. Two router-era addresses already exist for TWAMM/Orders/fetchers.
12. **Fee is a `uint64` fraction of 2⁶⁴, not a tier** (§2.0). To get the human fee: `fee / 2⁶⁴`. Tick spacing & pool type are packed in the same `config` word.
13. **Deployed ≠ active — gate monitoring by chain.** `eth_getCode` ✓ on a chain does not mean swaps are happening: V3 needs EIP-7939/`CLZ` live there. As of 2026-06-05 only **Ethereum** showed Core activity; Base/BNB/Arbitrum/Optimism/Polygon Core returned 0 logs in sample windows (OP-Stack chains await CLZ; the rest are simply low/no volume so far). Don't raise "Ekubo is down on chain X" alerts off zero volume — confirm CLZ availability + volume first.
