# Bancor V3 — Topics, Selectors, Addresses (Ethereum)

**Status:** verified against Ethereum mainnet RPC and the canonical `bancorprotocol/contracts-v3` repo; all 6 non-Ethereum target chains confirmed absent via `eth_getCode`; PoolCollection address confirmed via `collectionByPool(USDC)` on-chain; verified 2026-06-09.
**Scope:** Ethereum (chain 1) only. Bancor V3 ("Omnipool") has no deployment on Base, BNB, Avalanche, Arbitrum, Optimism, or Polygon — see §4. V1 and V2 are fully deprecated. Carbon DeFi is a separate Bancor-affiliated protocol (repo: `bancorprotocol/carbon-contracts`, separate addresses) and is not documented here.

Bancor V3 is the "Omnipool" architecture launched in May 2022. All token pools share a single unified liquidity layer: the **MasterVault** holds all pooled tokens (analogous to Balancer's Vault) while **BancorNetwork** is the single trade entry point for every pair. Liquidity is single-sided — a provider deposits either the base token or BNT, never both. The **BNT pool** (managed by `BNTPool`) is the hub that every trade routes through under the hood.

**IL protection history:** Bancor V3 originally offered IL protection for LPs. In June 2022, following an exploit that targeted the IL protection mechanism, the team suspended the IL protection feature. It has not been re-enabled. Withdrawals still succeed but LPs bear full IL.

**BancorNetwork** (`0xeEF417e1D5CC832e619ae18D2F140De2999dD4fB`) is the main entry point and emits all trade/flash-loan events as a transparent EIP-1967 proxy. Pool-specific deposit/withdrawal events are emitted by **PoolCollection** (for TKN pools) and **BNTPool** (for BNT liquidity). Withdrawal state transitions are tracked by **PendingWithdrawals**.

**PoolCollection versioning:** Bancor has deployed multiple PoolCollection versions (up to Type1V12 — 10 distinct contracts registered on-chain via `PoolCollectionAdded`, not all 12 numbered slots were used) as pool logic evolves. The active version is determined per-pool via `BancorNetwork.collectionByPool(token)`. The current version (Type1V12) is `0xde1B3CcfC45e3F5bff7f43516F2Cd43364D883E4`, confirmed both by the deployment manifest and a live `collectionByPool(USDC)` call.

## 2. Function signatures

Selectors are `keccak256(canonical signature)[0:4]`, derived from `IBancorNetwork.sol` source. The `Token` type is an `address` alias in ABI encoding.

### 2.1 BancorNetwork — state-changing

| Selector | Signature | Notes |
|----------|-----------|-------|
| `0xd3a4acd3` | `tradeBySourceAmount(address sourceToken, address targetToken, uint256 sourceAmount, uint256 minReturnAmount, uint256 deadline, address beneficiary)` → `uint256` | Exact-input swap. Emits `TokensTraded`. |
| `0x45d6602c` | `tradeByTargetAmount(address sourceToken, address targetToken, uint256 targetAmount, uint256 maxSourceAmount, uint256 deadline, address beneficiary)` → `uint256` | Exact-output swap. Emits `TokensTraded`. |
| `0xd895feee` | `tradeBySourceAmountArb(address,address,uint256,uint256,uint256,address)` → `uint256` | Restricted to `_bancorArbitrage` contract; same event. |
| `0x47e7ef24` | `deposit(address pool, uint256 tokenAmount)` → `uint256` | Single-sided deposit; returns pool token amount. |
| `0xb3db428b` | `depositFor(address provider, address pool, uint256 tokenAmount)` → `uint256` | Deposit on behalf of provider. |

## 6. Detection invariants and gotchas

1. **All trades emit from BancorNetwork proxy, not PoolCollection.** Filter `TokensTraded` on `0xeEF417e1D5CC832e619ae18D2F140De2999dD4fB`. There is no separate PoolCollection `TokensTraded` event.

2. **MasterVault is the single custody point.** Every token held by the protocol lives in `0x649765821D9f64198c905eC0B2B037a4a52Bc373`. Any unexpected `FundsWithdrawn` (topic0 `0xc322efa5...`) from MasterVault to an unrecognised `target` is a critical anomaly.

3. **PoolCollection address drifts.** Hardcoding `0xde1B3Cc...` will miss events on a future V13 upgrade. Always call `collectionByPool(token)` to resolve. For log-based monitoring, index `PoolCollectionAdded` from BancorNetwork to track which collection addresses to watch.

4. **`initWithdrawal` not `initWithdraw`.** The correct function name is `initWithdrawal` (selector `0x357a0333`). The three-step withdrawal flow: `initWithdrawal` → wait lock duration (~7 days at launch) → `withdraw`.

5. **`TokensDeposited`/`TokensWithdrawn` naming.** The events in both BNTPool and PoolCollection use the plural `TokensDeposited`/`TokensWithdrawn`, not the singular forms sometimes seen in older docs.

6. **Dual token-deposit path.** Depositing BNT goes through BNTPool (emits BNTPool `TokensDeposited`). Depositing any other token goes through PoolCollection (emits PoolCollection `TokensDeposited`). Both paths call through BancorNetwork's `deposit`/`depositFor`.

7. **IL protection is suspended.** Do not alert on IL protection payout events — the mechanism is dormant. The `ExternalILVault` holds residual balances but is not actively used.

8. **vBNT burn on BNT withdrawal.** Withdrawing BNT liquidity requires burning vBNT (1:1 ratio). Watch vBNT token `Transfer(provider, 0x000...dead/0x000)` in conjunction with `TokensWithdrawn` from BNTPool.

9. **Proxy events come from the proxy address.** `eth_getLogs` on the proxy address (`0xeEF417...`) returns all events including upgrades (`Upgraded(address indexed implementation)`). Do not query the implementation address for events.

10. **Carbon DeFi is entirely separate.** Bancor's newer Carbon DeFi product has different contracts, a different repo (`bancorprotocol/carbon-contracts`), and different addresses. Do not conflate with V3 monitoring.
