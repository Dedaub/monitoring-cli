## Cross-cutting facts worth knowing before you start

- **The headline: Core Pool is BNB-only; Isolated Pools are everywhere else.** The original monolithic Compound-fork pool (`Unitroller` + `vBep20` + Diamond comptroller) exists on **BNB Smart Chain alone**. On Ethereum/Arbitrum/Optimism/Base — and for *every non-core pool on BNB itself* — the lending markets are **Isolated Pools** (a totally different codebase: `PoolRegistry` + beacon-proxy `Comptroller`/`VToken`, with their own event set and bad-debt isolation). The "Core Pool" you see in the Venus UI on the non-BNB chains is an Isolated-Pools `Comptroller`, NOT the Core codebase. Author Core-Pool monitors (vBNB, `VBep20Delegate`, Diamond facets, `DistributedSupplierVenus`) for BNB only.
- **Three proxy/upgrade patterns coexist — getting them wrong is the #1 integration error.** (1) **OpenZeppelin Transparent** EIP-1967 (Isolated `PoolRegistry`/`RewardsDistributor`; periphery oracle stack/Prime/PLP/PSR/XVSBridgeAdmin/VTreasuryV8; Core `Liquidator`/`VBNBAdmin`/`ResilientOracle`) — read the EIP-1967 impl slot. (2) **Beacon proxy** (Isolated `Comptroller`/`VToken` → `ComptrollerBeacon`/`VTokenBeacon`) — read the beacon's `implementation()` (`0x5c60da1b`), never the proxy. (3) **Compound-style delegator** (Core `Unitroller` slot-2 + per-market `VBep20Delegator`; periphery `XVSVaultProxy`/`GovernorBravoDelegator`/`VaiUnitroller`/`VAIVaultProxy`/`VRTConverterProxy`) — the EIP-1967 slot is empty `0x0`; the logic lives in a plain getter. Plus a handful of **immutable** contracts (Core `vBNB`/IRMs; periphery XVS bridged token, the two `XVSProxyOFT*`, `OmnichainProposalSender`/`OmnichainGovernanceExecutor`, VTreasury).
- **The repo `deployments/*.json` `*Impl` fields are STALE.** On 2026-06-08 the live `VTokenBeacon.implementation()` differed from the JSON on every Isolated chain except BNB (e.g. ETH JSON `0xefdf5C…` vs live `0x33be30b3…`), and several periphery impls (VaiUnitroller, ProtocolShareReserve) were upgraded past their recorded artifact. Always resolve logic from the beacon/proxy slot live, never from the JSON.
- **`Upgraded(address)` on an Isolated beacon is the top-severity signal.** A single `Upgraded` (`0xbc7cd75a…`) on `ComptrollerBeacon` or `VTokenBeacon` atomically re-points every pool / every market on that chain at once. Watch the two beacon addresses directly. The Core Pool's analogous lever is a Diamond `DiamondCut` (Venus's **non-standard single-arg** form `DiamondCut(FacetCut[])` `0x2e97860c…`, NOT the EIP-2535 3-arg `0x8faa7087…`).
- **XVS + governance are BNB-canonical with LayerZero bridges.** XVS is minted on BNB and bridged to remote chains via a LayerZero OFT (`XVSProxyOFTSrc` locks on BNB; `XVSProxyOFTDest` mints/burns remotely) — **each chain's XVS address differs (no vanity)**. Only BNB has VAI, VRT, GovernorBravo, the three on-chain-voting Timelocks, `OmnichainProposalSender`, BinanceOracle and PythOracle. On every remote chain there is **no on-chain voting**: `OmnichainGovernanceExecutor` receives a LayerZero message from BNB and queues it into the local Normal/FastTrack/Critical Timelock. `SendToChain` is Venus's custom 4-arg `(uint16,address,bytes32,uint256)` `0xd81fc9b8…`, not stock LayerZero.
- **Topic0 collisions across the Compound-derived VToken events — disambiguate by emitter.** `Borrow` `0x13ed6866…`, `RepayBorrow` `0x1a2a22cb…`, `LiquidateBorrow` `0x298637f6…`, `MarketEntered` `0x3ab23ab0…`, ERC-20 `Transfer`/`Approval` are byte-identical across Core, Isolated, stock Compound and Moonwell. **`AccrueInterest` is 4-arg (128-byte data) `0x4dec04e7…` on BOTH Core and Isolated VTokens** (collides with Moonwell's 4-arg) — distinguish Venus Core vs Isolated by emitter, and from stock Compound by data length (Core/Isolated 128 B vs stock 3-arg 96 B). `Mint`/`Redeem` are the 4-arg Venus form (extra `totalSupply`/`accountBalance`) so differ from stock Compound's 3-arg; the legacy Core 3-arg `Mint` `0x4c209b5f…` collides with PancakeSwap/UniV2 `Mint`. XVS rewards differ by codebase: Core emits `DistributedSupplierVenus`/`DistributedBorrowerVenus` (4-arg) **from the Unitroller itself**, while Isolated emits `DistributedSupplier/BorrowerRewardToken` **from a separate `RewardsDistributor`**. In governance, `ProposalQueued`/`ProposalExecuted`/`ProposalCanceled`, `SetTrustedRemote`, `SetMaxDailyLimit`, `NewImplementation` and the three oracle `TokenConfigAdded` shapes all share topic0s across emitters.
- **Always key on `(chainId, address)`.** Nothing in Venus uses a shared cross-chain vanity address — every PoolRegistry, beacon, comptroller, vToken, XVS token, oracle and timelock is a fresh per-chain deployment, and governance ownership is per-chain (each chain's beacons + proxy admin are owned by that chain's local Venus Timelock).

## Verification methodology

- **Topic0 / selectors:** computed locally with keccak-256 from each contract's canonical signature (interface `uint`→`uint256`; enum params at their true `uint8`/`uint16`/`bytes32` widths — e.g. the PSR config events and the three distinct `TokenConfigAdded` shapes were recomputed at exact width). Tuple/struct args encoded faithfully (e.g. `PoolRegistered`'s `VenusPool` struct, the `RewardTokenBorrowIndexUpdated` `Exp{uint256}` 1-tuple). The standard-vs-Venus `DiamondCut` and the 4-arg-vs-3-arg `AccrueInterest`/`Mint` contrasts were double-checked. High-frequency topics validated against live `eth_getLogs` (e.g. Core vUSDT `AccrueInterest` over 369 logs, `DistributedSupplierVenus` 233 logs; Isolated `RewardTokenBorrowIndexUpdated`; periphery XVSVault `Deposit`/`Claim`, PSR `AssetsReservesUpdated`, Prime `InterestClaimed`, OmnichainGovExecutor `ProposalReceived`, OFT `ReceiveFromChain`).
- **Addresses + proxies:** every anchor existence-checked via `eth_getCode` on each chain's publicnode RPC; Diamond wiring read live via `facetAddresses()`/`facetAddress(selector)`; beacon/delegator logic read via `implementation()`/getter and the EIP-1967 impl/admin/beacon slots read via `eth_getStorageAt`. Pool counts confirmed live via `PoolRegistry.getAllPools()` (ETH 4, BNB 8, ARB 2, Base 1, OP 1). Absence on Avalanche + Polygon triangulated three ways (no deployment folder in any repo; `eth_getCode = 0x` for every anchor on both chains; no Venus UI presence).

## Coverage caveats (read these)

- **Per-market / per-pool addresses are discovered, not exhaustively listed.** Core Pool markets: re-read `Comptroller.getAllMarkets()` (48 and growing; each is an independently-deployed `VBep20Delegator`, no CREATE2 salt — addresses are not derivable). Isolated Pools: two-level discovery — `PoolRegistry.getAllPools()`/`PoolRegistered` → comptrollers, then `Comptroller.getAllMarkets()`/`PoolRegistry.MarketAdded` → vTokens (ETH Core pool = 23 markets and counting). The docs list the singletons/registries/beacons, not every instance. **Never freeze a market/pool list — re-enumerate.**
- **Resolve logic live.** Beacon (`VTokenBeacon`/`ComptrollerBeacon`) and proxy `*Impl` JSON fields are stale on multiple chains (§ above) — read `beacon.implementation()` / the EIP-1967 slot every time.
- **Periphery token/oracle addresses are per-chain.** XVS, the OFT endpoints, RewardsDistributor reward tokens, and the oracle adapters all differ per chain and per instance — read `rewardToken()` / the relevant getter rather than assuming XVS.
- **Also live outside the 7 requested chains** (same Isolated/periphery codebases, noted but not documented in detail): zkSync Era (324), opBNB (204), Unichain (130).
- **publicnode caveats:** getLogs ranges are capped (~50k blocks) and topic-only filters with no `address` silently return `[]`; BNB publicnode prunes old history, so pool-launch events may be unreachable there — use an archive node or read `getAllPools()`.
