# Velodrome V2 (Solidly ve(3,3) AMM) — Compressed Reference (Optimism)

**Status:** topic0/selectors via local keccak (pycryptodome) from `velodrome-finance/contracts` source; addresses from `contracts/deployment-addresses/optimism.json` re-verified on-chain (publicnode, 2026-06).
**Scope:** the classic **Solidly-style AMM** (volatile `xy=k` + stable `x³y+y³x` pools) and the **ve(3,3) governance** stack (Voter, VotingEscrow veNFT, Gauge, Minter, RewardsDistributor, VELO). **Live home = Optimism (chain 10).** Concentrated liquidity is in [`slipstream.md`](slipstream.md); cross-chain leaf deployments in [`superchain.md`](superchain.md).
**Key facts:** Velodrome V2 ≡ same codebase as **Aerodrome** (Base) — events/selectors are identical; only addresses + token differ (VELO vs AERO). The veToken is an **ERC-721 veNFT** (veVELO), not an ERC-20. The AMM `Swap` topic0 **differs from Uniswap V2** (different arg layout); several other events collide by signature with Uni V2 / Curve / Balancer / Sushi (see Detection).

### Relay (veNFT automation) + Sugar (lens) — Optimism

**Relay** automates managed-veNFT (mveNFT) reward handling: an **AutoCompounder** converts gauge/bribe/fee rewards → VELO and re-locks into the mveNFT (grows voting power); an **AutoConverter** converts rewards → a chosen output token. Public callers earn a keeper fee (≥1% of converted VELO). All on-chain ✓:
```
0x6b1253B116B5919932399295C75116d33F8EfF96 -> RelayFactoryRegistry (registry of relay factories)
0x8831DD83f24094862fBfa9bbbE2690088c3C8320 -> AutoCompounderFactory (21059B code ✓)
0x12419cfe11E85e3E28927eC84209cD7a4CfeCaB1 -> AutoConverterFactory (18170B code ✓)
0xA7B6243912FB3D752de29B1ec842f3B3313cE148 -> Optimizer (swap-route optimizer)   (registry 0x7feF5407…)
0x7bC95b327DF9d6dE05C1A02F6D252986Fcf45AF7 -> KeeperRegistry
```
Relays are mveNFTs created via the factories → each emits `CreateManaged`/`DepositManaged` on the VotingEscrow (see topics above). Source: [`velodrome-finance/relay`](https://github.com/velodrome-finance/relay).

**Sugar** is a read-only **lens** suite (LpSugar / VeSugar / RelaySugar / RewardsSugar) — view contracts for off-chain indexing; they emit no events and hold no funds (no monitoring relevance beyond convenience reads). Source: [`velodrome-finance/sugar`](https://github.com/velodrome-finance/sugar).

> **No lending/perps product.** Velodrome ships only the AMM (V2 + Slipstream), ve(3,3) governance, Relay automation, and the cross-chain superchain stack. There is **no Velodrome money market or perpetuals** to monitor.

## Proxies

- **AMM Pools: NO upgradeable proxy.** PoolFactory CREATE2-deploys each Pool (salt from `(token0, token1, stable)`); pool logic is immutable. Enumerate via `PoolCreated`.
- **Governance (Voter, VotingEscrow, Gauge, Minter, Distributor, token): immutable** Velodrome-V2 deploys (not behind upgrade proxies). Gauges are created per-pool by the Voter via gauge factories (fresh deploys, not EIP-1167 clones).

## Detection invariants & gotchas

1. **AMM `Swap` topic0 (`0xb3e27736…`) ≠ Uniswap V2 `Swap` (`0xd78ad95f…`).** Different arg layout. Don't reuse the Uni V2 constant.
2. **Same-name topic0 collisions** (disambiguate by emitter): `Mint` `0x4c209b5f…` = Uni V2/Sushi; `Sync` `0xcf2aa508…` = Sushi Trident; veNFT `Supply` `0x5e2aa66e…` = Curve/Balancer; Gauge `Deposit` `0xe1fffcc4…`/`Withdraw` `0x884edad9…` = Curve/Balancer gauge.
3. **veToken is an ERC-721 veNFT** — locks/votes keyed by `tokenId`; escrow emits ERC-721 `Transfer`. `Deposit` carries a `depositType` enum; its topic0 (`0x8835c22a…`) differs from Curve veCRV `Deposit`.
4. **`stable` bool in `PoolCreated`/`getPool`** separates stable from volatile — a pair can have BOTH; key lookups on `(token0, token1, stable)`.
5. **Three `NotifyReward` shapes**: Voter `(address,address,uint256)` `0xf70d5c69…`; Gauge `(address,uint256)` `0x09566775…`; Reward `(address,address,uint256,uint256)` `0x52977ea9…`. Different topic0s — don't conflate.
6. **Leaf-chain gauges (Mode/Soneium/…) emit a 3-arg `Deposit(address,address,uint256)` `0x5548c837…`**, NOT this 2-arg one. See [`superchain.md`](superchain.md).
7. **Bytea hex literals must have even digit count** (40 for addresses, 64 for topics).
