# Morpho V1 — Morpho Blue + MetaMorpho — Topics, Selectors, Addresses (Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon)

**Status:** verified against live RPC on every listed chain on 2026-05-29, and the canonical `morpho-org/morpho-blue`, `morpho-org/metamorpho`, `morpho-org/metamorpho-v1.1`, `morpho-org/morpho-blue-irm`, `morpho-org/public-allocator`, `morpho-org/pre-liquidation`, `morpho-org/morpho-blue-oracles`, `morpho-org/universal-rewards-distributor`, `morpho-org/bundler3` repos. Addresses parsed from `@morpho-org/blue-sdk` `addresses.ts` (chainId-keyed registry) and existence-checked via `eth_getCode`.
**Scope:** "Morpho V1" = the **Morpho Blue** singleton lending primitive ("Markets V1") + **MetaMorpho** ERC-4626 curated vaults ("Vaults V1") + their periphery (AdaptiveCurveIRM, PublicAllocator, oracle factory, pre-liquidation, Bundler3, URD, MORPHO token). Topics + selectors are chain-agnostic; addresses are network-specific. For the legacy P2P **Optimizers** see [optimizers.md](optimizers.md); for **Vaults V2 / Markets V2 (Midnight)** see [v2.md](v2.md); index in [README.md](README.md).

Morpho Blue is **immutable** — the singleton has no proxy, no admin upgrade path, no pause. Markets are created permissionlessly and are immutable post-creation (loanToken, collateralToken, oracle, irm, lltv can never change). MetaMorpho vaults are also immutable per-instance (deployed by a factory via `CREATE2`/`new`, not upgradeable). **The only governance levers on Morpho Blue are: enable an IRM, enable an LLTV, set the protocol fee (≤25%) per market, and set the fee recipient.** See §6 Proxies — the immutability is the single most important fact for monitoring.

> **Singleton address is chain-specific.** Unlike Aave/Uniswap, Morpho Blue does **not** share one address across all chains. Ethereum + Base use the vanity `0xBBBB…EEFFCb`; every other chain has a unique singleton (§3). **Always key on `(chainId, address)`.**

A market is identified by `Id = keccak256(abi.encode(MarketParams))` where `MarketParams = (address loanToken, address collateralToken, address oracle, address irm, uint256 lltv)` — a `bytes32`, **not** a numeric id and **not** an address. `Id` is the first indexed topic of nearly every Blue event.

## 6. Proxies — **almost nothing is upgradeable**

This is the defining architectural difference from Aave V3 (fully proxied). For Morpho V1:

| Contract | Upgradeable? | Pattern |
|----------|--------------|---------|
| **Morpho Blue** singleton | **NO** | Plain immutable contract. No EIP-1967 impl slot (verified `0x00` on every chain). No admin, no pause, no upgrade. |
| **MetaMorpho vault** (each instance) | **NO** | Deployed by factory via `new MetaMorpho{salt}(...)` — full immutable bytecode, not a proxy. |
| **MetaMorpho Factory** (V1.0 & V1.1) | **NO** | Immutable. "Upgrade" = deploy a new factory version (V1.0 → V1.1). |
| **AdaptiveCurveIRM / PublicAllocator / oracle adapters / PreLiquidation / Bundler3 / adapters / URD** | **NO** | All immutable. New versions are new addresses. |
| **MORPHO token (transferable, `0x58D9…`)** | **YES** | EIP-1967 upgradeable proxy → impl `0x4364fd2371b6318159366abfa51f190df5c24852`. The one notable exception. |
| MORPHO legacy (`0x9994…`) / Wrapper (`0x9D03…`) | NO | Immutable. |

EIP-1967 implementation slot (for the MORPHO token only): `0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc`. **Practical takeaway:** you never need to re-resolve a Morpho Blue / MetaMorpho impl — the address *is* the logic, permanently. (The legacy P2P Optimizers, by contrast, *are* upgradeable TransparentUpgradeableProxies — see [optimizers.md](optimizers.md).)

## 7. Detection invariants & gotchas

1. **Market `Id` is `keccak256(abi.encode(MarketParams))`** — a `bytes32`, indexed as topic1 on almost every Blue event. To resolve an `id` to its assets/oracle/irm/lltv, call `idToMarketParams(id)` (`0x2c3c9157`) or read the `CreateMarket` log's `marketParams`.
2. **Markets are immutable.** oracle/irm/lltv never change post-`CreateMarket`. There is no "reserve config changed" event because there are no mutable risk params (only the protocol `fee`, via `SetFee`, ≤25%).
3. **Two assets per market, asymmetric.** `Supply`/`Withdraw` = loan asset (earns yield). `SupplyCollateral`/`WithdrawCollateral` = collateral asset (earns nothing, cannot be borrowed). Don't sum them.
4. **`onBehalf` ≠ `tx.from` ≠ `caller`.** Position owner is `onBehalf`; `caller` is whoever called (often Bundler3 or a vault). Borrow/withdraw require the caller be authorized for `onBehalf` (`SetAuthorization`, which is **global across all markets**, not per-market). Attribute positions to `onBehalf`.
5. **Shares vs assets.** Positions are stored in **shares**; `market(id)` totals convert shares↔assets with `VIRTUAL_SHARES=1e6`/`VIRTUAL_ASSETS=1`. Interest accrues to the share price, so a fixed share balance grows in assets. Never store assets; store shares + reconstruct.
6. **Oracle price scale is `1e36`.** `oracle.price()` returns the collateral price quoted in loan-token, scaled by `1e36`. The MorphoChainlinkOracleV2 `SCALE_FACTOR` embeds `36 + loanDecimals − collateralDecimals` (so a WETH(18)/USDC(6) market's raw feed math nets a 24-decimal scale before the `1e36` quote). LLTV is `1e18`-scaled (e.g. `0.86e18` = 86%) and is the **liquidation** threshold, not a max-borrow.
7. **`Liquidate` carries bad-debt fields.** `badDebtAssets`/`badDebtShares` > 0 means the position was fully seized and remaining debt was socialized to suppliers of that market (totalSupplyAssets reduced). This is realized on the same event — no separate "deficit" event like Aave v3.3.
8. **AccrueInterest precedes actions.** Most state-changing calls emit `AccrueInterest(id, …)` first (same tx). The IRM also emits `BorrowRateUpdate(id, …)` from the IRM address. `SupplyCollateral`/`flashLoan` do **not** accrue.
9. **Shared/colliding topic0s — disambiguate by emitter & arity:**
   - `Withdraw`: Blue `(bytes32,address,address,address,uint256,uint256)` `0xa56fc0ad…` vs MetaMorpho ERC-4626 `(address,address,address,uint256,uint256)` `0xfbde797d…`.
   - `Deposit` `0xdcbc1c05…` and ERC-4626 `Withdraw` `0xfbde797d…` are shared by **MetaMorpho and Vaults V2** vaults. **`AccrueInterest` is NOT shared**: MetaMorpho = 2-field `0xf66f28b4…`, Vaults V2 = 4-field `0x4dec04e7…`, Morpho Blue = 4-field-with-id `0x9d9bd501…` — three distinct topics.
   - `SetFee`: Blue `(bytes32,uint256)` `0x139d6f58…`, MetaMorpho `(address,uint256)` `0x01fe2943…`, PublicAllocator `(address,address,uint256)` `0x44a6d70a…` — three different topics.
   - `SetOwner(address)` `0x167d3e9c…`, `SetCurator(address)` `0xbd0a63c1…`, `SetIsAllocator(address,bool)` `0x74dc60cb…` are shared by MetaMorpho and Vaults V2 — key on the vault address + factory provenance.
10. **MetaMorpho timelocked governance.** Cap raises, guardian, timelock changes are 2-step: `Submit*` then (after timelock) `Accept*`/`Set*`. A `SubmitCap` is a *pending* change; the `SetCap` is when it takes effect. Watch both.
11. **Vault discovery = factory `CreateMetaMorpho` logs.** Subscribe on the V1.1 factory (+ the V1.0 OLD factory on ETH/Base). The same topic0 is emitted by both factory versions — disambiguate by emitting address.
12. **publicnode getLogs:** topic-only filters (no `address`) silently return `[]`. Always include the contract address. Range cap ~40–50k blocks (arb/pol smaller). Rare events (`CreateMarket`, `CreateMetaMorpho`) cluster at chain launch — scan from the deploy block, not the head.
13. **BNB decoy (§3.6)** and **per-chain singleton (§4):** never hardcode `0xBBBB…` as "the Morpho address."
