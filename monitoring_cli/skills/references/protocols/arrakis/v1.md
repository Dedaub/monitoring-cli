# Arrakis V1 (G-UNI) — Topics, Selectors, Addresses (Ethereum + Polygon + Optimism + Arbitrum; absent on Base/BNB/Avalanche)

**Status:** topic0/selectors via `cast keccak`/`cast sig` and confirmed against the deployed implementation bytecode + **live `eth_getLogs`** on a real vault (verified 2026-06). Addresses from Arrakis docs + Etherscan/PolygonScan/Optimistic-Etherscan/Arbiscan, re-verified on-chain with `cast` vs `publicnode` RPCs. Each factory confirmed by live `numPools()`/`poolImplementation()` and proxy-slot reads.
**Scope:** Arrakis **V1**, originally launched as **G-UNI** by Gelato — an ERC-20 token that wraps a single managed Uniswap **V3** LP position (the `GUniPool` / "ArrakisVaultV1" contract). Plus the V1 factory (`GUniFactory`) and the V1 periphery (`GUniRouter` / Arrakis Router + `GUniResolver02`). Arrakis V2 (multi-position `ArrakisV2` vaults + `ArrakisV2Factory`) and Arrakis Modular are **separate codebases** — not covered here.

A G-UNI vault is an **ERC-20 wrapper over one Uniswap V3 position**: depositors `mint()` LP tokens against `token0/token1`, the vault holds a single `[lowerTick,upperTick]` Uniswap V3 position in a given `pool`, and **Gelato/Arrakis keepers** auto-compound fees and re-range the position via `rebalance()`/`executiveRebalance()`. Vaults are deployed by the `GUniFactory`. Two metadata generations coexist: original Gelato vaults report `name()="Gelato Uniswap <pair> LP"`, `symbol()="G-UNI"`; later vaults (after the Arrakis rebrand) report `name()="Arrakis Vault V1 <pair>"`, `symbol()="RAKIS-<n>"`. **Both are the same V1 contract family** — disambiguate by the deploying factory, never by symbol.

Arrakis V1 is live on **Ethereum (1), Polygon (137), Optimism (10), Arbitrum (42161)** — at **different (non-deterministic) factory/router addresses on each chain**. It is **absent on Base (8453), BNB (56), Avalanche (43114)** (those chains only ever got Arrakis V2 / Modular). Record `0x` (no code) explicitly per chain below.

## Proxies

- **G-UNI vaults are NOT EIP-1167 minimal-proxy clones.** Each vault is a **full custom EIP-1967 transparent proxy** (~1.6KB; deployed bytecode starts `0x60806040…` and embeds the standard EIP-1967 impl slot `0x360894…382bbc` and admin slot `0xb53127…5d6103`, plus `upgradeTo`/`upgradeToAndCall`/`changeAdmin` logic). Read the impl with `cast storage <vault> 0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc`. Example: ETH DAI/USDC vault → impl `0xb542d5cb…7369` (an older impl, NOT the factory's current `poolImplementation()` `0xAf4eDBd5…`). **A vault keeps its original implementation until the factory explicitly upgrades it** via `upgradePools()` — so vaults across one chain run a mix of impl versions.
  - The vault proxy emits **`ProxyImplementationUpdated(address,address)` `0x5570d70a…`** on upgrade and **`ProxyAdminTransferred(address,address)` `0xdf435d42…`** — it does **not** use the standard EIP-1967 `Upgraded(address)` topic.
  - The proxy **admin** is the `GUniFactory`. The factory's own `proxyAdmin()` returns the `0x…dEaD` sentinel and it administers vault proxies through `getProxyAdmin()`/`upgradePools()`/`makePoolsImmutable()`. Vault EIP-1967 admin slot reads `0x0` (admin is managed by factory logic).
- **The GUniFactory itself is also an upgradeable EIP-1967 proxy** (~3.3KB). Ethereum impl slot → `0xb9bb2856e0af9d3e855b0173a40059fc29b632da` (admin slot reads the `0x…dEaD` sentinel). Arbitrum factory shares the **same impl** `0xb9bb2856…`. Polygon impl `0x8c6a08cf…d66b`, Optimism impl `0xdbf8d32f…0691` (per-chain impls). Read live before assuming.
- **The Arrakis Router (GUniRouter) is an EIP-1967 proxy** on Ethereum/Polygon/Optimism (~3.3KB; ETH impl `0x9de3f484…`, ETH admin = `0xed5cf41b…`). The original Gelato "G-UNI Router 02" (`0x14E6D67F…`) is a **full non-proxy** contract.
- The GUniResolver02 (`0x0317650A…`) is a plain non-upgradeable helper contract.

## Detection invariants & gotchas

1. **A G-UNI vault is one Uniswap V3 position wrapped as an ERC-20.** Every vault emits `Transfer`/`Approval` (the LP token) plus `Minted`/`Burned` on deposit/withdraw and `Rebalance`/`FeesEarned` on keeper actions. `Minted`/`Burned`/`Rebalance`/`FeesEarned` topic0s are **distinctive to G-UNI** (not shared with Uniswap V3 pool events) — safe to attribute by topic0 alone, then confirm the emitter is in a factory's pool set.
2. **All four vault events are fully non-indexed** — `Minted.receiver` etc. live in `data`, not topics. You cannot filter by receiver via topics; decode the data.
3. **Two metadata generations, same contract.** Old: `symbol()="G-UNI"`, `name()="Gelato Uniswap …"`. New: `symbol()="RAKIS-<n>"`, `name()="Arrakis Vault V1 …"`. Do NOT key on symbol; attribute via the deploying `GUniFactory` (`PoolCreated.pool` = topic3) or `getGelatoPools()`/`getPools()`.
4. **`PoolCreated(uniPool, manager, pool)` — the G-UNI vault is topic3 (`pool`), the Uniswap V3 pool is topic1 (`uniPool`).** `manager` (topic2) is `0x0` for unmanaged/public vaults and a manager address for managed ones. **Managed vaults are NOT in `getGelatoPools()`** — enumerate `getDeployers()` then `getPools(deployer)` to cover all of them.
5. **`numPools()` is the liveness/coverage check.** Live this run: ETH 105, Polygon 117, Optimism 58, Arbitrum 3 (canonical factory). Arbitrum also has a separate standalone factory-shaped contract `0xb3442b0b…` (numPools 1, empty `getGelatoPools`) — treat as non-canonical.
6. **Per-chain addresses differ.** The factory/router are NOT at the same address across chains (only the *impl* `0xb9bb2856…` happens to be shared between the ETH and ARB factories). Always match `(chainId, factory address)`; never assume Ethereum's `0xea1aff9d…` exists elsewhere (it returns `0x` on every other chain).
7. **`rebalance()` is keeper-driven and pays the keeper.** `rebalance(uint160,uint256,bool,uint256 feeAmount,address paymentToken)` `0xb135c99f` lets a Gelato keeper take `feeAmount` of `paymentToken`; `executiveRebalance(...)` `0x24b8fd1b` is the manager re-range. Both emit `Rebalance` (+ `FeesEarned` when fees were collected). **Use the `uint256` forms** — `uint128` variants are wrong for this deployment.
8. **`mint`/`burn` selectors collide with common 4-byte values** (`mint(uint256,address)=0x94bf804d`, `burn(uint256,address)=0xfcd3533c`) but here mean deposit/withdraw of the LP wrapper, not Uniswap mint/burn. The router also exposes these via its own `addLiquidity`/`removeLiquidity` wrappers.
9. **Router `factory()` returns the Uniswap V3 factory** `0x1F98431c…`, not the GUniFactory — don't mistake it for the G-UNI factory pointer.
10. **First `mint` locks `MINIMUM_LIQUIDITY`** in the Uniswap position; `getMintAmounts()` is the canonical pre-deposit quote. `getPositionID()` = the vault's Uniswap V3 position key (`keccak256(vault,lowerTick,upperTick)`).
11. **Vault impl versions are mixed within a chain** (vaults aren't auto-upgraded). If reading vault internals via ABI, confirm the impl (EIP-1967 slot) — public-deposit vaults run an impl with `mint(uint256,address)`; the newest impl (`0xAf4e…` on ETH) is a restricted-mint managed variant.
