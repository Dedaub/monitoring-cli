# Uniswap V4 — Topics, Selectors, Addresses (Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon)

**Status:** verified against live RPC on every listed chain and the canonical `Uniswap/v4-core`, `Uniswap/v4-periphery`, `Uniswap/universal-router`, `Uniswap/permit2` repos on 2026-05-29.
**Scope:** Uniswap Labs' V4 deployment on Ethereum (1), Base (8453), BNB Smart Chain (56), Avalanche C-Chain (43114), Arbitrum One (42161), Optimism (10), Polygon PoS (137). Topics + selectors are chain-agnostic; addresses are network-specific.

Uniswap V4 is a **singleton** architecture and is **immutable** for all core swap/liquidity logic. Every pool on a chain lives inside **one** `PoolManager` contract — pools are NOT separate deployed contracts (unlike V2 pairs / V3 pools). A pool is identified by `PoolId = keccak256(abi.encode(PoolKey))` where `PoolKey = (Currency currency0, Currency currency1, uint24 fee, int24 tickSpacing, IHooks hooks)`. There is no `CREATE2` pool deployment and no pool init code hash — `Initialize` is just a state write inside the singleton.

Three V4-defining mechanics to internalize before indexing:

1. **Flash accounting (EIP-1153 transient storage).** Balances are netted as *deltas* during a `unlock()` callback and must net to zero before the call returns. There are no per-swap token `Transfer`s from a pool; tokens move via `take`/`settle`/`sync` and ERC-6909 claim tokens. To value a swap, read the `Swap` event's signed `amount0/amount1` (token deltas from the **pool's** perspective).
2. **Native currency is first-class.** `Currency` is just an `address`; `address(0)` = native ETH/BNB/AVAX/POL. V4 pools can hold native gas token directly with no WETH wrapping. `currency0 < currency1` is enforced, and `address(0)` always sorts first.
3. **Hooks.** A pool may attach a `hooks` contract whose **address low 14 bits encode which callbacks fire** (`beforeSwap`, `afterSwap`, `beforeAddLiquidity`, …). The hook address is part of the `PoolKey`, so the same token pair + fee + tickSpacing with a different hook is a *different* pool with a different `PoolId`.

Fee model: there are **no fixed fee tiers** in V4. `fee` and `tickSpacing` are arbitrary per pool, chosen at `initialize`. A pool with `fee == 0x800000` (`8388608`, the high bit of the `uint24`) is a **dynamic-fee pool** whose LP fee is set by its hook at runtime via `PoolManager.updateDynamicLPFee`.

### 1.1 PoolManager (singleton — every swap/LP/init action on the chain)

| `0x40e9cecb9f5f1f1c5b9c97dec2917b7ee92e57ba5563708daca94dd84ad7112f` | `Swap(PoolId indexed id, address indexed sender, int128 amount0, int128 amount1, uint160 sqrtPriceX96, uint128 liquidity, int24 tick, uint24 fee)` |
| `0xf208f4912782fd25c7f114ca3723a2d5dd6f3bcc3ac8db5af63baa85f711d5ec` | `ModifyLiquidity(PoolId indexed id, address indexed sender, int24 tickLower, int24 tickUpper, int256 liquidityDelta, bytes32 salt)` |

`Swap` is the workhorse: filter `topic0` + (singleton) `PoolManager` address, then group by `id` (topic1). `amount0`/`amount1` are signed **int128** and are from the **pool's** perspective: positive = token entered the pool, negative = token left. The trailing `fee` is the *total* fee in hundredths of a bip actually applied (includes dynamic-fee + any hook swap fee). `ModifyLiquidity` covers both add (positive `liquidityDelta`) and remove (negative); there is no separate Mint/Burn/Collect event — fee collection is a `liquidityDelta == 0` `ModifyLiquidity`.

### 1.3 PoolManager — ERC-6909 claim tokens (in-singleton balances)

The `PoolManager` is itself an ERC-6909 multi-token: an LP/router can leave funds inside as claim tokens (`id = uint256(uint160(currency))`) instead of withdrawing — cheaper for repeated ops.

| topic0 | Event |
|--------|-------|
| `0x1b3d7edb2e9c0b0e7c525b20aaaef0f5940d2ed71663c7d39266ecafac728859` | `Transfer(address caller, address indexed from, address indexed to, uint256 indexed id, uint256 amount)` *(ERC-6909)* |
| `0xceb576d9f15e4e200fdb5096d64d5dfd667e16def20c1eefd14256d8e3faa267` | `OperatorSet(address indexed owner, address indexed operator, bool approved)` *(ERC-6909)* |

The ERC-6909 `Transfer` topic0 (`0x1b3d…8859`) is **distinct** from the classic ERC-20/721 `Transfer` (`0xddf2…b3ef`) — it has an extra leading non-indexed `caller` arg and `(from, to, id)` indexed. Do not confuse the two; a V4 mint of claim tokens fires the 6909 topic with `from == address(0)`.

Each Universal Router is a fresh immutable deployment (no admin/upgrade slot) — track the latest by maintaining a per-chain allowlist, not by following an `Upgraded` event. The V4-aware build's address set is in §10.

## 12. Detection invariants & gotchas

1. **All pool activity comes from ONE address per chain.** Filter `topic0 ∈ {Initialize, Swap, ModifyLiquidity, Donate}` + `address == PoolManager`. Discriminate pools by `id` (topic1 = `PoolId`). No need to discover pool addresses — there are none.
2. **`PoolId = keccak256(abi.encode(PoolKey))`**, `PoolKey = (currency0, currency1, fee, tickSpacing, hooks)`. To resolve a `PoolId` to its tokens, either capture the `Initialize` event (carries the full key) or read `posm.poolKeys(bytes25(poolId))`. The 25-byte truncation posm uses is `bytes25(poolId)` — collisions are astronomically unlikely but key on full 32-byte `id` in your own index.
3. **`Swap.amount0/amount1` are signed int128, pool-perspective.** Positive = into pool, negative = out. The trailing `fee` is the realized total fee (LP + dynamic + hook), in pips (1e-6). There is **no** per-swap ERC-20 `Transfer` from the pool — flash accounting means token movement is via `take`/`settle`/6909, often netted. Don't try to value a swap from `Transfer` joins; use the event amounts.
4. **`ModifyLiquidity` is add AND remove AND fee-collect.** Positive `liquidityDelta` = add, negative = remove, `0` = poke/collect. There is no separate Mint/Burn/Collect (V3) split. `sender` is the posm (or a custom router), never the end user.
5. **Native currency is `address(0)`.** A pool's `currency0` is frequently `0x0000…0000` (native ETH/BNB/AVAX/POL). `currency0 < currency1` always holds and native sorts first. Code that assumes ERC-20 `Transfer`s for every leg will miss native legs entirely.
6. **Hooks change pool identity and economics.** Same `(tokenA, tokenB, fee, tickSpacing)` with two different `hooks` = two different pools. A hook can take its own fee (`beforeSwap`/`afterSwap` returns-delta), mutate the swap, or block it. The `hooks` address's low 14 bits declare which callbacks fire (§12.7) — read them off the address; you don't need the hook's source.
7. **Dynamic-fee pools** carry `fee == 0x800000` (`8388608`). Their LP fee isn't fixed — the hook sets it via `updateDynamicLPFee`. The `Swap.fee` field is the source of truth for the fee actually charged on that swap.
8. **ERC-6909 `Transfer` (`0x1b3d…8859`) ≠ ERC-20/721 `Transfer` (`0xddf2…b3ef`).** The 6909 topic carries `(caller, from, to, id, amount)`. Claim-token mints/burns (funds parked inside the singleton) fire 6909 with `from`/`to == 0`. Don't fold these into ERC-20 volume.
9. **No `IncreaseLiquidity`/`Collect` on posm.** If you ported a V3 indexer, those events don't exist in V4 — watch `PoolManager.ModifyLiquidity` with `sender == posm` and the posm ERC-721 `Transfer` for mint (`from == 0`) / burn (`to == 0`).
10. **Reading pool state needs `StateView` or `extsload`.** There's no `slot0()` on the singleton. Use `StateView.getSlot0(poolId)` → `(sqrtPriceX96, tick, protocolFee, lpFee)` and `getLiquidity(poolId)`.
11. **Universal Router command map changed.** In the V4-aware UR, `V4_SWAP = 0x10` (was transitional `0x21` in the V3-era UR). A UR tx can mix V2/V3/V4 swaps + Permit2 in one `execute`. Anchor V4 swap detection on the PoolManager `Swap` event, then attribute to UR via `tx.to ∈ {UR allowlist}`.
12. **Anyone can call `PoolManager` directly** (inside their own `unlock` callback), bypassing posm/UR entirely. The `Swap.sender` is whoever called `swap()` — an arbitrary contract, not necessarily a Uniswap router. Don't assume a router.
13. **`PoolManager.owner` == V2 `feeToSetter` == V3 factory owner** on each chain. Protocol-fee config across all three Uniswap versions is one key per chain (§10).
14. **No pool init code hash / no CREATE2 pool address.** Any tooling that derives a "pool address" is V2/V3 thinking. V4 pools have only a `PoolId`, not an address.
15. **BSC 18-decimal stablecoins** and **WPOL-not-WMATIC** caveats carry over from V2/V3 unchanged — read `decimals()`/`symbol()` per chain.

### 12.7 Hook permission flags (low 14 bits of the hook address)

| Bit | Flag | Bit | Flag |
|-----|------|-----|------|
| 13 | BEFORE_INITIALIZE | 6 | AFTER_SWAP |
| 12 | AFTER_INITIALIZE | 5 | BEFORE_DONATE |
| 11 | BEFORE_ADD_LIQUIDITY | 4 | AFTER_DONATE |
| 10 | AFTER_ADD_LIQUIDITY | 3 | BEFORE_SWAP_RETURNS_DELTA |
| 9 | BEFORE_REMOVE_LIQUIDITY | 2 | AFTER_SWAP_RETURNS_DELTA |
| 8 | AFTER_REMOVE_LIQUIDITY | 1 | AFTER_ADD_LIQUIDITY_RETURNS_DELTA |
| 7 | BEFORE_SWAP | 0 | AFTER_REMOVE_LIQUIDITY_RETURNS_DELTA |

Example: a hook at `0x…2400` has bits 13 & 10 set (`0x2400 = 1<<13 | 1<<10`) → fires `beforeInitialize` + `afterAddLiquidity`. Hook addresses are mined (CREATE2 salt search) to land on the exact required flag bits.

## 14. Verification & sources

How every constant in this doc was verified:

- **Function selectors and topic0 hashes:** computed locally as `keccak256(canonical signature)[0:4]` / full `keccak256` (pycryptodome `Crypto.Hash.keccak`). Canonical signatures taken from `Uniswap/v4-core` (`PoolManager`, `ProtocolFees`, `ERC6909`), `Uniswap/v4-periphery` (`PositionManager`, `V4Quoter`, `StateView`, `Actions`, `IHooks`), and `Uniswap/universal-router` (`Commands`).
- **Core topics confirmed live:** tallied 25 533 logs from the Ethereum `PoolManager` over a 2 000-block window — `Swap` (`0x40e9…112f`) 24 129, `ModifyLiquidity` (`0xf208…d5ec`) 1 112, ERC-6909 `Transfer` (`0x1b3d…8859`) 182, `Initialize` (`0xdd46…6438`) 108, `Donate` (`0x29ef…56cb`) 2 — all matching the locally-computed hashes.
- **Selectors confirmed live:** `eth_call` against the Ethereum posm returned `poolManager()` (`0xdc4c90d3`) → `0x0000…08A90`, `permit2()` (`0x12261ee7`) → canonical Permit2, `nextTokenId()` (`0x75794a3c`) → `296 598`, `name()` → `"Uniswap v4 Positions NFT"`. `StateView` and `V4Quoter` `poolManager()` both resolved to the singleton.
- **Address bytecode existence + equivalence:** `eth_getCode` on each chain's public RPC returns non-empty bytecode for every listed contract. Runtime sizes are byte-for-byte identical across all 7 chains: PoolManager 24 009 B, PositionManager 23 877 B, V4Quoter 5 820 B, StateView 3 531 B, PositionDescriptor proxy 1 074 B, Universal Router 24 546 B.
- **PoolManager owner:** `eth_call(poolManager, 0x8da5cb5b)` per chain returns the listed governance signer (each equal to that chain's V2 `feeToSetter` / V3 factory owner).
- **PositionDescriptor proxy:** `eth_getStorageAt(proxy, 0x360894…bbc)` (impl) and `…b53127…6103` (admin) per chain return the listed impl + admin byte-for-byte, confirming the EIP-1967 transparent-proxy pattern. All other V4 contracts return `0x` at the impl slot (not proxies).
- **Deployment addresses:** sourced from the official Uniswap V4 deployments registry (`developers.uniswap.org/contracts/v4/deployments`) and cross-checked against on-chain bytecode.
