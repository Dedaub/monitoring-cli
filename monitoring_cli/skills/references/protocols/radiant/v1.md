# Radiant V1 — Topics, Selectors, Addresses (Arbitrum only — deprecated)

**Status:** verified against Arbitrum One mainnet RPC and the canonical `radiant-capital/v2` source repo on 2026-06-08. **Radiant V1 is fully deprecated** — superseded by Radiant V2 in 2022/2023 and carrying ~zero activity. This file exists for historical position/event decoding; for the live product use [v2.md](v2.md).
**Scope:** the requested chains are Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon. **Radiant V1 was only ever deployed on Arbitrum One (chain ID 42161).** It does not exist on any other chain. Topics + selectors are chain-agnostic; addresses are Arbitrum-specific.

Radiant V1 is an **Aave V2 fork** (the same upstream as V2), so **its event topics and function selectors are byte-for-byte identical to Aave V2** — see [aave/v2.md](../aave/v2.md). The core contract is the **`LendingPool`** (Aave's `Pool` predecessor), the supply event is **`Deposit`** (not V3's `Supply`), each reserve has an `rToken` (Aave aToken equivalent), a `variableDebtToken`, and a `stableDebtToken`. The V1 `LendingPool` is an **EIP-1967 upgradeable proxy** wired to its own `LendingPoolAddressesProvider` (`0xe21b295e…`), which is **distinct** from the V2 provider — that is the cleanest way to tell V1 from V2 on Arbitrum. V1 held only **6 reserves** (WBTC, WETH, USDC.e, plus a few others). It is **not** part of the October-2024 exploit narrative (that hit the live V2 markets); V1 was already retired.

> **Why V1 still matters for monitoring:** if you crawl historic Arbitrum logs (2022–2023) you will see `Deposit`/`Borrow`/`LiquidationCall` from V1's pool `0x2032b9A8…` with the *same topic0s* as V2's pool `0xE23B4AE3…`. **Disambiguate strictly by emitting Pool address**, never by topic0.

## 3. Addresses — Arbitrum One (chain ID 42161) — the only Radiant V1 deployment

All verified via `eth_getCode` / `eth_call` on `https://arbitrum-one-rpc.publicnode.com` on 2026-06-08. Pool wiring confirmed: `LendingPool.getAddressesProvider()` → `0xe21b295e…` (the V1 provider, ≠ V2's `0x454a8daf…`), `getReservesList()` returns 6 reserves.

| Role | Address | One-liner |
|------|---------|-----------|
| **LendingPool** (proxy, impl `0xab843bec…`) | `0x2032b9A8e9F7e76768CA9271003d3e43E1616B1F` | V1 main entrypoint; emits all §1.1 events. Deprecated. |
| **LendingPoolAddressesProvider** | `0xe21B295ED46528EFD5F3EF66E18bc6ad1c87F003` | V1 registry — the V1/V2 discriminator. |

First three V1 reserve underlyings (from `getReservesList()`): WBTC `0x2f2a2543B76A4166549F7aaB2e75Bef0aefC5B0f`, WETH `0x82aF49447D8a07e3bd95BD0d56f35241523fBab1` (canonical WETH; the raw list also returns the legacy `0xfd086b…` and `0xff970a…` USDC.e wrappers depending on configuration era). The full V1 rToken/debt-token set is recoverable on-chain via `getReserveData(underlying)` per reserve — they are *not* re-enumerated here because V1 is deprecated and carries no live monitoring value.

**Not deployed on any other chain.** Verified absent (no V1 LendingPool / provider) on Ethereum, Base, BNB, Avalanche, Optimism, Polygon.

## 5. Detection invariants & gotchas

1. **Radiant V1 = Arbitrum only, deprecated.** No other chain has V1. Use [v2.md](v2.md) for the live product.
2. **Topics/selectors are Aave V2's, identical to Radiant V2.** The *only* reliable V1-vs-V2 discriminator on Arbitrum is the **emitting Pool address** (`0x2032b9A8…` = V1, `0xE23B4AE3…` = V2) or the `getAddressesProvider()` return (`0xe21b295e…` = V1, `0x454a8daf…` = V2).
3. **`Deposit`, not `Supply`** (Aave V2 lineage; topic `0xde685721…`).
4. **rToken vs debt-token `Mint`/`Burn` have different topic0s** (Aave V2 behavior). Match on `(topic0, emitting address)`.
5. **Expect ~zero current activity** — V1 logs are historic (2022–2023). A tip-window scan returns 0 (verified 2026-06-08).
