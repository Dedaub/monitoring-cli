# Radiant V2 — Topics, Selectors, Addresses (Arbitrum, BNB, Ethereum, Base)

**Status:** verified against Arbitrum One, BNB Smart Chain, Ethereum, and Base mainnet RPC (live logs + storage-slot reads) and the canonical `radiant-capital/v2` source repo + official Radiant docs deployment pages, on 2026-06-08. Non-obvious claims additionally fact-checked against primary sources — see §11.
**Scope:** the requested chains are Ethereum (1), Base (8453), BNB (56), Avalanche (43114), Arbitrum (42161), Optimism (10), Polygon PoS (137). **Radiant V2 lending markets exist on only four chains — Arbitrum One (42161), BNB Smart Chain (56), Ethereum (1), and Base (8453).** **Avalanche, Optimism, and Polygon have NO Radiant contracts at all** (every Radiant address `eth_getCode`-checks to `0x` on those three — recorded in §4 and §7). Topics + selectors are chain-agnostic; addresses are network-specific.

Radiant V2 is an **omnichain money market built as an Aave V2 fork** with a **LayerZero + Stargate** cross-chain layer (RDNT is a LayerZero OFTV2 token; Stargate routes cross-chain borrows/deposits). Because the lending core forks Aave V2, **its `LendingPool` event topics and function selectors are byte-for-byte identical to Aave V2** — see [aave/v2.md](../aave/v2.md). The core contract is the **`LendingPool`** (Aave's `Pool` predecessor), the supply event is **`Deposit`** (not V3's `Supply`), each reserve has an **`rToken`** (Aave aToken equivalent, e.g. `rWETH`/`rUSDC`), a **`variableDebtToken`** (e.g. `vdWETH`), and a `stableDebtToken`. Every core contract is an **EIP-1967 upgradeable proxy** (the same factor that made the protocol exploitable through key compromise).

> **OPERATIONAL STATUS — exploited & winding down (verified 2026-06-08).** On **2024-10-16** Radiant suffered a **~$50M exploit on its Arbitrum and BNB markets**. This was **not a core-contract bug** — it was an **operational/private-key compromise**: attackers used malware (Mandiant attributed it to the North-Korea-linked **AppleJeus / Lazarus** campaign) to compromise multiple developers' hardware wallets, obtaining ≥3 of the multisig/owner keys and pushing a malicious upgrade/transfer through the contracts' *legitimate* upgradeability. The DAO never recovered the funds and **began winding down on 2026-06-01**; the protocol is now in **maintenance mode** — front-end + core contracts stay online so users can withdraw deposits, repay loans, and adjust existing positions, but new growth has stopped. **Live `Deposit`/`Borrow` volume is near zero; `Withdraw`/`Repay`/`ReserveDataUpdated` dominate** (verified via live `eth_getLogs` on all four chains — see §11). Because the exploit vector was *the upgrade path*, **the `Upgraded(address)` topic on the core proxies and the staking-proxy ProxyAdmin/timelock are the highest-value things to monitor here** (§8–§9).

> **Oracle base is 8-dec USD, not Aave-V2's ETH/wei (verified).** Standard Aave V2 quotes prices and `getUserAccountData` in **ETH (1e18)**. Radiant's `AaveOracle` returns **8-decimal USD** (verified live: Arbitrum `getAssetPrice(WETH) ≈ 1.686e11` = ~$1686, `getAssetPrice(USDC) = 99,966,000` ≈ $1.00). Treat account-data / price scaling like Aave **V3**, not V2.

## 9. Detection invariants & gotchas

1. **Exploited + winding down (maintenance mode since 2026-06-01).** Expect mostly `Withdraw`/`Repay`/`LiquidationCall`/`ReserveDataUpdated`; `Deposit`/`Borrow` near zero (verified live on all four chains). The exploit (2024-10-16, ~$50M, Arbitrum+BNB) was a **private-key/operational compromise (AppleJeus/Lazarus)**, not a core-contract bug — so the alpha for monitoring is **admin/upgrade actions**, not the lending math.
2. **Watch the upgrade & ownership surface.** Core contracts are upgradeable proxies. Monitor `Upgraded(address)` on the LendingPool/token/staking proxies, ProxyAdmin owner changes, and the security timelock. An unexpected `Upgraded` or admin transfer is the canonical replay of the exploit pattern.
3. **Lending markets on 4 chains only** — Arbitrum, BNB, Ethereum, Base. **Avalanche / Optimism / Polygon = nothing** (verified `0x`).
4. **Per-chain unique addresses; RDNT collides with an unrelated "NOVA" token on Arbitrum** (§7.2). Always key on `(chainId, address)`; never trust an RDNT address on the wrong chain.
5. **It's an Aave V2 fork** — `Deposit`/`Borrow`/`Repay`/`Withdraw`/`LiquidationCall`/`FlashLoan`/`ReserveDataUpdated` topics and selectors are identical to [aave/v2.md](../aave/v2.md) **and to Radiant V1**. Disambiguate by `(Pool address, chain)`.
6. **`Deposit`, not `Supply`** (topic `0xde685721…`, selector `deposit` `0xe8eda9df`).
7. **Oracle/account-data scale is 8-dec USD, not Aave-V2's ETH/wei** (fork divergence; verified live). `getUserAccountData` collateral/debt in 8-dec USD; HF in 1e18; `getAssetPrice` 8-dec USD.
8. **rToken vs variableDebtToken `Mint`/`Burn` have different topic0s** (Aave V2 lineage). Match on `(topic0, emitting address)`. rTokens rebase — store `scaledBalanceOf`, reconstruct with `liquidityIndex` (ray).
9. **`onBehalfOf` ≠ `tx.from`.** Attribute positions to the event's `onBehalfOf`/`user`, not the sender — the WethGateway, Leverager (`loop`), Compounder, and StargateBorrowV2 all act on users' behalf.
10. **Two unrelated `Paused`/`Unpaused` shapes.** LendingPool emits no-arg `Paused()`/`Unpaused()` (`0x9e87fac8…`/`0xa45f47fd…`); RDNT (OZ Pausable) emits `Paused(address)`/`Unpaused(address)` (`0x62e78cea…`/`0x5db9ee0a…`). Don't conflate.
11. **`TreasuryUpdated(address)` topic0 `0x7dae230f…` is shared by RadiantOFT and Leverager** — disambiguate by emitter.
12. **Base has no ChefIncentivesController** — §1.3 emission events do not exist there (Base used Merkl). Don't scan for them on Base.
13. **Two GMX `gm*` + two `GLV` reserves on Arbitrum** are Radiant-specific (GMX-market collateral) with no Aave analogue.
14. **Cross-chain RDNT supply** moves via LayerZero OFTV2 (`SendToChain`/`ReceiveFromChain`) — net these across the 4 chains, don't double-count, the OFT burns on send and mints on receive.
