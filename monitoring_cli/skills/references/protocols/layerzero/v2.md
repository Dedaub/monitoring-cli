# LayerZero V2 — Topics, Selectors, Addresses (Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon)

**Status:** verified against live RPC on every listed chain and the canonical `LayerZero-Labs/LayerZero-v2` repo + the official deployments metadata registry on 2026-06-09.
**Scope:** LayerZero V2 generic-messaging stack — `EndpointV2`, the message libraries `SendUln302`/`ReceiveUln302`/`ReadLib1002`, the `Executor` + `LzExecutor`, `DeadDVN`, `BlockedMessageLib` and `EndpointV2View` — on Ethereum (1), Base (8453), BNB Smart Chain (56), Avalanche C-Chain (43114), Arbitrum One (42161), Optimism (10), Polygon PoS (137). Topics/selectors are chain-agnostic; addresses are network-specific. V1 (Endpoint + UltraLightNodeV2) is documented separately in [v1.md](./v1.md).

LayerZero V2 is a **generic cross-chain messaging protocol**, not a token bridge: it moves arbitrary bytes between chains. Applications (OFTs, OApps) call into `EndpointV2`, which dispatches the outbound message to a configured **send message library** (`SendUln302`) and accepts inbound verification through a **receive message library** (`ReceiveUln302`). Security is modular — each OApp picks a set of **DVNs** (Decentralized Verifier Networks) and an **Executor**; there is no single privileged relayer like V1. The single workhorse event for indexing outbound traffic is **`PacketSent`** emitted by `EndpointV2`; inbound is **`PacketVerified`** then **`PacketDelivered`**.

**`EndpointV2` is deployed at one deterministic address — `0x1a44076050125825900e736c501f859c50fE728c` — on all seven chains** (and on essentially every LayerZero V2 EVM chain). It is **immutable / non-upgradeable** (EIP-1967 impl slot reads `0x0`), as are the ULN message libraries and `ReadLib1002`. The **`Executor` and `LzExecutor` are the only upgradeable pieces** (EIP-1967 Transparent proxies). Message-library addresses are **per-chain** (different on every network) and are *registered* into the Endpoint, then selected per-OApp; an OApp that has not overridden config uses the Endpoint's default send/receive library for the destination `eid`.

> **`eid` ≠ `chainId`.** LayerZero V2 addresses chains by **Endpoint ID** (`eid`), a protocol-internal identifier with no relation to EVM `chainId`. Mainnet V2 eids are `30000 + n`. The seven targets: Ethereum `30101`, BNB `30102`, Avalanche `30106`, Polygon `30109`, Arbitrum `30110`, Optimism `30111`, Base `30184`. Confirmed live: `EndpointV2.eid()` on Ethereum returns `0x7595` = `30101`. Always translate `eid → chainId` from the registry, never assume.

## 0. Contract families & versions

| Component | Role | Upgradeable? | Address shape |
|-----------|------|--------------|---------------|
| **EndpointV2** | Protocol entrypoint; emits `PacketSent`/`PacketVerified`/`PacketDelivered`/compose events; routes to libs | **immutable** | deterministic, same on all 7 |
| **SendUln302** | Default send message library (ULN 302); pays DVN+Executor fees, emits `PacketSent` payload | **immutable** | per-chain |
| **ReceiveUln302** | Default receive message library; DVNs call `verify`, then `commitVerification` → `PacketVerified` | **immutable** | per-chain |
| **ReadLib1002** | lzRead message library (on-chain read requests) | **immutable** | per-chain |
| **Executor** | Executes inbound `lzReceive`/`lzCompose` on the destination with the app's gas | **EIP-1967 Transparent proxy** | per-chain |
| **LzExecutor** | Newer executor wrapper used by the default config | **EIP-1967 Transparent proxy** | per-chain |
| **DeadDVN** | Placeholder DVN for inactive default config slots (always reverts verification) | immutable | per-chain |
| **BlockedMessageLib** | Library that disables a pathway (any send/receive reverts) | immutable | **same on all 7** (`0x1ccb…d862`) |
| **EndpointV2View** | Read-only helper mirroring Endpoint state | immutable | per-chain |
| **ZRO token** | LayerZero protocol token (OFT), optional fee payment | OFT (see v1.md §) | **same on all 7** (`0x6985…71cd`) |

### 1.3 ReceiveUln302 (per-chain receive library) — verification

| topic0 | Event |
|--------|-------|
| `0x2cb0eed7538baeae4c6fde038c0fd0384d27de0dd55a228c65847bda6aa1ab56` | `PayloadVerified(address dvn, bytes header, uint256 confirmations, bytes32 proofHash)` |

## 7. Detection invariants & gotchas

1. **`eid` ≠ `chainId`.** Every V2 event/struct uses `srcEid`/`dstEid` (uint32), not EVM chainId. Mainnet eids are `30000+n` (ETH 30101, BNB 30102, Avax 30106, Polygon 30109, Arbitrum 30110, Optimism 30111, Base 30184). Translate via the registry; never assume.
2. **`PacketSent` is the outbound workhorse and has NO indexed fields.** Filter by emitter (`EndpointV2` `0x1a44…728c`) + topic0; you cannot pre-filter by destination. The destination/sender/nonce/guid live inside `encodedPayload` — decode the packet header.
3. **`guid` is the cross-chain join key.** Match a source `PacketSent` to a destination `PacketVerified`/`PacketDelivered` by `guid` (bytes32). `(srcEid,sender,nonce)` is the equivalent tuple key carried in the `Origin` struct of inbound events.
4. **EndpointV2 is the same address on all 7 chains.** `0x1a44076050125825900e736c501f859c50fE728c`. Do NOT use the Endpoint address to distinguish chains — use `chainId` from your RPC and `eid()` from the contract.
5. **Message libraries are per-chain and per-app-selectable.** An OApp can override its send/receive library (`SendLibrarySet`/`ReceiveLibrarySet`). If you only watch the default `SendUln302`, you miss apps on a custom library. To capture *all* outbound traffic, key off `EndpointV2.PacketSent` (every send funnels through the Endpoint regardless of library).
6. **V2 address-role collision:** Optimism `SendUln302` == Polygon `ReceiveUln302` == `0x1322871e4ab09Bc7f5717189434f97bBD9546e95`. Key on `(chainId, address, role)`.
7. **`UlnConfigSet` topic0 is shared between SendUln302 and ReceiveUln302.** Disambiguate by emitting library address (send-side vs receive-side config).
8. **DVN verification is multi-event.** One `PayloadVerified` per DVN on `ReceiveUln302`; the packet is only `PacketVerified` (on the Endpoint) once the app's required-DVN threshold is met. A `PayloadVerified` alone does not mean delivery.
9. **Failed execution emits an alert, not a revert log loss.** `LzReceiveAlert`/`LzComposeAlert` fire when the Executor's `lzReceive`/`lzCompose` reverts — the packet stays verified-but-undelivered and can be retried via `clear`. Monitor alerts as stuck-message signals.
10. **`nilify`/`burn` are message-kill paths.** `PacketNilified`/`PacketBurnt` mean an OApp/owner invalidated a verified payload (security action). Rare but high-signal.
11. **Only the Executors are upgradeable.** Endpoint + ULN libs are immutable; a "protocol upgrade" is a new library registration, surfaced by `LibraryRegistered`/`DefaultSendLibrarySet`/`DefaultReceiveLibrarySet` on the Endpoint — treat those like an `Upgraded` event.
12. **ZRO fee is opt-in.** `EndpointV2.lzToken()` returns `0x0` on Ethereum even though the ZRO token (`0x6985…71cd`) exists; apps must opt into `payInLzToken=true`. Don't assume ZRO is wired as the fee token.
13. **`BlockedMessageLib` (`0x1ccb…d862`, same on all 7) reverts on use** — if an OApp's configured lib resolves to it, that pathway is intentionally disabled.
