# LayerZero V1 — Topics, Selectors, Addresses (Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon)

**Status:** verified against live RPC on every listed chain and the canonical `LayerZero-Labs/LayerZero` repo + the official deployments metadata registry on 2026-06-09.
**Scope:** LayerZero V1 generic-messaging stack — the per-chain `Endpoint`, `UltraLightNodeV2` (ULN — the v2 message library inside V1), the `NonceContract`, the V1.5 `SendUln301`/`ReceiveUln301` libraries, `TreasuryV2`, and the off-chain `Relayer`/`RelayerV2` (on-chain fee sinks) — on Ethereum (1), Base (8453), BNB Smart Chain (56), Avalanche C-Chain (43114), Arbitrum One (42161), Optimism (10), Polygon PoS (137). Topics/selectors are chain-agnostic; addresses are network-specific. The current generation (EndpointV2 + SendUln302/ReceiveUln302) is documented in [v2.md](./v2.md).

LayerZero V1 is the **original generic cross-chain messaging protocol**: an application (UA = User Application, e.g. an OFT-V1) calls `Endpoint.send(...)`, the Endpoint forwards to the configured **send library** (in practice `UltraLightNodeV2`), which emits the **`Packet(bytes)`** event carrying the full wire payload. An off-chain **Relayer** delivers the packet and an off-chain **Oracle** delivers the block header; the destination `UltraLightNodeV2.validateTransactionProof(...)` checks both and calls `Endpoint.receivePayload(...)`, which invokes the UA's `lzReceive`. The single workhorse event for indexing outbound V1 traffic is **`Packet(bytes)`** emitted by `UltraLightNodeV2`.

> **Naming trap.** "UltraLightNode**V2**" is the message-library version inside LayerZero **V1** — it is NOT LayerZero V2. The whole stack here (Endpoint + ULNv2) is protocol **V1**, addressed by 16-bit **chainId** (`uint16`), e.g. Ethereum `101`, BNB `102`, Avalanche `106`, Polygon `109`, Arbitrum `110`, Optimism `111`, Base `184`. These V1 chainIds are distinct from V2 `eid`s (`30000+`) and from EVM `chainId`s. LayerZero V1 is still live but is largely superseded by V2; new OFTs deploy on V2.

**Addresses are per-chain and exhibit heavy address-role collisions (CREATE-nonce reuse across chains).** The `Endpoint` is one literal on Ethereum, a *different* literal on Base, and a *third* literal shared by the other five chains. `UltraLightNodeV2` is one shared literal on six chains and a different one on Base. Worse, the same literal `0x66A71D…Cd675` is the **Endpoint on Ethereum**, the **NonceContract on Base**, and the **original (v1) UltraLightNode on BNB/Avax/Arb/OP/Polygon**. **Key every V1 address on `(chainId, address, role)` — never address alone.**

## 7. Proxies (old & new)

EIP-1967 implementation slot `0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc`. Read with `eth_getStorageAt`.

| Contract | Pattern | Detection | Upgrade auth |
|----------|---------|-----------|--------------|
| **Endpoint** | **immutable** | impl slot = `0x0` on Ethereum (`0x66A71D…`, 15,282 B); same shape on shared `0x3c22…cf62` | none |
| **UltraLightNodeV2** | **immutable** | impl slot = `0x0` (`0x4D73…`, 22,613 B) | `Ownable` (owner-set config only; no impl swap) |
| **NonceContract** | **immutable** | small (1,113 B) plain contract | none |
| **SendUln301 / ReceiveUln301** | **immutable** | impl slot = `0x0` (SendUln301 `0xD23108…`, 22,620 B) | none |
| **TreasuryV2** | **immutable** | impl slot = `0x0` (`0x3773E1…`, 3,837 B) | `Ownable` |

**There are NO proxies in the LayerZero V1 stack** — every core V1 contract is deployed without an upgradeability proxy (impl slot reads all-zero, confirmed live on Ethereum 2026-06-09). V1 is "upgraded" by **registering a new library version** on the Endpoint (`NewLibraryVersionAdded` → `DefaultSendVersionSet`/`DefaultReceiveVersionSet`) and pointing UAs at it via `setSendVersion`/`setReceiveVersion` — there is no `Upgraded(address)` event to watch on the V1 contracts themselves. Monitor `NewLibraryVersionAdded` / `Default*VersionSet` on the Endpoint as the V1 equivalent of an upgrade.

## 8. Detection invariants & gotchas

1. **`Packet(bytes)` on UltraLightNodeV2 is the V1 outbound workhorse** (analogue of V2 `PacketSent`). Filter by emitter (the chain's ULNv2 address) + topic0 `0xe9bded5f…`. The destination/sender/nonce are inside the `payload` bytes — decode the packet, there are no indexed topics.
2. **"UltraLightNodeV2" is part of LayerZero V1, not V2.** Do not confuse with EndpointV2/SendUln302 (those are [v2.md](./v2.md)).
3. **V1 chainId (`uint16`) ≠ EVM chainId ≠ V2 eid.** ETH 101, BNB 102, Avax 106, Polygon 109, Arbitrum 110, Optimism 111, Base 184. Always translate via the registry.
4. **Triple-role address collision on `0x66A71D…Cd675`:** Endpoint on Ethereum, NonceContract on Base, original `UltraLightNode` on BNB/Avax/Arb/OP/Polygon. **Key on `(chainId, address, role)`** — an indexer that treats this literal as "the V1 Endpoint" globally will misattribute on 6 chains.
5. **Three distinct Endpoint literals:** ETH `0x66A71D…`, Base `0xb631…0dd7`, shared `0x3c22…cf62` (the other 5). ULNv2 is shared `0x4D73…78E2` on all but Base (`0x38dE71…`).
6. **Other lib-address collisions:** Arbitrum SendUln301 == Avalanche SendUln301; Optimism SendUln301 == Polygon ReceiveUln301. Resolve by `(chainId, role)`.
7. **`PayloadStored` = a failed inbound delivery (channel blocked).** It is emitted on the **Endpoint** when `lzReceive` reverts; the channel stays blocked until `retryPayload` or `forceResumeReceive`. A `PayloadStored` without a later `PayloadCleared`/`UaForceResumeReceive` is a stuck-message alert.
8. **Two-actor delivery model (V1-specific):** an off-chain **Relayer** delivers the packet AND an off-chain **Oracle** delivers the block header (`HashReceived`); `validateTransactionProof` requires both. This differs fundamentally from V2's DVN model. The relayer/oracle are off-chain; the on-chain `RelayerV2`/`UltraLightNodeV2` hold their fees (`accruedNativeFee`, `WithdrawNative`).
9. **Attribute to the UA, not `tx.from`.** OFT-V1 sends route through the token contract → Endpoint; the user is the OFT `from`/`SendToChain.from`, not necessarily the tx sender (relayers, routers, multisends).
10. **OFT signatures vary by version.** V1 tokens emit `SendToChain`/`ReceiveFromChain`; V2/V3 tokens (and ZRO) emit `OFTSent`/`OFTReceived`. Match both topic sets when indexing token flows.
11. **No proxies in V1** — `Upgraded(address)` never fires on the V1 stack. Watch `NewLibraryVersionAdded`/`DefaultSendVersionSet`/`DefaultReceiveVersionSet` on the Endpoint for path changes.
12. **V1 is legacy but live.** Outbound `Packet` traffic still flows; new omnichain apps deploy on V2. Confirmed live: V1 `Packet` + `RelayerParams` events present on Ethereum ULNv2 in a recent window (§10).
