# Synthetix V3 — Topics, Selectors, Addresses (Optimism, Base, Arbitrum, Ethereum)

**Status:** Verified on-chain via eth_getCode + eth_call (June 2026). Addresses from `Synthetixio/v3-contracts` meta.json files; event signatures from `Synthetixio/synthetix-v3` interface sources; topic0s computed via `cast keccak`.
**Absent chains:** BNB Smart Chain (56), Avalanche C-Chain (43114), Polygon PoS (137) — `eth_getCode` returns `0x` on all three.
**Scope:** CoreProxy (liquidity layer), AccountProxy (NFT accounts), USDProxy (synthetic USD — `sUSD` on Optimism, `snxUSD` on Base, `USDx` on Arbitrum), SpotMarketProxy, PerpsMarketProxy, OracleManagerProxy.

---

## 0. Contract Families

| Family | Description | Proxy pattern |
|---|---|---|
| CoreProxy | Central liquidity router — account/collateral/vault/pool/mint-USD | Router (Diamond-like); delegates all calls to a per-chain Router contract |
| AccountProxy | ERC-721 NFT representing a V3 account (non-transferable by default) | Same Router proxy shape |
| USDProxy | Synthetic USD ERC-20 — symbol varies per chain: `sUSD` (Optimism), `snxUSD` (Base), `USDx` (Arbitrum); not deployed on Ethereum | Same Router proxy shape |
| SpotMarketProxy | Atomic + async synth swaps, synth registry | Router proxy |
| PerpsMarketProxy | Perpetuals — order commit/settle, position liquidation | Router proxy (Base, Arbitrum only) |
| OracleManagerProxy | On-chain oracle aggregation (Chainlink, Pyth, etc.) | Router proxy |
| TreasuryMarketProxy | V3 treasury market; mints/redeems protocol-owned debt | Router proxy (Optimism, Ethereum only) |
| Router (implementation) | The Diamond-router contract pointed to by each proxy via `getImplementation()` | Immutable per deployment; chains selector → module address |

**Proxy storage note:** The proxy keeps the Router address in its own ERC-7201-like storage. Call `getImplementation()` (selector `0xaaf10f42`) on any proxy to retrieve the current Router. The Router is upgraded by calling `upgradeTo(address)` (selector `0x3659cfe6`) or `upgradeToAndCall(address,bytes)` (selector `0x4f1ef286`) which fires an `Upgraded(address indexed self, address implementation)` event from the **proxy** address.

## 9. Proxies

### Proxy pattern: Router (Diamond-like)

V3 proxies are **not** EIP-1967 transparent proxies and do **not** follow the Diamond EIP-2535 spec exactly. The pattern is:

```
EOA → CoreProxy (fallback → delegatecall → Router)
                                    ↓
                         Router maps selector → Module address
                         (all modules share CoreProxy storage)
```

1. **CoreProxy** (and all other proxies) hold zero logic of their own. The fallback delegates every call to the **Router** contract.
2. The **Router** contains a large `if-else` (or jump table) mapping 4-byte selectors to module implementation addresses. These module addresses are baked into the Router bytecode at deploy time.
3. The Router is **not** the module itself; the modules are separate contracts. Delegatecall still executes in the context of the proxy's storage.
4. **Upgrading:** A new Router contract is deployed (with updated module addresses), then `upgradeTo(newRouter)` is called on the proxy. This emits `Upgraded(address indexed self, address implementation)` from the **proxy** address.

### Reading the current Router

```bash
cast call <proxy_address> "getImplementation()(address)" --rpc-url <rpc>
```

### Storage layout note

The Synthetix V3 storage uses namespaced slots (ERC-7201 style): each module struct is stored at `keccak256(abi.encode(keccak256("io.synthetix.<module-name>"), 0))`. This means standard slot-scanning tools will not find the Router pointer at slot 0 on any chain (including Base — slot 0 is zero on Base, confirmed on-chain). Use `getImplementation()` to resolve the current Router.

### Proxy ownership

Ownership uses `NominateNewOwner` / `AcceptOwnership` two-step pattern. There is no `ProxyAdmin`; admin calls go directly to the proxy via the Router.

## 10. Detection Invariants & Gotchas

1. **All events from proxy address.** Modules emit events via delegatecall, so `address` in the log is the proxy (CoreProxy, PerpsMarketProxy, etc.), not a module implementation.

2. **Base CoreProxy is NOT the vanity address.** If monitoring by address across chains, include `0x32C222A9A159782aFD7529c87FA34b96CA72C696` for Base in addition to `0xffffffaEff0B96Ea8e4f94b2253f31abdD875847` for OP/Arb/ETH.

3. **USDProxy same address, different token name.** `0xb2F30A7C980f052f02563fb518dcc39e6bf38175` is snxUSD on OP/ETH but "USDx" on Arbitrum. Same contract address, different branding.

4. **Liquidation event uses a struct tuple.** The `Liquidation` event `topic0` is `0xe6c1b266...` (with `LiquidationData` ABI-encoded as `(uint256,uint256,uint256)` tuple). Using the flat 6-param signature `Liquidation(uint128,uint128,address,uint256,uint256,uint128)` will yield the **wrong** topic0.

5. **No PerpsMarket on Optimism or Ethereum.** Perps-specific events (OrderSettled, PositionLiquidated, etc.) only appear on Base and Arbitrum.

6. **eth_getLogs on publicnode may return empty for recent blocks** — confirmed that code exists and tokens show non-zero totalSupply; log availability through public endpoints may vary.

7. **Router changes on upgrade.** After an upgrade, `getImplementation()` returns a new Router address. Watch for `Upgraded` events from proxy addresses to detect upgrades.

8. **TrustedMulticallForwarder (`0xE2C5658...`) is the same address on all chains** including Base. Transactions sent via this forwarder will show `msg.sender = forwarder` in protocol logic unless ERC-2771 meta-tx is in use.

9. **PerpsAccountProxy vs AccountProxy.** On Base and Arbitrum there is a separate `PerpsAccountProxy` (`0xcb68b813...` on Base, `0xD935444f...` on Arb) for perps accounts — these are distinct from the core `AccountProxy`.
