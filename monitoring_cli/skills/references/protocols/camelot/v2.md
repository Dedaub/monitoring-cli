# Camelot V2 — Compressed Deployment Reference (Arbitrum-only)

**Status:** All addresses verified via `eth_getCode` on Arbitrum One (42161). `eth_getCode = 0x` confirmed on all 6 non-Arbitrum target chains (Ethereum/1, Base/8453, BNB/56, Avalanche/43114, Optimism/10, Polygon/137). All topic0s extracted from contract bytecode (PUSH32 opcode analysis) and cross-checked via openchain.xyz / 4byte.directory. Five topic0s confirmed against live `eth_getLogs` on Arbitrum (2026-06-09, publicnode). No proxies detected (EIP-1967 slot = 0x000…0 on all contracts).
**Scope:** Camelot **V2** — the classic constant-product / Solidly-stable two-pool AMM with dynamic directional fees and referral system. Includes the GRAIL/xGRAIL tokenomics layer (ve(3,3)-style escrowed governance), deprecated NFTPool + NitroPool farming, and associated periphery. Other versions: [`v3.md`](v3.md) (Algebra V1-based concentrated liquidity).

**Key architecture facts:**
- **Arbitrum-only.** Never deployed on other chains in this form.
- **Dual pool types**: volatile (`xy = k`) and stable (`x³y + xy³`). The `stable` flag is stored per-pair and readable via `pair.stable()` / `pair.stablePair()` — it is **NOT** emitted in `PairCreated` (no bool in that event).
- **Dynamic directional fees**: each pair has independent `token0FeePercent` and `token1FeePercent` (set by `setFeePercent(uint16,uint16)`, capped at `FEE_DENOMINATOR=100000`). Global defaults: `ownerFeeShare` + `referrerFeeShare` split from pair fees.
- **Referral system**: Router `swap*` functions accept an extra `address referrer` parameter. Referrers earn `referrerFeeShare` of swap fees. This is why Camelot V2 swap functions are **not ABI-compatible** with Uniswap V2 Router (different selector for FOT variants).
- **Pair `swap()` ABI**: identical to Uni V2 (`swap(uint256,uint256,address,bytes)` = `0x022c0d9f`). The `Swap` event topic0 is **identical** to Uniswap V2 (`0xd78ad95f…`). Disambiguate via factory address or factory.allPairs lookup.
- **xGRAIL**: non-transferable escrowed GRAIL (transfer only whitelisted addresses). Users `convert()` GRAIL → xGRAIL at 1:1; `redeem()` with 15-day minimum to 180-day maximum vesting at 50%–100% GRAIL back. Plugin system: `allocate()` / `deallocate()` locks xGRAIL in protocol plugins (staking, yield booster, launchpad).
- **Deprecated farming**: CamelotMaster, NFTPoolFactory, NitroPoolFactory still have deployed code but are no longer used for new incentives.
- **FeeManager** (`feeTo`): receives protocol swap fees from all pairs; distributes to `dividendsShare` (56.25%) and `buybackAndBurnShare` (31.25%) wallets.

## Monitoring notes

- **Swap monitoring on Camelot V2**: filter `topic0 = 0xd78ad95f…` + `address IN (factory.allPairs)`. Cannot distinguish from Uni V2 by topic alone — must scope by pair address.
- **Large liquidity removal**: watch `Burn` (`0xdccd412f…`) on specific pairs. `topics[1] = sender`, `topics[2] = to` (both indexed).
- **xGRAIL large conversions**: `Convert` (`0xccfaeb30…`) on `0x3CAaE25Ee…` — `data[0]` is amount.
- **xGRAIL governance changes**: `UpdateRedeemSettings` (`0x5b37d107…`) changes the min/max redeem ratio and duration — affects GRAIL unlock economics.
- **Fee parameter changes on pairs**: `FeePercentUpdated` (`0xa4877b8e…`) on any pair contract. Can affect swap costs and arbitrage opportunities.
- **Protocol fee redirection**: `FeeToTransferred` (`0x941bed5e…`) on Factory = critical — changes where all protocol fees flow.
- **Referrer fee changes**: `ReferrerFeeShareUpdated` (`0x4deb2bb3…`) on Factory; specific referrer + old/new share in data.
- **Pair type locked**: `SetPairTypeImmutable` (`0x09122c41…`) — after this fires, stable/volatile type can never change on that pair.
- **Deprecated farming**: NFTPoolFactory and NitroPoolFactory still emit events but new pools are not incentivised. CamelotMaster `masterAddress()` on GRAIL returns `0x55401A4F…`.
