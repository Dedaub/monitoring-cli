# Fraxswap V2 — Topics, Selectors, Addresses (Ethereum, BNB, Avalanche, Arbitrum, Optimism, Polygon; NOT Base)

**Status:** verified against Ethereum, BNB, Avalanche, Arbitrum, Optimism, and Polygon mainnet RPC and the `FraxFinance/fraxswap` repo (Etherscan-verified source) on 2026-06-09.
**Scope:** all six chains where Fraxswap V2 is deployed. Topics + selectors are chain-agnostic; addresses are network-specific.

Fraxswap V2 is a **Uniswap V2 fork with an embedded TWAMM** (Time-Weighted Average Market Maker). Each `FraxswapPair` is simultaneously a standard AMM pair (constant-product, configurable LP fee) and a TWAMM engine that executes long-term DCA orders as virtual swaps across discrete time intervals (`orderTimeInterval`, default 3600 s). V2 added a per-pool configurable LP fee (`LpFeeUpdated` event) on top of V1's fixed 30 bps. Pairs are deployed by the `FraxswapFactory` via `CREATE2` with `token0 < token1` enforced; each pair doubles as the ERC-20 LP token. The factory exposes `globalPause()` — a pause flag absent from Uniswap V2. All contracts are **fully immutable** (no proxy); EIP-1967 slots return `bytes32(0)` on Factory and Router.

**Base (chain ID 8453): NOT DEPLOYED.** `eth_getCode` returns `0x` for all known Fraxswap addresses on Base. Base is confirmed empty.

Factory addresses are **non-deterministic across chains** (different deployers, different EOA nonces) — each chain has its own distinct factory and router addresses, unlike Uniswap V2's deterministic deployment.

## 1. Topics (chain-agnostic — `topic0 = keccak256(event signature)`)

### 1.1 FraxswapFactory

| topic0 | Event |
|--------|-------|
| `0x0d3648bd0f6ba80134a33ba9275ac585d9d315f0ad8355cddefde31afa28d0e9` | `PairCreated(address indexed token0, address indexed token1, address pair, uint256)` |

Identical signature to Uniswap V2 `PairCreated`. The trailing `uint256` is the new `allPairs.length` (1-indexed). `token0 < token1` enforced. Verified on Ethereum pair deployment history.

### 1.2 FraxswapPair (standard AMM events — identical to Uniswap V2)

| topic0 | Event |
|--------|-------|
| `0x4c209b5fc8ad50758f13e2e1088ba56a560dff690a1c6fef26394f4c03821c4f` | `Mint(address indexed sender, uint256 amount0, uint256 amount1)` |
| `0xdccd412f0b1252819cb1fd330b93224ca42612892bb3f4f789976e6d81936496` | `Burn(address indexed sender, uint256 amount0, uint256 amount1, address indexed to)` |
| `0xd78ad95fa46c994b6551d0da85fc275fe613ce37657fb8d5e3d130840159d822` | `Swap(address indexed sender, uint256 amount0In, uint256 amount1In, uint256 amount0Out, uint256 amount1Out, address indexed to)` |
| `0x1c411e9a96e071241c2f21f7726b17ae89e3cab4c78be50e062b03a9fffbbad1` | `Sync(uint112 reserve0, uint112 reserve1)` |

The `Swap` event signature is **identical to Uniswap V2** — no extra TWAMM parameters. TWAMM virtual swaps do not emit individual `Swap` events per interval; they surface via `VirtualOrderExecution` and update reserves (triggering `Sync`). All four topics confirmed live against Ethereum pair `0x56695c26b3Cdb528815cd22fF7B47510ab821EFd` via `eth_getLogs`.

### 1.4 FraxswapPair (TWAMM-specific events — Fraxswap discriminators)

These events do not exist in plain Uniswap V2 pairs. Filtering on these topic0s will hit exclusively Fraxswap pairs.

| topic0 | Event |
|--------|-------|
| `0x9971294258b76b481032b9c1f7f5594619d7cf40e29e224de9e71481bd0a4f85` | `LongTermSwap0To1(address indexed addr, uint256 orderId, uint256 amount0In, uint256 numberOfTimeIntervals)` |
| `0xe1ce07267c05b1609d3bd4046ca369b74e64cd2b45ee8321ccc79783252c60b4` | `LongTermSwap1To0(address indexed addr, uint256 orderId, uint256 amount1In, uint256 numberOfTimeIntervals)` |
| `0x3c5d5e0947e8b8050cf53e91c7496de2499da1b7613ec86b8fda870578966390` | `CancelLongTermOrder(address indexed addr, uint256 orderId, address sellToken, uint256 unsoldAmount, address buyToken, uint256 purchasedAmount)` |
| `0x43168622ddb54ed84ccad30626ace7077235dc531c67aaf639752c4519535448` | `WithdrawProceedsFromLongTermOrder(address indexed addr, uint256 orderId, address indexed proceedToken, uint256 proceeds, bool orderExpired)` |
| `0x793ee8b0d8020fc042a920607e3cbd37f5132c011786c8dd10a685f4414ed381` | `VirtualOrderExecution(uint256 blockTimestamp, uint256 nextExpiryBlock, uint256 newReserve0, uint256 newReserve1, uint256 token0SalesRate, uint256 token1SalesRate, uint256 token0Earnings, uint256 token1Earnings, uint256 fraxswapFee, uint256 reserve0Virtual)` |
| `0x2ade3fe6cec488ed9b13e36f5179edfbca9998fecc5926cf57335558671d5f95` | `LpFeeUpdated(uint256 fee)` |

**Notes:**
- `LongTermSwap0To1` / `LongTermSwap1To0`: emitted when a user places a TWAMM DCA order (token0→token1 or token1→token0). `numberOfTimeIntervals` controls order duration. `orderId` is the sequence number for this order in the pair. Confirmed live on Ethereum pair `0x56695c…` (block 0xEB4495, tx `0xeb3075fa…`).
- `CancelLongTermOrder`: emitted when the order owner cancels an active order; returns unsold tokens and any proceeds so far. Confirmed live on same pair.
- `WithdrawProceedsFromLongTermOrder`: emitted when a user withdraws accumulated buy proceeds from a live or expired order. `orderExpired = true` means the full order is complete.
- `VirtualOrderExecution`: emitted by `executeVirtualOrders()` each time accumulated virtual orders are settled up to `blockTimestamp`. 10 `uint256` data fields encode the full TWAMM state snapshot (reserves, sales rates, earnings, fee, virtual reserve). Confirmed live on Ethereum pair `0x56695c…` (same blocks as `LongTermSwap0To1`).
- `LpFeeUpdated`: emitted by the factory owner when adjusting the per-pool LP fee (V2 feature). Not emitted on every swap.

### 2.4 FraxswapRouter

The standard router wraps AMM liquidity and swap operations and also exposes `executeVirtualOrders` and TWAMM-aware price queries. TWAMM order placement (`longTermSwapFrom*`) is called directly on the pair, not through the router.

| Selector | Signature | Notes |
|----------|-----------|-------|
| `0xc45a0155` | `factory()` | Immutable — returns the chain-specific factory. Confirmed via `eth_call`. |
| `0xad5c4648` | `WETH()` | Immutable — wrapped native token for this chain. |

## 10. Proxy / upgradeability

All verified: EIP-1967 implementation slot (`0x3608…bbc`) returns `bytes32(0)` on Ethereum Factory and Router. Fraxswap V2 contracts are **fully immutable** — no proxy, no admin, no upgrade path. FraxswapPair contracts are similarly immutable once deployed.
