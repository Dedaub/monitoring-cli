# DODO V2 — Pools, Factories & Templates — Topics, Selectors, Addresses (Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon)

**Status:** verified against live RPC on every listed chain and the canonical `DODOEX/contractV2` repo on 2026-06-02.
**Scope:** the DODO V2 **pool layer** — the PMM pool *templates* (DVM, DPP, DSP, CP, **GSP/GasSavingPool**, plus DPPAdmin / DPPAdvanced / DPPAdvancedAdmin and the shared `CloneFactory`, `FeeRateModel`, `PermissionManager`, ERC-20 token templates) and the *factories* that clone them (DVMFactory, DPPFactory, DSPFactory, CrowdPoolingFactory, UpCpFactory, GSPFactory, ERC20Factory / ERC20V2Factory / ERC20V3Factory) on Ethereum (1), Base (8453), BNB Smart Chain (56), Avalanche C-Chain (43114), Arbitrum One (42161), Optimism (10), Polygon PoS (137). Topics + selectors are chain-agnostic (computed from the canonical signature); addresses are per-chain. **Excludes** routers/proxies (`DODOV2Proxy`, `DPPProxy`, `DSPProxy`, `CpProxy`, `RouteProxy`, `DODOFeeRouteProxy`, `DODOApprove(Proxy)`), the NFT/CrowdFunding/mining stacks, and `DODOMineV3Registry` (covered by the token/mining file).

DODO V2 splits each on-chain pool into two contracts the way Uniswap V3 does not: a **logic template** (one per pool *type* per chain, holding the bytecode) and **per-pair clones** of that template. A clone is an **EIP-1167 minimal proxy** (`0x363d3d373d3d3d363d73‹impl›5af43d82803e903d91602b57fd5bf3`, 45 bytes) `CREATE`-deployed by a factory through the `CloneFactory.clone(address)` helper. The clone holds all pool state; every call `DELEGATECALL`s into the template. **You do not enumerate individual pools** (there are thousands) — you discover them from the factory `NewDVM` / `NewDPP` / `NewDSP` / `NewCP` events (the 4th address field is the pool), and you watch the pool's own `DODOSwap` / `BuyShares` / `SellShares` / `DODOFlashLoan` events thereafter.

The templates and factories are **immutable** — they are *not* EIP-1967/upgradeable proxies. Every template and factory returns `0x` at the EIP-1967 implementation slot (read live; see §9). DODO does not upgrade a pool in place; instead it points a factory's template pointer at a **new** template address for *future* clones via the owner-only `update…Template` setters. **This is the single most important monitoring fact:** the template a factory currently clones is **not** the address printed in DODO's deploy configs / address lists — those list an *older* template. Always read the live `_DVM_TEMPLATE_()` / `_DPP_TEMPLATE_()` / `_DSP_TEMPLATE_()` / `_CP_TEMPLATE_()` off the factory (verified live: on Ethereum the DVMFactory clones `0x3a64ec36…` = `DVM 1.0.3`, not the listed `0x7cA7B5…` = `DVM 1.0.2`; the DSPFactory clones `0x8b09db11…` = `DSP 1.0.1`, not the listed `0x8735AAd3…` = `DSP 1.0.0`). All pool/factory event signatures are shared across template versions, so the topic0s below hold regardless of which template version is live.

| Family | Role | Deploy pattern | Verified name / `version()` |
|--------|------|----------------|------------------------------|
| **DVM** (DODOVendingMachine) | Public PMM pool — anyone can `buyShares`/`sellShares`; LP-funded, no owner reset. | **EIP-1167 clone** of the DVM template, by DVMFactory | `DVM 1.0.2` / `DVM 1.0.3` |

| **CP** (CrowdPooling) | Fixed-price crowd-pooling pool: bid → settle → vest → becomes a DVM. | **EIP-1167 clone** of the CP template, by CrowdPoolingFactory **and** UpCpFactory | `CP 1.0.0` (and CP-vesting variants) |

| **DPPAdvanced / DPPAdvancedAdmin** | Newer DPP variant + its admin (oracle-aware private pool). **Optimism has them as listed templates**; on ETH/AVAX/MATIC the *standard* DPPFactory already clones the Advanced template. | clone | `DPP Advanced 1.0.0` |
| **CloneFactory** | `clone(address)` → deploys an EIP-1167 minimal proxy of any prototype. Used by every pool factory. | plain immutable contract (280 B, identical bytecode every chain) | `ICloneFactory` / `CloneFactory` (solc 0.6.9) |
| **FeeRateModel** | `getFeeRate(trader)` → the mt-fee a pool charges; pools hold a pointer to it (`_MT_FEE_RATE_MODEL_`). | plain immutable contract | `FeeRateModel` |

`UpCpFactory` ("ascending crowd-pooling") clones the **same CP template** as CrowdPoolingFactory but with a different price schedule; it emits `NewCP` too. Its pools are CP clones.

**GSP** is sourced from a **separate** canonical repo, GitHub `DODOEX/dodo-gassaving-pool` (`contracts/GasSavingPool/impl/{GSP,GSPTrader,GSPFunding,GSPVault,GSPStorage}.sol`; the same files are mirrored into `DODOEX/contractV2` under `contracts/DODOGasSavingPool/`). A `GSPFactory` (emitting `NewGSP` / `CreateGSP`, holding a `_GSP_TEMPLATE_()` pointer) exists in DODO's deployments **only on non-target chains** (e.g. X-Layer: GSPFactory `0x5e132c0aBB9e4E5f1471eF6d1C9740c276Fe520e`, GSP template `0x668D4eD054aa62d12F95a64aa1C7e2791f176D0e`, both verified live `GSP 1.0.1`). On the **7 target chains the named GSPFactory is not in the registry** (the DODO contract API returns zero GSP entries on all 25 listed chains; the X-Layer GSPFactory address returns `0x` from `eth_getCode` on every target chain) — GSP pools that do exist are **discovered as EIP-1167 clones**, deployed directly by their pool admin rather than by a permissionless factory.

## 1. Topics (chain-agnostic — `topic0 = keccak256(event signature)`)

### 1.1 Pool trade core — DVM / DPP / DSP / **GSP** (and CP after it converts) all emit these

Inherited from `DVMTrader` / `DPPTrader` / `DSPTrader` / `GSPTrader` + the `*Funding` modules; **byte-identical signatures across all pool types**, so a single set of topic0s covers every PMM pool. **GSP reuses the exact same `DODOSwap` / `DODOFlashLoan` / `BuyShares` / `SellShares` / `RChange` / `LpFeeRateChange` topic0s** as DSP — recomputed locally from the 0.8.16 `GSPTrader`/`GSPFunding`/`GSPVault` source and **confirmed live** on the Ethereum GSP clone (`DODOSwap` 0xc2c0245e… and `BuyShares` 0x1c172440… both fired, ≥7 / 1 events). The *only* pool-event caveat for GSP: its `Transfer`/`Approval` (LP-share ERC-20) declare the address fields `indexed` (the topic0 is unchanged — `indexed` never affects the hash). **None of the trade/funding fields are `indexed`** — all args live in `data`.

### 1.6 GSP (GasSavingPool) — admin & LP events on top of the shared trade core

### 2.8 GSP (GasSavingPool) — trade/funding shared, plus the distinguishing admin surface

GSP reuses the **same swap/funding/view selectors** as the DVM/DSP family (`sellBase 0xbd6015b4`, `sellQuote 0xdd93f59a`, `flashLoan 0xd0a494e4`, `buyShares 0x4c85b425`, `sellShares 0xb56ceaa6`, `querySellBase 0x79a04876`, `querySellQuote 0x66410a21`, `getVaultReserve 0x36223ce9`, `getPMMState 0xa382d1b9`, `getPMMStateForCall 0xfd1ed7e9`, `getUserFeeRate 0x44096609`, `_BASE_TOKEN_ 0x4a248d2a`, `_QUOTE_TOKEN_ 0xd4b97046`, `_LP_FEE_RATE_ 0xab44a7a3`, `_MAINTAINER_ 0x4322ec83`) — so a swap-side monitor needs no GSP-specific change. The table below is the **GSP-distinguishing** surface (admin price/fee control + the different `init`). **All present-checked in the live ETH GSP template bytecode `0x78d43a88…`.**

| Selector | Signature | Returns / notes |
|----------|-----------|-----------------|
| `0x72bf079e` | `adjustPrice(uint256 i)` | **The GSP fingerprint.** `onlyAdmin` — moves the pool's oracle price `_I_`; emits `IChange`. **No DVM/DPP/DSP has this** — its presence in a pool's bytecode flags GSP. **verified present in template code.** |
| `0xe6485c8e` | `adjustPriceLimit(uint256 priceLimit)` | `onlyAdmin` — caps the per-move price deviation (`_PRICE_LIMIT_`, 1e6 decimals, default 1e3). No event. |
| `0xb59b8555` | `adjustMtFeeRate(uint256 mtFeeRate)` | `onlyMaintainer` — sets `_MT_FEE_RATE_`; emits `MtFeeRateChange`. |
| `0x996cd0c8` | `adjustLpFeeRate(uint256 lpFeeRate)` | `onlyMaintainer` — sets `_LP_FEE_RATE_`; emits `LpFeeRateChange`. |
| `0xc61e2567` | `adjustK(uint256 k)` | `onlyMaintainer` — sets curve param `_K_`; emits `KChange`. |

| `0x039658d1` | `init(address maintainer, address admin, address baseTokenAddress, address quoteTokenAddress, uint256 lpFeeRate, uint256 mtFeeRate, uint256 i, uint256 k, uint256 priceLimit, bool isOpenTWAP)` | One-shot clone initializer. **10 args** — distinct from DVM/DSP `init` (`0x5039972a`, 8 args, fee-rate-*model* arg) and DPP `init` (`0x01a3c30b`): GSP passes **both `lpFeeRate` and `mtFeeRate` as raw uint256** plus an `admin` and a `priceLimit`. **verified present in template code.** |
| `0xd505accf` | `permit(address owner, address spender, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s)` | EIP-2612 permit on the GLP share token (GSP adds `DOMAIN_SEPARATOR`/`nonces`; DSP/DVM LP shares have no permit). |

## 8. Pool discovery & the clone mechanism

You never list pools; you derive them.

1. **Subscribe to the factory `NewXXX` events once per chain** (`NewDVM` from DVMFactory, `NewDPP` from DPPFactory, `NewDSP` from DSPFactory, `NewCP` from **both** CrowdPoolingFactory and UpCpFactory). The **4th address field** is the new pool; the 1st/2nd are base/quote token, the 3rd is the creator. (All non-indexed → decode from `data`, not topics.)
2. **The pool is an EIP-1167 minimal-proxy clone.** Its runtime bytecode is exactly `0x363d3d373d3d3d363d73` ‖ `‹20-byte template addr›` ‖ `5af43d82803e903d91602b57fd5bf3` (45 B). The embedded 20 bytes **are** the template the factory cloned. *(Verified live: ETH pool `0x235e7876c193da32ecb7bc30f8c543c8cf755b4a` →* `0x363d3d373d3d3d363d73`**`3a64ec3606ff7310e8fad6fcc008e39705fb496d`**`5af43d82803e903d91602b57fd5bf3` *— and `0x3a64ec36…` is exactly `DVMFactory._DVM_TEMPLATE_()`.)*
3. **Identify a pool's type** either by which factory's `NewXXX` minted it, or by reading the embedded template address out of its bytecode and matching it to a known template, or by calling `version()` on the pool (returns `"DVM 1.0.3"`, `"DPP Advanced 1.0.0"`, `"DSP 1.0.1"`, …).
4. **Thereafter watch the pool address itself** for `DODOSwap` / `BuyShares` / `SellShares` / `DODOFlashLoan` / `RChange` / `LpFeeRateChange`. Read `getVaultReserve()`, `getPMMState()`, `_BASE_TOKEN_()`, `_QUOTE_TOKEN_()` against the **pool (clone)** address.
5. **The CREATE2-style off-chain prediction used by Uniswap V3 does NOT apply.** DODO pools are deployed by plain `CREATE` (nonce-based) inside `CloneFactory`, with a salt-less EIP-1167 proxy — so the pool address is **not** derivable from `(tokens, fee)`. You must read it from the `NewXXX` event or the factory registry (`getDODOPool` / `getCrowdPooling` / `_REGISTRY_`).

## 10. Detection invariants & gotchas

1. **`DODOSwap` is the universal swap event** for every DODO V2 pool type (DVM/DPP/DSP and a converted CP). Filter `topic0 = 0xc2c0245e…` + restrict to known DODO pool addresses (from `NewXXX` history) to avoid confusing it with other protocols. `fromToken`/`toToken` carry direction; amounts are unsigned.
2. **Live templates ≠ listed templates.** The address a factory clones (`_DVM_TEMPLATE_()` etc.) is frequently **newer** than DODO's published deploy-config / address list. On Ethereum the DVMFactory clones `DVM 1.0.3` (`0x3a64ec36…`) while the list shows `DVM 1.0.2` (`0x7cA7B5…`); the DSPFactory clones `DSP 1.0.1` not the listed `DSP 1.0.0`. **Always read the factory pointer.** This is the #1 source of "my template allow-list is wrong."
3. **A DODO pool's logic address lives in its bytecode, not in an EIP-1967 storage slot.** To learn a clone's template, read its 45-B code and slice bytes 10..29 (`0x363d3d373d3d3d363d73` is 10 bytes), or just match the whole runtime. Reading the EIP-1967 impl slot returns `0x` — that is expected, **not** a missing proxy.
4. **DPP has no `buyShares`/`SellShares` and emits no LP `BuyShares`/`SellShares`** — it's a single-owner private pool. Its distinctive events are `RChange` + `LpFeeRateChange` (from owner `reset`). DVM emits neither `RChange` nor `reset`. DSP emits both LP shares **and** `RChange`.
5. **CP/UpCp pools change behavior over their lifecycle.** Pre-settlement they emit `Bid`/`Cancel`/`Settle` and have no PMM trading; post-settlement they convert to a DVM and start emitting `DODOSwap`/`BuyShares`. Don't assume a `NewCP` address is a tradable pool from block 0.
6. **`NewCP` is emitted by two different factories** (CrowdPoolingFactory and UpCpFactory). To attribute a CP to fixed-price vs ascending, key on **which factory** emitted it, not the event alone.
7. **Pool addresses are NOT CREATE2-deterministic.** Unlike Uniswap V3, you cannot compute a DODO pool address from `(token0, token1, fee)`. Clones are nonce-`CREATE`'d inside `CloneFactory` with a salt-less EIP-1167 proxy. Discover via `NewXXX` or the factory registry getters.
8. **The pool's LP token is the pool itself** (DVM/DSP/CP). ERC-20 `Transfer`/`Approval` with the pool as the emitting contract are LP-share moves, not the traded base/quote tokens. The traded tokens' `Transfer`s come from the token contracts.
9. **Pool state must be read against the clone, not the template.** `eth_call`ing `getVaultReserve()`/`getPMMState()` on a template returns garbage/zeros (template storage is uninitialized). Use the pool address from `NewXXX`.
10. **DPP's `init` arg order differs from DVM/DSP** (`owner` first; `k` before `i`). If you decode factory init calldata, use the right ABI per pool type — the selectors differ (`0x01a3c30b` DPP vs `0x5039972a` DVM/DSP) so you can tell them apart.
11. **On ETH/AVAX/MATIC the standard DPPFactory already clones the "DPP Advanced" template** (`version() == "DPP Advanced 1.0.0"`) even though those chains don't list a separate `DPPAdvanced` template. Optimism lists DPPAdvanced/DPPAdvancedAdmin explicitly. Treat "DPP" and "DPP Advanced" as the same pool *family* for `DODOSwap` detection — the trade events are identical.
12. **`RChange(uint8)` not `RChange(uint256)`** — the param is the `PMMPricing.RState` enum, which ABI-encodes as `uint8`. Using `uint256` yields the wrong topic0.
13. **Base list points DSP at `0xA2DDEF1a…`, which has `0x` bytecode on Base** (it's BNB/Avalanche's DSP template address). The real Base DSP template is `0x8319b708…` (read off `DSPFactory._DSP_TEMPLATE_()`). Trusting the list blind misses every Base DSP pool.
14. **`getFeeRate` exists in two arities.** The pool-facing `FeeRateModel.getFeeRate(address)` (`0x8198edbf`) delegates to a newer impl's `getFeeRate(address,uint256)` (`0xee75718a`) on chains with a `FeeRateImpl` (Optimism). When probing, prefer the single-arg form on the FeeRateModel itself.
15. **GSP looks like DSP on the swap event but is a different contract — disambiguate by bytecode / `version()` / the `adjustPrice` selector, never by `DODOSwap` alone.** A GSP pool emits the **identical** `DODOSwap` (`0xc2c0245e…`), `BuyShares`, `SellShares`, `RChange`, `LpFeeRateChange`, `DODOFlashLoan` topic0s as DSP (verified live on the ETH GSP clone). To tell them apart: (a) call `version()` — GSP returns `"GSP 1.0.1"`, DSP `"DSP 1.0.x"`; (b) check the clone's embedded EIP-1167 impl against the GSP template (ETH `0x78d43a88…`); (c) probe `adjustPrice(uint256)` `0x72bf079e` or `_ADMIN_()` `0x63ab5993` — **present only on GSP**; (d) read `_MT_FEE_RATE_MODEL_()` — GSP returns `0x0` (DSP/DVM return a real FeeRateModel). The GSP template is **solc 0.8.16** vs DSP's 0.6.9.
16. **GSP can move its own price via admin — a monitoring signal DSP/DVM/DPP lack.** GSP's `_ADMIN_` can call `adjustPrice(i)` (within `_PRICE_LIMIT_`) to reprice the pool, emitting **`IChange(uint256)`** (`0x0caf6249…`). Watch this topic on GSP pools: a price move that is **not** the result of a trade (no accompanying `DODOSwap`) is an admin oracle update, unique to GSP. DSP repegs only through `RChange` driven by trades; it has no admin price lever. Maintainer-side, GSP also emits `MtFeeRateChange`/`KChange`/`WithdrawMtFee` that DSP does not.
17. **GSP pools on the 7 target chains are not factory-enumerable.** Unlike DVM/DPP/DSP/CP (discover via `NewXXX`), there is **no `NewGSP` to subscribe to** on any target chain — no GSPFactory is deployed (the named factory exists only on non-target chains like X-Layer). Discover GSP pools by matching the GSP template in a contract's EIP-1167 code, by `version() == "GSP 1.0.1"`, or by the presence of `adjustPrice`/`_ADMIN_`. On Ethereum the one known GSP pool is `0x6e2002a6…`; its `_ADMIN_` is also its deployer (no permissionless factory step).
