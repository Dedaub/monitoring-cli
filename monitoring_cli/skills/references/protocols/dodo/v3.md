# DODO V3 (D3 / D3MM + D3Vault) — Topics, Selectors, Addresses (Ethereum, BNB, Avalanche, Arbitrum, Optimism, Polygon)

**Status:** verified against live RPC on every listed chain and the canonical `DODOEX/dodo-v3` repo on 2026-06-02.
**Scope:** the entire DODO V3 / "D3" generation — the shared lending **D3Vault**, the **D3MM** maker pools (+ paired **D3Maker** price controllers) cloned by **D3MMFactory**, the user-entry **D3Proxy**, and the periphery (`D3Oracle`, `D3RateManager`, `D3MMLiquidationRouter`, `D3UserQuota`, `D3PoolQuota`, `D3FeeRateModel`) plus the clone **templates** (`D3MMTemplate`, `D3MakerTemplate`, `D3TokenTemplate`). Topics + selectors are **chain-agnostic** (computed from the canonical signature); addresses are network-specific. Deployed on Ethereum (1), BNB Smart Chain (56), Avalanche C-Chain (43114), Arbitrum One (42161), Optimism (10), Polygon PoS (137). **NOT deployed on Base (8453)** — see §7 and §10. **This is a different protocol from DODO V1/V2:** D3 is a *decentralized market-making + margin-lending* system, **not** a classic PMM AMM. There is no `DODOSwap`/`NewDVM`/`buyShares` here — the swap event is `Swap(...)`, pool creation is `D3Birth(...)`, and the unique monitoring surface is the **borrow / collateral / liquidation** lifecycle on the D3Vault.

DODO V3 is two halves. **D3Vault** is a shared **lending pool**: liquidity providers `userDeposit` a token and receive a `D3Token` ("dToken") interest-bearing receipt (exchange-rate accrual, no rebasing); **D3MM market-maker pools BORROW** from the vault (`poolBorrow`) to make markets. The vault tracks `totalBorrows`, a per-token `borrowIndex`, compound interest, per-pool collateral, and **liquidations** — a pool whose collateral ratio falls below the maintenance margin (`MM`) is liquidated, either permissionlessly (`liquidate`) or, for bad debt, by the DODO team via `startLiquidation → liquidateByDODO → finishLiquidation` routed through `D3MMLiquidationRouter`. **D3MM pools** are per-maker pools (EIP-1167 clones of `D3MMTemplate`) each paired with a **D3Maker** clone holding the maker's packed price/amount/fee config; the pool borrows from the vault, quotes via a PMM range-order model, and serves swaps (`sellToken`/`buyToken`).

Like DODO V2, every D3MM pool is an **EIP-1167 minimal-proxy clone** (`0x363d3d373d3d3d363d73‹impl›5af43d82803e903d91602b57fd5bf3`, 45 bytes) created — together with its D3Maker clone — by `D3MMFactory.breedD3Pool(...)` through the **same `CloneFactory`** DODO V2 uses (`0x5e5a7b76…` on Ethereum; not in the D3 registry — read it off the factory). The dToken is likewise an EIP-1167 clone of `D3TokenTemplate`, minted by the vault. **You do not enumerate pools** — discover them from the factory `D3Birth(address d3MM, address creator)` event (or `D3Vault.AddPool`), then watch the pool's own `Swap` / `MakerDeposit` / `MakerWithdraw` and the vault's `PoolBorrow` / `PoolRepay` / `Liquidate`. **All D3 singletons and periphery are plain immutable contracts — not EIP-1967/UUPS proxies** (impl slot reads `0x0` on every one, every chain). `breedD3Pool` is **owner-gated** (permissioned pool creation), unlike V2's permissionless factories.

**Single protocol admin** across the core on each chain: `0x95c4f5b83aa70810d4f142d58e5f7242bd891cb0` — `owner()` of D3Vault (OpenZeppelin `Ownable`) **and** `_OWNER_()` of D3MMFactory (DODO `InitializableOwnable`), and the pools' `maintainer` (fee recipient). Verified live on Ethereum.

The factory keeps template *mappings* (`_D3POOL_TEMPS[type]`, `_D3MAKER_TEMPS_[type]`) so future types/upgrades repoint per index — exactly like V2's `update…Template`, and with the **same live-vs-listed caveat** (see §9, §10): older pools embed an *earlier* template address than the one the factory clones now, even though both report `D3MM 1.0.0`.

### 1.5 Ownership (two distinct models)

D3 mixes **OpenZeppelin `Ownable`** (D3Vault, D3RateManager, D3UserQuota, D3PoolQuota) with DODO's **`InitializableOwnable`** (D3MMFactory, D3MM, D3Maker, D3Oracle, D3FeeRateModel, D3Token). `OwnershipTransferred` shares topic0; the two-step *prepare* event differs (and D3's `InitializableOwnable` makes both args **indexed**, unlike DODO V2 which left them non-indexed — but `indexed` does not change topic0, so the hashes match V2).

## 8. Pool discovery & the clone mechanism

You never list pools; you derive them.

1. **Subscribe to `D3MMFactory.D3Birth` once per chain** (topic0 `0x9ee893b7…`). The **1st address field is the new D3MM pool**, the 2nd is the creator. The same tx fires **`D3Vault.AddPool(pool)`** (topic0 `0x8c82d670…`) — either is a valid discovery anchor. Pool creation is **owner-gated** (`breedD3Pool` is `onlyOwner`), so pools appear only from the protocol team.
2. **The pool is an EIP-1167 minimal-proxy clone.** Runtime = `0x363d3d373d3d3d363d73` ‖ `‹20-byte template›` ‖ `5af43d82803e903d91602b57fd5bf3` (45 B). The paired **D3Maker** is also a clone (read it from `D3MM.getD3MMInfo()` → `maker`). *(Verified live: ETH pool `0x7bce750a34f69d33ac785c04fbc87d6455c19453` is a 45-B clone embedding `0xa376762070…`; its maker `0xebdc2e5d…` is a clone embedding `0x5e1251f4…`.)*
3. **Identify a D3MM** by calling `version()` (`0x54fd4d50` → `"D3MM 1.0.0"`) or `_CREATOR_()`/`getD3MMInfo()` on the pool, or by matching the embedded template against `D3MMFactory._D3POOL_TEMPS(0)`.
4. **Thereafter watch the pool** for `Swap` / `MakerDeposit` / `MakerWithdraw`, the **paired D3Maker** for `SetNewToken` / `SetPoolInfo`, and the **vault** for that pool's `PoolBorrow` / `PoolRepay` / `Liquidate` / `StartLiquidation` / `FinishLiquidation`. Read pool inventory via `getTokenReserve`, and the pool's health via `D3Vault.getCollateralRatio(pool)` / `checkCanBeLiquidated(pool)`.
5. **Pool addresses are NOT CREATE2-deterministic.** As in V2, clones are nonce-`CREATE`'d inside `CloneFactory` (salt-less EIP-1167) — you cannot compute a pool address; discover it from `D3Birth`/`AddPool`.

## 10. Detection invariants & gotchas

1. **D3's swap event is `Swap`, not `DODOSwap`** (topic0 `0xb24b74…`, **8 non-indexed fields**). `sellOrNot` (last field) is `0` for `sellToken`, `1` for `buyToken`. Restrict to known D3MM pool addresses (from `D3Birth`) before keying on topic0 — `Swap(...)` is a common name.
2. **Pool creation is `D3Birth(d3MM, creator)` (no token fields), not `NewD3MM`/`NewD3Pool`.** Pair it with `D3Vault.AddPool(pool)` (same tx). Tokens enter the pool *later* (D3Maker `SetNewToken` + `MakerDeposit`), so a fresh `D3Birth` address is not yet a tradable pair.
3. **The unique D3 monitoring surface is the lending/liquidation lifecycle on the D3Vault**, not the swap: `UserDeposit`/`UserWithdraw` (LP), `PoolBorrow`/`PoolRepay` (leverage), and the liquidation triplet. A pool with `getCollateralRatio(pool) < 1e18 + MM` is liquidatable. **Two liquidation paths:** (a) permissionless `liquidate(...)` → `Liquidate` event; (b) DODO-team bad-debt path `startLiquidation → liquidateByDODO → finishLiquidation` → `StartLiquidation` then `FinishLiquidation` (note: `liquidateByDODO` itself emits **no** event — bracket it with the Start/Finish pair). Watch `StartLiquidation`/`FinishLiquidation` as the high-severity alert.
4. **`PoolRepay` fires on more than user repays** — `poolRepayAll` and the pool-removal flow (`removeD3Pool` → `pendingRemovePoolRepayAll`) also emit it. Likewise `AddPool` fires both from `addD3Pool` (owner) and `addD3PoolByFactory` (inside `breedD3Pool`).
5. **D3Proxy emits no events.** A user swap shows up as `Swap` on the *pool*; an LP deposit as `UserDeposit`/`Mint` on the *vault*/*dToken*. Don't expect proxy-level logs. The proxy `multicall` (`0xac9650d8`) wraps these (e.g. ETH→WETH then swap then refund).
6. **`AddRouter`/`RemoveRouter` share topic0 across D3Vault and D3MMFactory** (`0x45d58bd7…` / `0xe3ef6300…`). Disambiguate by the emitting address. The vault's `AddRouter`/`AddLiquidator` allow-lists are security-critical (they gate who can run DODO-team liquidations).
7. **Two ownership models coexist.** D3Vault / D3RateManager / D3UserQuota / D3PoolQuota are **OZ `Ownable`** → read `owner()` (`0x8da5cb5b`); `_OWNER_()` returns nothing. D3MMFactory / D3MM / D3Maker / D3Oracle / D3FeeRateModel / D3Token are **DODO `InitializableOwnable`** → read `_OWNER_()` (`0x16048bc4`); OZ `owner()` **reverts**. Both resolve to the same admin `0x95c4f5b8…`. The InitOwnable handoff is two-step (`transferOwnership`→`claimOwnership`, events `OwnershipTransferPrepared`→`OwnershipTransferred`).
8. **Live template ≠ a pool's embedded template.** `D3MMFactory._D3POOL_TEMPS(0)` is what the factory clones **now** (the registry D3MMTemplate); an *older* pool embeds an *earlier* template address (e.g. ETH pool `0x7bce750a…` embeds `0xa376762070…`, not the listed `0xFeAFe253…`) — both report `version()="D3MM 1.0.0"`. To learn a clone's logic, slice bytes 10..29 of its 45-B runtime; to learn what *new* pools use, read the factory mapping. Same caveat for D3Maker/dToken.
9. **The CloneFactory is not in the D3 registry** — it is the **same contract DODO V2 uses** (`0x5e5a7b76…` on Ethereum). Read it off `D3MMFactory._CLONE_FACTORY_()` or `D3Vault._CLONE_FACTORY_()` per chain; don't assume a D3-specific deploy.
10. **Address ≠ role across chains.** Many registry addresses are reused for *different* D3 roles on different chains and *different DODO versions* (examples in §5/§6: `0x49186E32…` = ETH D3Vault **and** Avax D3MMTemplate; `0xFeAFe253…` = ETH D3MMTemplate **and** Polygon D3MMFactory; `0x6de4d882…` = OP D3MMFactory **and** Base's DODOApproveProxy in V2). **Key every lookup on `(chainId, address)`.**
11. **Not on Base.** Every D3-defining address is `0x` or an unrelated DODO V2 contract on Base (§7). A non-empty `eth_getCode` on Base for a "D3" address is a coincidence, not a deployment.
12. **dToken (`D3Token`) is a per-token ERC-20 clone**, symbol `d3<UNDERLYING_SYMBOL>`, owner = the vault, `decimals()` mirrors the underlying. Its `Transfer`/`Approval` (with the dToken as emitter) are *receipt-token* moves; `Mint`/`Burn` track LP supply/withdraw. The underlying tokens' `Transfer`s come from their own contracts. A small `DEFAULT_MINIMUM_DTOKEN` (1000) is permanently locked to `address(1)` on first deposit (anti-drain) — expect a one-off mint to `0x…0001`.
13. **Swaps are heartbeat-gated and callback-pull.** `sellToken`/`buyToken` revert if the maker's price is stale (`checkHeartbeat`) and require the caller to implement `d3MMSwapCallBack` to fund the trade — the standard path is through `D3Proxy`. The pool also enforces `checkSafe()` (IM) after each trade.
14. **Interest accrues on the dToken exchange rate, not by rebasing.** `getExchangeRate(token)` rises over time; `borrowIndex` compounds. There is no per-block `Transfer` for accrual — to value an LP position read `getExchangeRate` × dToken balance.
