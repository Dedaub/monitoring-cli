# Circle CCTP v1 — Topics, Selectors, Addresses (Ethereum, Avalanche, Optimism, Arbitrum, Base, Polygon PoS; not BNB)

**Status:** verified against live RPC on Ethereum, Avalanche, Optimism, Arbitrum, Base and Polygon PoS, and the canonical `circlefin/evm-cctp-contracts` repo, on 2026-06-09.
**Scope:** CCTP **v1** — the original `TokenMessenger` / `MessageTransmitter` / `TokenMinter` triad that powers native USDC burn-and-mint between EVM domains (plus non-EVM domains Noble, Solana, Sui, Aptos that fall outside the seven requested chains). Topics and selectors are **chain-agnostic** (keccak of the canonical signature); addresses are **per-chain and NON-deterministic** (different literal on every chain — there is no shared vanity address in v1). Of the seven requested chains, v1 is live on six: Ethereum, Avalanche, Optimism, Arbitrum, Base, Polygon PoS. **It is NOT on BNB Smart Chain** (BNB only ever got v2 contracts — see [v2.md](v2.md)).

CCTP v1 moves USDC across chains by **burning** it on the source domain and **minting** an equal amount on the destination domain, gated by an off-chain attestation signed by Circle's attester set. The flow is two transactions on two chains: `TokenMessenger.depositForBurn(...)` on the source emits `DepositForBurn` and (via `MessageTransmitter.sendMessage`) `MessageSent(bytes)`; an off-chain service fetches the attestation; anyone then calls `MessageTransmitter.receiveMessage(message, attestation)` on the destination, which emits `MessageReceived` and dispatches to `TokenMessenger.handleReceiveMessage` → `TokenMinter.mint` → `MintAndWithdraw`. **The user-relevant amount/recipient lives in the message body (ABI-packed bytes), not in indexed topics** — see §6.

> **Critical correction to the common assumption — v1 is NOT an upgradeable proxy on these chains.** The repo ships an `AdminUpgradableProxy` (a fork of OZ `TransparentUpgradeableProxy`, EIP-1967 slots), but on **every** mainnet the canonical v1 addresses below are **direct, non-proxied deployments**: `eth_getCode` returns full 8–18 KB runtime bytecode and **both** the EIP-1967 implementation slot `0x360894…bbc` **and** admin slot `0xb53127…6103` read `0x0` (also confirmed empty for the legacy zos slots). `implementation()`/`admin()` revert (calls are not delegated). So there is **no `Upgraded(address)` event to watch on v1** and no impl pointer to track — the bytecode at the address is the implementation. (v2, by contrast, *is* a transparent proxy — see [v2.md](v2.md).)

## 6. Detection invariants & gotchas

1. **A transfer is two events on two chains, joined by the message.** Source: `DepositForBurn` (TokenMessenger) + `MessageSent(bytes)` (MessageTransmitter). Destination: `MessageReceived` (MessageTransmitter) + `MintAndWithdraw` (TokenMessenger). **The join key is the message hash / `(sourceDomain, nonce)`** — `nonce` is `uint64` indexed in both `DepositForBurn` (topic1) and the source-chain message, and re-appears in `MessageReceived` (topic2, indexed).
2. **`amount` is NOT indexed** in `DepositForBurn` or `MintAndWithdraw` — decode it from `data`. Indexed fields are `(nonce, burnToken, depositor)` for `DepositForBurn` and `(mintRecipient, mintToken)` for `MintAndWithdraw` (verified live).
3. **`depositor` ≠ tx sender, and `mintRecipient` is the *intended* recipient encoded as `bytes32`.** Routers and aggregators (LI.FI, Wormhole, Across-style relayers) call `depositForBurn` on the user's behalf; attribute by the `depositor`/`mintRecipient` fields, not `tx.from`. `mintRecipient`/`destinationCaller` are **left-padded bytes32** because CCTP also targets non-EVM domains.
4. **`MessageSent(bytes)` has the same topic0 on v1 and v2** (`0x8c52…b036`). Disambiguate strictly by emitting contract: v1 MessageTransmitter (e.g. ETH `0x0a99…`) vs v2 (`0x81d4…`). The message header's version byte (0 = v1, 1 = v2) inside the body is the secondary tell.
5. **v1 is non-upgradeable here** — there is no `Upgraded(address)` to watch and no impl slot to read. If you see EIP-1967 slot activity, you are looking at a v2 contract.
6. **`receiveMessage` can be called by anyone** (unless `destinationCaller` was set). The `caller` in `MessageReceived` (indexed) is whoever submitted the attestation — often a Circle/relayer EOA, not the recipient. Do not attribute the mint to `caller`.
7. **Burn caps are per-token, per-message** (`burnLimitsPerMessage`) — a spike in `SetBurnLimitPerMessage` is a config/risk signal. There is no global TVL cap event.
8. **`MessageReceived` carries arbitrary CCTP messages, not only USDC.** A `MessageReceived` with no following `MintAndWithdraw` is a non-token message (e.g. a generic cross-chain call). Filter for token transfers by requiring the paired `MintAndWithdraw` on the same tx.
9. **Pause is a real kill-switch.** `Pause()` on MessageTransmitter halts `receiveMessage`; on TokenMessenger/TokenMinter halts burns/mints. Monitor `Pause`/`Unpause` and `PauserChanged` as incident signals. (ETH MessageTransmitter live = not paused.)
10. **The same address is BOTH source and destination.** A chain runs one TokenMessenger that emits `DepositForBurn` for outbound and `MintAndWithdraw` for inbound — do not assume a log is outbound just because it's on the TokenMessenger.
