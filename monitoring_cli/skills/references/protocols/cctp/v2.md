# Circle CCTP v2 — Topics, Selectors, Addresses (Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon PoS)

**Status:** verified against live RPC on all seven listed chains and the canonical `circlefin/evm-cctp-contracts` repo, on 2026-06-09.
**Scope:** CCTP **v2** — the "fast transfer" generation (`TokenMessengerV2` / `MessageTransmitterV2` / `TokenMinterV2`, plus the `MessageV2` library) adding soft-finality (faster-than-finality) transfers, on-chain destination **fees**, and optional execution **hooks**. Topics and selectors are **chain-agnostic** (keccak of the canonical signature); addresses are network-specific but here **v2 uses ONE shared vanity address per contract on every chain** (CREATE2/deterministic deploy). **v2 is live on all seven requested chains, including BNB** (the only generation present on BNB). Note: BNB has the v2 contracts but **no live USDC `DepositForBurn` activity** (BNB is "USYC-only" — §6).

CCTP v2 keeps the burn-and-mint model of [v1](v1.md) but: (1) splits message handling into **finalized** vs **unfinalized** paths (`handleReceiveFinalizedMessage` / `handleReceiveUnfinalizedMessage`) gated by a `minFinalityThreshold`, enabling fast transfers attested before source finality; (2) charges an on-chain **fee** taken from the minted amount (`MintAndWithdraw` now carries `feeCollected`; `DepositForBurn` carries `maxFee`); (3) supports an optional **`hookData`** blob for post-mint execution on the destination. The contracts are **EIP-1967 transparent proxies** (unlike v1's direct deployments).

> **The deployed `TokenMessengerV2` impl predates the master-source "min-fee controller" feature.** The live impl `0x555e2725…` (shared by ETH/Base/Avax/Arb/OP/Polygon) exposes `feeRecipient()`/`setFeeRecipient()` and emits `FeeRecipientSet`, but **`minFee()`/`minFeeController()`/`setMinFee()`/`setMinFeeController()` and the `MinFeeSet`/`MinFeeControllerSet` events are ABSENT** from the deployed bytecode (PUSH4 scan: present-in-source, absent-on-chain). Treat the min-fee surface as **not-yet-deployed**; alert on `FeeRecipientSet` and the `maxFee`/`feeCollected` fields, not on `MinFeeSet`. On **BNB** the `TokenMessengerV2` impl is a *different* build (`0x1ccafdff…`) — re-scan its selectors before assuming parity.

## 6. Detection invariants & gotchas

1. **A v2 transfer is two events on two chains.** Source: `DepositForBurn` (TokenMessengerV2) + `MessageSent(bytes)` (MessageTransmitterV2). Destination: `MessageReceived` (MessageTransmitterV2) + `MintAndWithdraw` (TokenMessengerV2). **Join key = the 32-byte `nonce`** (indexed `bytes32` in v2 `MessageReceived`; also the dedupe key in `usedNonces`). v2's nonce is a **hash**, not a counter.
2. **v2 topic0s differ from v1's for the workhorses.** `DepositForBurn`: v2 `0x0c8c…30a5` vs v1 `0x2fa9…43c0`. `MintAndWithdraw`: v2 `0x50c5…ab63` vs v1 `0x1b2a…42e6`. `MessageReceived`: v2 `0xff48…db1c` vs v1 `0x5820…cc5d`. **Only `MessageSent(bytes)` (`0x8c52…b036`) is shared** — disambiguate by emitting contract.
3. **`maxFee`/`feeCollected` are real.** v2 `DepositForBurn.maxFee` is the user's fee ceiling; the actual fee taken on the destination is `MintAndWithdraw.feeCollected` (and the split goes to `feeRecipient`, ETH = `0x6efa3205…7bee`). **The minted-to-user amount is `amount − feeCollected`** — do not treat `DepositForBurn.amount` as net received.
4. **Indexed topic3 of v2 `DepositForBurn` is `minFinalityThreshold`, NOT a nonce** (verified live: `0x7d0` = 2000 = finalized). Low values = fast/unfinalized. There is **no `nonce` field in v2 `DepositForBurn`** (unlike v1).
5. **Finalized vs unfinalized paths.** `handleReceiveFinalizedMessage` vs `handleReceiveUnfinalizedMessage` distinguish standard (post-finality) from fast transfers. `MessageReceived.finalityThresholdExecuted` (indexed) tells you which path executed. Fast transfers settle before source finality — a monitor watching for "confirmed" transfers should look at `finalityThresholdExecuted`.
6. **The deployed messenger impl lacks the min-fee surface.** `MinFeeSet`/`MinFeeControllerSet` events and `setMinFee`/`minFee` selectors are in the master source but **absent from the live impl** `0x555e2725…`. Do not alert on them; they would only appear after an `Upgraded` rotation. `FeeRecipientSet` IS present.
7. **`depositor` ≠ tx sender; `mintRecipient` is `bytes32` left-padded.** Routers/aggregators call on behalf of users; attribute by event fields. `destinationCaller != 0` restricts who can `receiveMessage` on the destination.
8. **v2 is a transparent proxy — watch `Upgraded(address)` `0xbc7cd75a…2d3b` and `AdminChanged` on the two proxies** (TokenMessengerV2, MessageTransmitterV2). TokenMinterV2 and the MessageV2 library are NOT proxies (no `Upgraded`).
9. **The shared vanity address is a trap.** `0x28b5…cf5d` is the TokenMessengerV2 on ALL seven chains. A log from that address tells you nothing about the chain — key on `(chainId, address)`. The impl behind the MessageTransmitterV2 proxy is unique per chain (domain baked in).
10. **BNB is present but dormant for USDC.** All three v2 contracts exist on BNB (domain 17), but `TokenMessengerV2` emitted **zero `DepositForBurn` in 200k blocks** — Circle lists BNB as "USYC-only." Treat BNB CCTP-USDC alerts as expected-empty; the BNB impl is a different build (`0x1ccafdff…`) so re-scan its selectors.
11. **`receiveMessage` is permissionless** unless gated by `destinationCaller`. The `caller` in `MessageReceived` is the attestation submitter (relayer), not the recipient.
