# Lido V2 — Topics, Selectors, Addresses (Ethereum mainnet)

**Status:** verified 2026-05-29 against Ethereum mainnet via `eth_getLogs` / `eth_getCode` / `eth_getStorageAt` on `https://ethereum-rpc.publicnode.com`, the canonical `lidofinance/lido-dao` source, `docs.lido.fi/deployed-contracts`, and keccak-256 computed locally + cross-checked against live logs.
**Scope:** **Lido on Ethereum is only deployed on Ethereum mainnet (chain ID 1).** Every other chain has only a **bridged `wstETH`** (± a rebasable L2 `stETH`) — see [l2.md](l2.md). Event `topic0` and function `selector` values are **chain-agnostic** (identical wherever a given contract is deployed); addresses are Ethereum-specific.

**What "V2" means.** Lido V2 is the **post-Shapella (May 2023)** architecture: real **withdrawals** (`WithdrawalQueueERC721`), a modular **`StakingRouter`** with pluggable staking modules (Curated / Simple DVT / Community Staking), and a rebuilt oracle stack (`AccountingOracle` + `HashConsensus`, `ValidatorsExitBusOracle`, `OracleReportSanityChecker`) wired together through the **`LidoLocator`** registry. For the pre-withdrawals architecture (LidoOracle push model) see [v1.md](v1.md). For the **stVaults** additions (live in the deployed stETH impl as of 2026) see [v3.md](v3.md).

> **Live-impl note.** The deployed `stETH` and `AccountingOracle` implementations are the **V3 (stVaults)** build (`stETH` impl `0x6ca8…a600`). The *core* V2 topics/selectors below are unchanged by V3; the V3-only additions and the `handleOracleReport` → `processClStateUpdate`/`collectRewardsAndProcessWithdrawals`/`emitTokenRebase` split live in [v3.md](v3.md). Read it alongside this file.

Lido is **liquid staking**: deposit ETH → mint **stETH** (rebasing share token) → optionally wrap to **wstETH** (non-rebasing). `Lido`/`stETH` is a single Aragon-app contract that is *both* the staking pool and the stETH ERC-20. Daily, the oracle reports CL balances → `stETH` rebases (shares constant, balances grow). Withdrawals are async: request → NFT → finalized after oracle → claim ETH. Use **shares** (`TransferShares`, `sharesOf`) for rebase-safe accounting — never store raw stETH balances.

**Two proxy systems coexist.** The **V1-era core** (stETH, NOR, LegacyOracle) are **Aragon** apps — their EIP-1967 impl slot reads **zero**; resolve the impl via `implementation()` (`0x5c60da1b`) or `Kernel.getApp(0xf1f3eb40…namespace, appId)`. The **V2/V3 contracts** use **OssifiableProxy** — read the standard EIP-1967 slot `0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc`. **Always read the live impl slot** — Lido upgrades in place (the deployed stETH/AccountingOracle impls are already the V3 build).

## 5. Detection invariants & gotchas

1. **Index shares, not stETH balances.** stETH rebases daily; `Transfer.value` is the post-rebase amount at emit time. Use `TransferShares` / `sharesOf` / `getSharesByPooledEth` for accounting. wstETH does **not** rebase (it's the canonical store-of-value).
2. **`Submitted` is the stake event; the rebase is `TokenRebased`.** One `TokenRebased` per oracle report (~daily). `ETHDistributed` in the same tx carries the CL delta + EL/withdrawal inflows.
3. **wstETH has no wrap/unwrap event.** Detect wrap/unwrap via paired stETH `Transfer`/`TransferShares` + wstETH `Transfer` from/to `0x0`, or via the `wrap`/`unwrap` selectors (`0xea598cb0` / `0xde0e9a3e`).
4. **Withdrawals are async & NFT-based.** `WithdrawalRequested` → (oracle finalizes) `WithdrawalsFinalized` → `WithdrawalClaimed`. The request is an ERC-721 (`tokenId` = `requestId`); transferring the NFT transfers claim rights. ERC-721 `Transfer` shares topic0 with ERC-20 — key on the WithdrawalQueue address.
5. **Two HashConsensus instances** (Accounting `0xD624…`, VEBO `0x7FaD…`) share all topic0s — disambiguate by address.
6. **Two NodeOperatorsRegistry instances** (Curated `0x5503…`, Simple DVT `0xaE7B…`) share the entire ABI — disambiguate by address; CSM (module 3) has a **different** ABI.
7. **Aragon vs OssifiableProxy.** Don't read the EIP-1967 slot on stETH/NOR/LegacyOracle (it's zero) — use `implementation()`. Do read it on the V2/V3 contracts.
8. **DepositSecurityModule & Burner get redeployed**, not upgraded — there are multiple historical addresses; always resolve the current ones from `LidoLocator`.
9. **`getUserAccountData`-style health doesn't exist** — Lido is staking, not lending. The protocol-health signals are the oracle `TokenRebased` (negative rebase = slashing), `BunkerModeEnabled` (mass-slashing withdrawal throttle), and queue backlog (`unfinalizedStETH`).
10. **Live impl ≠ V2 ABI.** The deployed stETH/AccountingOracle are the V3 build; `handleOracleReport` and the V2 `ReportData` struct are gone. See [v3.md](v3.md) before decoding oracle-report calldata.
