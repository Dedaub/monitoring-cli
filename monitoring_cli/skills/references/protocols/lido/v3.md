# Lido V3 — stVaults (Ethereum mainnet)

**Status:** verified 2026-05-29 against Ethereum mainnet (`eth_getCode` / EIP-1967 slot reads on `https://ethereum-rpc.publicnode.com`), `docs.lido.fi/deployed-contracts`, and the live stETH/AccountingOracle implementation bytecode.
**Scope:** the **V3 ("stVaults") additions** layered on top of the V2 core. This file documents only what V3 *adds or changes* — the stETH base ABI, withdrawals, staking router, oracle stack and addresses are in [v2.md](v2.md) and still apply. **Lido V3 is Ethereum-mainnet only** (no L2 vaults yet — see [l2.md](l2.md)). Topics/selectors are chain-agnostic; addresses are network-specific.

**What V3 adds.** **Staking Vaults** (`stVaults`): isolated, operator-run vaults that can mint stETH against their own validators while keeping custody separated from the core pool. A `VaultHub` connects each vault to the core protocol (minting/burning **external shares**), `OperatorGrid` tiers operators, `PredepositGuarantee` protects deposits, and `LazyOracle` feeds per-vault rates. To support this, the **core stETH contract gained "external shares" accounting** and the **oracle report split + grew a vaults Merkle root** — these are live in the deployed impls *today*, so V3 changes affect anyone decoding current Lido oracle/rebase data even if they never touch a vault.

> **Live everywhere.** Every contract below has on-chain code as of 2026-05-29. The stETH impl (`0x6ca8…a600`) and AccountingOracle impl (`0x1455…141C`) referenced in [v2.md](v2.md) **are** the V3 build.

## 1. Topics — V3-specific (chain-agnostic, `topic0 = keccak256(sig)`)

`abi` = computed from canonical source (not log-confirmed here). For the full core event set (Submitted, TransferShares, TokenRebased, ETHDistributed, withdrawals, oracle, NOR) see [v2.md](v2.md) §1 — those are unchanged.

### 1.1 stETH external-shares (stVaults accounting, on the core stETH contract)

| topic0 | Event | ✓ |
|--------|-------|---|
| `0xee473f96486a2f4b93ccb6729f121223e96975db6e0d6a5ef01f56477e3eab3b` | `ExternalSharesMinted(address indexed receiver, uint256 amountOfShares)` | abi |
| `0xad21467656c56eb8c99f7916faa12f7a657a04d8802100acec62e92451ac5606` | `ExternalSharesBurnt(uint256 amountOfShares)` | abi |

> **External shares** are stETH shares minted *outside* the staking flow, on behalf of connected stVaults via `VaultHub`. They count toward total shares but are backed by vault ETH rather than the pooled buffer — this is why V3 added separate `getExternalShares()` / `getExternalEther()` accounting (§2.1).

### 1.2 VaultHub / StakingVault (best-effort — pull live ABI before alerting)

VaultHub and per-vault `StakingVault`/`Dashboard` emit vault-lifecycle events (connect/disconnect, mint/burn shares against a vault, fee/threshold updates, rebalance, force-rebalance, socialized bad-debt). Signatures vary across the V3 release; the contracts are listed in §3 — **fetch the deployed ABI** rather than relying on guessed topic0s here.

## 2. Function signatures — V3-specific (chain-agnostic)

`bytecode` = confirmed in live implementation bytecode (PUSH4 scan).

### 2.1 stETH — oracle-report split + external shares (replaces V2 `handleOracleReport`)

The monolithic `handleOracleReport` (V2 selector `0xbac3f3c5`) was **removed** and split:

| Selector | Signature | Verify |
|----------|-----------|--------|
| `0x603540cd` | `processClStateUpdate(uint256 _reportTimestamp, uint256 _preClValidators, uint256 _reportClValidators, uint256 _reportClBalance)` | bytecode |
| `0x9271e3e6` | `collectRewardsAndProcessWithdrawals(uint256,uint256,uint256,uint256,uint256,uint256,uint256,uint256)` | bytecode |
| `0x1b250097` | `emitTokenRebase(uint256,uint256,uint256,uint256,uint256,uint256,uint256,uint256,uint256)` | bytecode |
| `0x06f187a4` | `mintExternalShares(address _receiver, uint256 _amountOfShares)` | bytecode |
| `0x72e62e56` | `burnExternalShares(uint256 _amountOfShares)` | bytecode |
| `0xe16a9065` | `getExternalEther()` → `uint256` | bytecode |
| `0x63021d8b` | `getExternalShares()` → `uint256` | bytecode |

> If you decode Lido oracle-report transactions, **`handleOracleReport` no longer exists** — the report is applied via these calls (orchestrated by `AccountingOracle` → `Accounting` → stETH). The classic V2 selector is absent from the deployed bytecode.

### 2.2 AccountingOracle — V3 `ReportData` struct (grew vaults fields)

`submitReportData` selector is now `0x1001587f` with the **V3 vaults `ReportData`**; the V2-struct selector (`0xfc7377cd`) is **absent**:

```
struct ReportData {
  uint256 consensusVersion;
  uint256 refSlot;
  uint256 numValidators;
  uint256 clBalanceGwei;
  uint256[] stakingModuleIdsWithNewlyExitedValidators;
  uint256[] numExitedValidatorsByStakingModule;
  uint256 withdrawalVaultBalance;
  uint256 elRewardsVaultBalance;
  uint256 sharesRequestedToBurn;
  uint256[] withdrawalFinalizationBatches;
  uint256 simulatedShareRate;
  bool isBunkerMode;
  bytes32 vaultsDataTreeRoot;   // <-- V3 addition
  string  vaultsDataTreeCid;    // <-- V3 addition (IPFS CID of per-vault data)
  uint256 extraDataFormat;
  bytes32 extraDataHash;
  uint256 extraDataItemsCount;
}
```

The two `vaultsData*` fields carry a Merkle root + IPFS pointer for per-vault balances, consumed by `LazyOracle`.

## 5. Proxies

Same two-system model as [v2.md](v2.md) §4: **all V3 contracts use OssifiableProxy (EIP-1967)** except the immutable factory/beacon-impl/gateway/verifier logic contracts and the `StakingVault Beacon` (an `UpgradeableBeacon`; individual vaults are beacon proxies pointing at it). Read the live EIP-1967 slot `0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc`; for vaults, read the beacon. Upgrade authority is the Aragon Agent via Dual Governance.

## 6. Detection invariants & gotchas

1. **The V3 build is already live in core.** stETH impl `0x6ca8…a600` and AccountingOracle impl `0x1455…141C` are V3 — so `handleOracleReport` is gone and the oracle `ReportData` carries `vaultsDataTreeRoot`/`vaultsDataTreeCid` *now*, regardless of whether you touch vaults.
2. **External shares ≠ pooled shares.** Total shares = pooled + external. Vault-minted stETH appears via `ExternalSharesMinted`; reconcile supply with `getExternalShares()` / `getExternalEther()`.
3. **Each vault is a beacon proxy.** Don't read the EIP-1967 impl slot on a StakingVault — read the `StakingVault Beacon` (`0x5FbE…a789`). The `StakingVaultFactory` deploys vault + Dashboard pairs; enumerate via VaultHub or factory events.
4. **CSM v2 ≠ v1 addresses.** The repo artifact serves v1; use the on-chain v2 set above (CSVerifier, gates especially changed).
5. **Pectra plumbing.** `TriggerableWithdrawalsGateway` (EIP-7002) and `ValidatorConsolidationRequests` (EIP-7251) change how exits/consolidations flow — and reshape VEBO's event model (verify the live VEBO ABI per [v2.md](v2.md) §1.7).
