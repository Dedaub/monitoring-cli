# Aave V4 — Topics, Selectors, Addresses (Ethereum) — NEW / EVOLVING

**Status:** Aave V4 launched on **Ethereum mainnet 2026-03-30** (announced at EthCC). Addresses parsed from `bgd-labs/aave-address-book` (`AaveV4Ethereum.sol`) and existence-checked on-chain; core Spoke/Hub event topics computed and **verified against live `eth_getLogs` ground truth** on 2026-05-29. **V4 is new and still expanding** — only Ethereum is live so far, more spokes/chains are being added, and struct-bearing event ABIs (Repay/LiquidationCall) should be re-verified against the deployed source before relying on their topic0. Treat this file as a living document.

Aave V4 is a ground-up redesign: a **hub-and-spoke** architecture replacing V3's monolithic Pool.

```
                 ┌───────────────────────────────────────────┐
   suppliers ──▶ │  LIQUIDITY HUB (Core / Plus / Prime)        │  holds & nets all liquidity per asset
                 │  - one shared liquidity pool per asset      │  - emits UpdateAsset (index/rate), fees
                 └───────────────┬───────────────┬─────────────┘
                                 │               │  (hub<->spoke share accounting)
                 ┌───────────────▼──┐   ┌────────▼───────────┐
   borrowers ──▶ │  SPOKE (Bluechip,│   │  SPOKE (Ethena,    │  isolated risk markets ("reserves")
                 │  Gold, Forex,…)  │   │  Lombard BTC,…)    │  - emits Supply/Borrow/Repay/Liquidation
                 │  reserves by id  │   │  reserves by id    │  - own oracle + risk config per spoke
                 └──────────────────┘   └────────────────────┘
   TokenizationSpoke  → ERC-20 "aToken-like" wrapper per (hub-asset) for supply positions
   PositionManager (Config/Giver/Taker) → delegated actions; gateways for native ETH & EIP-712 sigs
   AccessManager (OpenZeppelin) → role-based governance (replaces ACLManager)
```

Key shifts vs V3:
- **Liquidity is pooled at the Hub, not the market.** Multiple spokes share one Hub's liquidity per asset → higher utilization, no per-market fragmentation. Three hub tiers: **Core**, **Plus**, **Prime**.
- **Markets are "reserves" inside a Spoke, identified by a `uint256` reserveId** — events index `reserveId`, **not** the asset address (big change for indexers).
- **Supply positions are tokenized by separate `TokenizationSpoke` ERC-20 wrappers** (one per hub-asset), instead of an aToken per reserve.
- **Governance via OpenZeppelin `AccessManager`** (role+delay based) instead of `ACLManager`.
- **Position Managers** (Config/Giver/Taker) generalize V3.4's position-manager concept.
- **All Hubs/Spokes/TokenizationSpokes are proxies** (1419-byte minimal/transparent proxies) behind shared implementations.

## 4. Proxies

Every Hub, Spoke, and TokenizationSpoke is a **proxy** (~1419 bytes — EIP-1967 transparent / minimal-proxy form) pointing at a shared implementation; the Configurators + AccessManager control upgrades. The Configurators, ConfigEngine, PositionManagers, gateways, and `LiquidationLogic` are the logic contracts. Read the EIP-1967 impl slot (`0x360894…bbc`) to resolve a Hub/Spoke's live implementation — do **not** hard-code impls (V4 is iterating quickly). `AccessManager` (OZ) governs roles with time-delays, replacing V3's `ACLManager`.

## 5. Detection invariants & gotchas

1. **Markets are `uint256 reserveId` (Spoke) / `assetId` (Hub), NOT asset addresses.** V4 Spoke `Supply`/`Borrow` index `reserveId`, not the token. To map reserveId→asset you must read `Spoke.getReserve(reserveId)` (or index `AddReserve`). This is the single biggest indexing change from V2/V3.
2. **Liquidity lives at the Hub; risk lives at the Spoke.** A supply emits a Spoke `Supply` **and** a Hub `UpdateAsset` (+ Hub share-accounting events) in the same tx. Attribute user activity to the Spoke event; attribute liquidity/rate to the Hub.
3. **Supply positions are ERC-20 TokenizationSpoke tokens,** not per-reserve aTokens. Track `Transfer` on the TokenizationSpoke for position movements.
4. **`getUserAccountData(address)` reuses selector `0xbf92857c`** from V2/V3 — disambiguate by the target Spoke address.
5. **`liquidationCall` here is `(uint256,uint256,address,uint256,bool)`** (reserveId-based) = selector `0xc2fa746c` — totally different from the V2/V3 `0x00a718a9`.
6. **Three hubs (Core/Plus/Prime) share asset liquidity but isolate risk tiers.** A single asset (e.g. USDC) has positions across multiple hubs/spokes; aggregate by underlying via the Hub `assetId`.
7. **Governance is OZ `AccessManager`,** not ACLManager — role checks and event shapes differ.
8. **Liquidations & repays carry a `PremiumDelta` tuple** `(int256,int256,uint256)`. `Repay` (`0xd765a026…`) and `LiquidationCall` (`0x2a1f12d9…`) are verified; `ReportDeficit`/`RefreshPremiumDebt` topic0s are computed (not yet log-matched). Remaining high-frequency Hub share-accounting topics are still unlabeled.
9. **Ethereum-only for now.** No V4 on Base/BNB/Avalanche/Arbitrum/Optimism/Polygon yet (as of 2026-05). V3 remains live on all of them — see [v3.md](v3.md).
