# Aave V3 — Topics, Selectors, Addresses (Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, Polygon)

**Status:** verified against Ethereum, Base, BNB, Avalanche, Arbitrum, Optimism, and Polygon mainnet RPC, and the canonical `bgd-labs/aave-address-book` + `aave-dao/aave-v3-origin` repos, on 2026-05-29.
**Scope:** the seven requested chains. Aave V3 also lives on the Ethereum **Lido** and **EtherFi** instances (separate Pools, same ABI) — both covered in §3. Topics + selectors are chain-agnostic; addresses are network-specific.

Aave V3 is a **fully proxied, upgradeable** lending protocol. Every core contract (Pool, PoolConfigurator, AaveOracle config, aTokens, debt tokens) sits behind a proxy controlled by the `PoolAddressesProvider` / `ACLManager` governance stack. The `Pool` emits all supply/borrow/repay/liquidation events; per-reserve `aToken` (interest-bearing, rebasing) and `variableDebtToken` (debt) contracts emit scaled `Mint`/`Burn`. **Stable-rate borrowing was removed in v3.2** — `stableDebtToken`s exist on legacy reserves but carry ~zero supply and the `borrowRate`/`interestRateMode` stable path reverts. Indices are **rays (27 decimals)**; account data (collateral/debt/borrows) is **8-decimal USD** except `healthFactor` which is **18-decimal**.

> **Version note (verified 2026-05-29):** the live Pool impls are **v3.4–v3.6** (v3.6 = Jan 2026, the latest V3 release). Relevant milestones: **v3.2** removed legacy stable rate; **v3.3** added bad-debt accounting (`DeficitCreated`/`DeficitCovered`, `eliminateReserveDeficit`); **v3.4** (Jul 2025) added `Multicall`, **Position Managers** (`approvePositionManager`, delegated collateral/eMode actions), and migrated **GHO to standard aToken/variableDebtToken** (minted by `GhoDirectMinter`); **v3.6** added Liquid eMode + renounce-allowance. **Aave V4** (hub-and-spoke) launched on Ethereum mainnet **2026-03-30** as the successor — V3 markets remain live. See [v4.md](v4.md).

> **Architecture caveat (verified on-chain 2026-05-29):** the live `Pool` implementations have **drifted from the `aave-address-book` `POOL_IMPL` snapshot** on every chain except Optimism. On the live v3.4+ impls, `liquidationCall` (selector `0x00a718a9`) is **not present in the implementation bytecode** (confirmed by raw byte-scan on ETH/ARB/POL — while `supply`/`borrow`/`multicall`/`flashLoanSimple` *are* present), yet `LiquidationCall` events fire normally and the canonical `aave-v3-origin` `Pool.sol` still declares it. This is consistent with the v3.4+ Pool being kept under the 24 KB EIP-170 limit by reaching liquidation logic through an extension/fallback path. The selector is still the canonical entrypoint. **Detect liquidations by the `LiquidationCall` event, not the selector.** See §11.2. Read the live EIP-1967 implementation slot, never trust a hard-coded impl address.

### 11.2 The v3.4 `liquidationCall` dispatcher gotcha (verified)

On the **live v3.4+ Pool impls** (ETH `0x728a138a…`, ARB `0xf05fd3cc…`, POL `0x6030db98…`), the 4-byte selector `0x00a718a9` (`liquidationCall`) is **absent from the implementation bytecode entirely** (raw byte-scan, not just the PUSH4 table) — yet `LiquidationCall` events fire and liquidations succeed (dozens of LiquidationCall logs in a 40k-block window). The newer v3.4 selectors **are** present in the same impls: `multicall` (`0xac9650d8`), `approvePositionManager` (`0xb8caa7c5`), `getReserveAToken` (`0xcff027d9`), plus `supply`/`borrow`/`flashLoanSimple`. The canonical `aave-dao/aave-v3-origin` `Pool.sol` **still declares** `liquidationCall` as a public function — so the deployed Pool reaches it through a path other than direct dispatch (consistent with keeping the v3.4+ impl, now carrying Multicall + Position Managers + Liquid eMode + deficit logic, under the 24 KB EIP-170 limit via an extension/fallback). **Practical takeaway for monitoring:** detect liquidations by the **`LiquidationCall` topic0** (`0xe413a321…`), never by the `0x00a718a9` selector — and most liquidations route through bots/adapters anyway.

## 12. Detection invariants & gotchas

1. **`Supply`, not `Deposit`.** V3 renamed the event (topic0 `0x2b627736…`). Filtering V2's `Deposit` topic on a V3 Pool returns nothing.
2. **`tx.from` ≠ borrower/supplier.** `onBehalfOf` (in the `Supply`/`Borrow` event and calldata) is the position owner; relayers, WrappedTokenGateway, and credit-delegation flows make `tx.from` the operator. Attribute to `onBehalfOf` / `user`.
3. **aTokens rebase every block.** Never store `balanceOf`; store `scaledBalanceOf` and reconstruct with the reserve `liquidityIndex` (ray). Same for debt with `variableBorrowIndex`.
4. **`Mint`/`Burn` topic0 is shared by aToken and debt token.** Disambiguate by emitting address — look up via `getReserveData(asset)` token addresses or the `ReserveInitialized` event.
5. **Detect liquidations by the `LiquidationCall` topic0, not the function selector** — `liquidationCall` (`0x00a718a9`) is not in the live v3.4 Pool impl dispatcher (§11.2), and most liquidations route through bots/adapters anyway.
6. **One `ReserveDataUpdated` per state change.** It carries fresh `liquidityIndex`/`variableBorrowIndex`/rates — use it to track indices at log granularity instead of polling `getReserveNormalizedIncome`.
7. **Stable rate is dead (removed in v3.2).** `interestRateMode = 1` reverts on new borrows; `SwapBorrowRateMode`/`RebalanceStableBorrowRate` are defunct. `stableDebtToken` balances are ~0.
8. **Account-data units differ.** `getUserAccountData` returns collateral/debt/borrows in **8-dec USD**, but `healthFactor` in **1e18**. HF `< 1e18` = liquidatable; `type(uint256).max` = no debt.
9. **The L2 quartet shares the Pool address.** `0x794a…14aD` is Avalanche AND Arbitrum AND Optimism AND Polygon. Condition every query on `(chainId, address)`.
10. **BNB stablecoins are 18-decimal.** USDC/USDT on BNB use 18 decimals; read `decimals()` per chain.
11. **Bridged vs native USDC coexist** on Arbitrum/Optimism/Polygon as separate reserves (bridged `symbol()` often still returns `"USDC"`). Distinguish by address.
12. **Flash-loan premium accrues to `Collector` via `mintToTreasury`,** not a direct transfer; `MintedToTreasury` marks it.
13. **GHO is special.** Minted by the `GhoDirectMinter` facilitator, not user-supplied; borrow rate is governance-set (not utilization-based). Since v3.4 it uses standard `aToken`/`variableDebtToken` (pre-v3.4 it had bespoke GHO tokens). Live on Ethereum, Arbitrum, Base, Avalanche — **not** BNB/Optimism/Polygon.
14. **The live Pool impl ≠ address-book `POOL_IMPL`** on six of seven chains (§11.1). Read the EIP-1967 slot for the true impl.
15. **L2Encoder / `L2Pool` calldata packing** (Arbitrum/Optimism/Base) — gas-optimised entrypoints encode args into a single `bytes32`; the decoded action still emits the standard §1.1 events, so log-level detection is unaffected.
