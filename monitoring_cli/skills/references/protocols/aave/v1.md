# Aave V1 — Topics, Selectors, Addresses (Ethereum only) — LEGACY

**Status:** event topics verified against live `eth_getLogs` on the Ethereum V1 `LendingPool` (`0x398eC7…d119`) in a 2020 active window; addresses from `bgd-labs/aave-address-book` (`AaveV1.sol`) + `aave/aave-protocol`. Verified 2026-05-29.
**Scope:** Aave V1 (the original "Aave Protocol", launched Jan 2020) was **only ever on Ethereum mainnet.** It is **fully deprecated** — superseded by V2 (Dec 2020) then V3. Liquidity migrated out years ago; activity today is negligible. Included for completeness / historical indexing. For live markets see [v2.md](v2.md) and [v3.md](v3.md).

Aave V1's architecture differs fundamentally from V2/V3: funds live in a separate **`LendingPoolCore`** contract; **`LendingPool`** is the logic/entrypoint; a **`LendingPoolDataProvider`** serves reads. The supply event is **`Deposit`** (5 args with a trailing `_timestamp`), and ETH is represented by the sentinel address **`0xEeeeeEeeeEeEeeEeEeEeeEEEeeeeEeeeeeeeEEeE`** (no WETH wrapping — V1 handled native ETH directly). aTokens use an **interest-redirection** model (not the scaled-balance model of V2/V3). All amounts carry an explicit `_timestamp` in events (V2/V3 dropped this).

## 4. Proxies

V1 uses an **`AdminUpgradeabilityProxy`** (OpenZeppelin transparent proxy, pre-EIP-1967 storage in early versions — implementation pointer stored at a custom slot). The `LendingPoolAddressesProvider` is itself an upgradeable proxy in V1, and it returns the (also-proxied) `LendingPool`, `LendingPoolCore`, and `LendingPoolDataProvider` impls.

| Contract | Pattern | Upgrade authority |
|----------|---------|-------------------|
| LendingPoolAddressesProvider | OZ ownable + proxy | Aave governance (frozen) |
| LendingPool / LendingPoolCore / DataProvider | `AdminUpgradeabilityProxy` behind the provider | provider `setLendingPoolImpl()` etc. |
| aTokens | upgradeable | provider/governance |

Because V1 predates EIP-1967, **do not assume the standard `0x360894…bbc` impl slot** — early V1 proxies use OZ's legacy `keccak256("org.zeppelinos.proxy.implementation")` (`0x7050c9e0…f8c3`) slot. Read both if resolving impls.

## 5. Detection invariants & gotchas

1. **V1 is its own world.** Different event signatures (all with `_timestamp`), different topic0s, a separate fund-holding `LendingPoolCore`, native-ETH handling via a sentinel address. Don't reuse V2/V3 decoders.
2. **`Deposit` topic `0xc12c57b1…` is unique to V1** — distinct from V2 `Deposit` and V3 `Supply`. Only `ReserveUsedAsCollateralEnabled/Disabled` share topic0 across all versions.
3. **No `Withdraw` event — it's `RedeemUnderlying`,** emitted when a user redeems aTokens.
4. **ETH is the sentinel `0xEeee…EEeE`,** not WETH. Filtering for WETH misses all native-ETH V1 activity.
5. **`liquidationCall` and `setUserUseReserveAsCollateral` selectors collide with V2/V3** (`0x00a718a9`, `0x5a3b74b9`) — disambiguate by the target Pool address.
6. **Account data is ETH-denominated** (like V2, unlike V3's USD).
7. **Effectively dead.** Use only for historical backfills; no monitoring value for current activity.
