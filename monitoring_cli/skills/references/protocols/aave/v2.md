# Aave V2 — Topics, Selectors, Addresses (Ethereum, Polygon, Avalanche)

**Status:** verified against Ethereum, Polygon, and Avalanche mainnet RPC (live + historic logs) and the canonical `bgd-labs/aave-address-book` + `aave/protocol-v2` repos on 2026-05-29.
**Scope:** Aave V2 was **only ever deployed on Ethereum, Polygon, and Avalanche.** It does **not** exist on Base, BNB, Arbitrum, or Optimism — those chains launched in the V3 era (use [v3.md](v3.md)). On Ethereum there are three V2 markets: **Main**, **AMM** (Uniswap/Balancer LP collateral), and **Arc** (permissioned, whitelisted). Topics + selectors are chain-agnostic; addresses are network-specific.

Aave V2 is **proxied and upgradeable** (like V3). The contract is the **`LendingPool`** (renamed to `Pool` in V3) and the supply event is **`Deposit`** (renamed `Supply` in V3). Each reserve has an `aToken` (rebasing), a `variableDebtToken`, and a `stableDebtToken` (stable rate was live in V2, unlike V3 where it's disabled). Indices are **rays (27 decimals)**; `getUserAccountData` returns values in **ETH (wei, 1e18)** — **not USD** (this is the key difference from V3, which uses 8-dec USD). **Aave V2 is deprecated and frozen** — most reserves are frozen (no new deposits/borrows) but withdrawals, repayments, and liquidations remain active.

## 2. Function signatures (chain-agnostic)

Selectors = `keccak256(sig)[0:4]`. All verified present in the deployed `LendingPool` implementation `0x02D84abD…` (Ethereum). Note **`borrow`/`repay`/`withdraw`/`flashLoan` share selectors with V3** (same signatures) — disambiguate by Pool address.

### 2.1 LendingPool — state-changing

| Selector | Signature | Notes |
|----------|-----------|-------|
| `0xe8eda9df` | `deposit(address asset, uint256 amount, address onBehalfOf, uint16 referralCode)` | Emits `Deposit`. **V3 renamed this `supply` (`0x617ba037`).** |
| `0x69328dec` | `withdraw(address asset, uint256 amount, address to)` → `uint256` | shared w/ V3. |
| `0xa415bcad` | `borrow(address asset, uint256 amount, uint256 interestRateMode, uint16 referralCode, address onBehalfOf)` | shared w/ V3. Mode 1=stable, 2=variable. |
| `0x573ade81` | `repay(address asset, uint256 amount, uint256 rateMode, address onBehalfOf)` → `uint256` | shared w/ V3. |
| `0x94ba89a2` | `swapBorrowRateMode(address asset, uint256 rateMode)` | Stable↔variable (functional in V2). |
| `0xcd112382` | `rebalanceStableBorrowRate(address asset, address user)` | |
| `0x5a3b74b9` | `setUserUseReserveAsCollateral(address asset, bool useAsCollateral)` | shared w/ V3. |
| `0x00a718a9` | `liquidationCall(address collateralAsset, address debtAsset, address user, uint256 debtToCover, bool receiveAToken)` | **Present in the V2 impl dispatcher** (unlike v3.4 — see v3.md §11). |
| `0xab9c4b5d` | `flashLoan(address receiverAddress, address[] assets, uint256[] amounts, uint256[] modes, address onBehalfOf, bytes params, uint16 referralCode)` | shared w/ V3. 0.09% fee in V2. |

## 7. Proxies

Every V2 core contract is an **upgradeable proxy** (`InitializableImmutableAdminUpgradeabilityProxy`, EIP-1967 impl slot). Same pattern as V3.

| Contract | Proxy? | Upgrade authority |
|----------|--------|-------------------|
| **LendingPool** | ✓ EIP-1967 | `LendingPoolAddressesProvider.setLendingPoolImpl()` |
| **LendingPoolConfigurator** | ✓ | `setLendingPoolConfiguratorImpl()` |
| **aToken / variable / stable debt token** | ✓ (admin = Configurator) | `LendingPoolConfigurator.updateAToken()` etc. |
| Collector / IncentivesController | ✓ | governance |
| **LendingPoolAddressesProvider** | ✗ plain ownable registry | the stable anchor |
| **AaveOracle / LendingRateOracle** | ✗ immutable | replaced via `setPriceOracle()` |
| **LendingPoolCollateralManager** | ✗ — logic contract `delegatecall`ed by the Pool during `liquidationCall` | swapped via provider |

EIP-1967 impl slot: `0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc` — read with `eth_getStorageAt(pool, slot)`. The address-book `POOL_IMPL` is a point-in-time snapshot; V2 is frozen so impls rarely change now, but verify the live slot before relying on an impl address.

## 8. Detection invariants & gotchas

1. **`Deposit`, not `Supply`.** V2's supply event is `Deposit` (topic `0xde685721…`) — V3 renamed it `Supply` (`0x2b627736…`). The selectors differ too (`deposit` `0xe8eda9df` vs `supply` `0x617ba037`).
2. **`getUserAccountData` returns ETH/wei, not USD.** V2 quotes collateral/debt/borrows in **ETH (1e18)** via the ETH-base oracle. V3 switched to 8-dec USD. Don't mix the two.
3. **Stable rate is real in V2.** `borrowRateMode`/`rateMode` = 1 (stable) or 2 (variable); `Swap`, `RebalanceStableBorrowRate`, and `stableDebtToken`s are all active. (V3 removed stable rate in v3.2.)
4. **aToken vs debt-token `Mint`/`Burn` have different topic0s in V2** (unlike V3). aToken `Mint` = `0x4c209b5f`, varDebt `Mint` = `0x2f00e3cd`. Match on `(topic0, emitting address)`.
5. **Several topics are shared with V3** — `Withdraw`, `LiquidationCall`, `ReserveDataUpdated`, collateral enabled/disabled, and selectors `borrow`/`repay`/`withdraw`/`flashLoan`/`liquidationCall`. Disambiguate by Pool address + chain, never by topic/selector alone.
6. **`liquidationCall` (`0x00a718a9`) IS in the V2 impl dispatcher** — unlike the v3.4 split impl. On V2 you can detect liquidations by either the selector or the `LiquidationCall` event.
7. **aTokens rebase.** Store `scaledBalanceOf`, reconstruct with `liquidityIndex` (ray). Same for debt with `variableBorrowIndex`.
8. **Liquidations `delegatecall` the LendingPoolCollateralManager** — the logic runs in the Pool's context, so `LiquidationCall` is emitted by the Pool address (not the collateral manager).
9. **V2 is frozen.** Most reserves accept no new deposits/borrows; expect `Withdraw`/`Repay`/`LiquidationCall` to dominate current activity, with `Deposit`/`Borrow` near zero (verified: ~0 in recent windows, thousands in 2021 windows).
10. **Flash-loan premium is 0.09%** in V2 (V3 default is 0.05%). The `FlashLoan` event carries `premium`.
11. **Avalanche assets are bridged `.e` tokens** (Aave Bridge), distinct from the native/Circle assets used by Avalanche V3.

## 10. Verification & sources

- **Event topic0:** computed locally (`keccak256(signature)`) and verified against live + historic `eth_getLogs` on the Ethereum LendingPool `0x7d27…c7A9`: `Deposit` (2221), `Borrow` (1683), `Repay`, `Swap` (4), `FlashLoan` (211), `LiquidationCall` (9) in a 2021 window (block 13.0M–13.05M, when V2 was active). aToken `Mint`/`Burn` verified on aUSDC `0xBcca…263C` (1819 / 354 logs); variableDebt `Mint`/`Burn` on `0x619b…D25b` (707 / 208 logs).
- **Function selectors:** `keccak256(sig)[0:4]`, verified present in the V2 LendingPool implementation `0x02D84abD…` dispatcher (incl. `liquidationCall`, which is **absent** from v3.4 V3 impls).
