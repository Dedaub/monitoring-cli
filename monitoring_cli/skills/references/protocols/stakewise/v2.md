# StakeWise V2 — Topics, Selectors, Addresses (Ethereum mainnet)

**Status:** verified against live RPC on `https://ethereum-rpc.publicnode.com` and the canonical `stakewise/contracts` repo on 2026-06-09. Every address below was existence-checked with `eth_getCode` (code present); proxy implementations and ProxyAdmin read live from the EIP-1967 slots; topic0s / selectors recomputed locally as keccak256 of the canonical signature and the high-volume ones confirmed against live `eth_getLogs`.
**Scope:** StakeWise **V2** (the pre-vault, pooled liquid-staking model). **Of the 7 target chains it is deployed on Ethereum mainnet (chain ID 1) ONLY** — `eth_getCode` returns `0x` for every V2 address on Base, BNB, Avalanche, Arbitrum, Optimism, and Polygon. V2 also runs on **Gnosis Chain (100)** with its own xDAI-side tokens (outside the 7; noted, not enumerated). Topic0 / selector values are chain-agnostic; addresses are Ethereum-specific.

**What V2 is.** A user deposits ETH into a single shared **Pool** and is minted **sETH2** (`StakedEthToken`) **1:1** with the deposit. sETH2 is a *principal* token — it does **not** grow. Staking rewards accrue instead into a **separate** rebasing token, **rETH2** (`RewardEthToken`): a 4-of-N **Oracles** committee votes a daily reward Merkle root, the total-rewards delta is pushed via `updateTotalRewards`, and rETH2 balances rebase up. **SWISE** is the governance/utility token. As of the V3 launch (Apr 2024) the Pool was migrated into the V3 **GenesisVault** (see [v3.md](v3.md)) and **new V2 deposits are closed** — but the V2 contracts remain live: rETH2 still rebases and the V2 Oracles still emit `RewardsUpdated`. For the vault/osETH model see [v3.md](v3.md).

> **Live-impl note.** Every V2 token/pool was upgraded to a `v3.0.0` implementation when V2 merged into V3 (Pool impl `0x481f28C0…`, sETH2 impl `0x82FE8C78…`, rETH2 impl `0x01d34aeE…`, Oracles redeployed to `0xF0C1670364…`). The **proxy addresses are permanent**; the impls below are the *current live* ones. Always resolve the impl from the EIP-1967 slot, never hardcode it.

## 6. Detection invariants & gotchas

1. **sETH2 ≠ ETH-with-yield.** sETH2 is the *principal* (1:1 with deposit) and **does not grow**; yield is the **separate** rETH2 token. To value a V2 position you must track **both** balances. (This is the opposite of Lido stETH, where one token rebases.)
2. **Correct sETH2 = `0xFe2e…089Ab0A043A`.** The widely-copied `0xFe2e…20Ae81B5d52` returns **no code** on-chain (`eth_getCode` = `0x`) — keying on it captures nothing.
3. **The V2 rebase is `RewardEthToken.RewardsUpdated`** (`0xb9c8611b…`), driven by `updateTotalRewards(int256)`. It still fires (V2 validators keep earning) even though deposits are closed. A **negative** period reward / non-zero `distributorPenalty` = slashing.
4. **V2 staking is closed** — `StakedWithPartner`/`StakedWithReferrer` on the Pool are dormant (migrated into the V3 GenesisVault Apr 2024). For *new* StakeWise staking activity, monitor [v3.md](v3.md), not this Pool.
5. **`OracleAdded`/`OracleRemoved` collide with the V3 Keeper** (identical topic0s). Disambiguate by emitter: V2 Oracles `0xF0C1670364…` vs V3 Keeper `0x6B581546…`.
6. **SWISE is a TransparentUpgradeableProxy** behind ProxyAdmin `0x3EB0175d…` — the same admin governs sETH2/rETH2/Pool. An `Upgraded` event from any of these is a protocol-level change.
7. **PoolEscrow is shared with V3.** The V3 GenesisVault references this same `0x2296e1…` escrow as `LegacyPoolEscrow`; ETH flowing out of it may be a V3 action, not V2.
8. **Single-chain.** There is no bridged sETH2/rETH2/SWISE on any L2 — any token by that name on Base/Arb/etc. is unrelated. The Gnosis V2 (sGNO/rGNO) is a distinct GNO-staking deployment, not a bridge.
