# GMX V2 — Topics, Selectors, Addresses (Arbitrum, Avalanche; absent on Ethereum, Base, BSC, Optimism, Polygon)

**Status:** verified against live RPC on Arbitrum One (`https://arbitrum-one-rpc.publicnode.com`) and Avalanche C-Chain (`https://avalanche-c-chain-rpc.publicnode.com`) on 2026-06-04, and against the canonical `gmx-io/gmx-synthetics` repo (source + `deployments/`). The three `EventLog`/`EventLog1`/`EventLog2` topic0s and the important `eventName` hashes were computed from the exact deployed ABI and **cross-checked against live emissions** on the Arbitrum and Avalanche `EventEmitter`. Absence on the other five target chains verified by `eth_getCode` returning `0x`.
**Scope:** GMX **V2** — the "gmx-synthetics" generation: synthetic isolated markets, GM liquidity pools, and the async order/keeper engine. Live on **Arbitrum One (42161)** and **Avalanche C-Chain (43114)** only among the seven target chains. This file does NOT cover GMX V1 (the GLP/Vault perpetuals system — a separate set of contracts at different addresses).

GMX V2 is an **isolated-market perpetuals + spot AMM**. Each market is its own `Market` (an indexToken + longToken + shortToken triple) backed by a fungible **GM** ERC-20 (a separate token per market; *not* fungible across markets). Trading is **asynchronous and two-step**: a user calls `ExchangeRouter.createOrder/createDeposit/createWithdrawal` (which records a request and escrows funds in a Vault), and a **keeper** later executes it against an oracle price in a second transaction. Sizes and prices are carried in **30-decimal USD** (`1e30 = $1.00`). Execution fees are paid in wrapped native (WETH on Arbitrum, WAVAX on Avalanche) up front. Pricing comes from Chainlink Data Streams / low-latency oracle feeds supplied by the keeper at execution time, not from on-chain push feeds.

**The single most important fact for monitoring GMX V2:** the protocol does **NOT emit typed, per-action events.** There is no `PositionIncrease(...)` event with its own topic0, no `OrderCreated(...)` event, etc. Instead, *almost all* protocol activity across *every* core contract is funneled through one shared **`EventEmitter`** contract that emits just three generic events:

- `EventLog(address msgSender, string eventName, string indexed eventNameHash, EventUtils.EventLogData eventData)`
- `EventLog1(address msgSender, string eventName, string indexed eventNameHash, bytes32 indexed topic1, EventUtils.EventLogData eventData)`
- `EventLog2(address msgSender, string eventName, string indexed eventNameHash, bytes32 indexed topic1, bytes32 indexed topic2, EventUtils.EventLogData eventData)`

The *actual* action name (`"PositionIncrease"`, `"OrderExecuted"`, `"SwapInfo"`, …) lives in the `eventName` **string** and, crucially, in the **indexed `eventNameHash` topic** which is `keccak256(bytes(eventName))`. All payload data (addresses, amounts, prices, keys) is packed into the `EventLogData` tuple. **So monitoring GMX V2 = subscribe to the `EventEmitter` address, filter on `topic0 ∈ {EventLog1, EventLog2, EventLog}`, then discriminate on `topics[1] = keccak256(eventName)`** (see the eventName→hash table below). The on-chain state lives in a single central **`DataStore`** key-value contract; positions/orders/markets are not their own contracts but rows in the DataStore keyed by `bytes32`.

## Topics (chain-agnostic — identical on Arbitrum and Avalanche, live-confirmed on both)

### The three generic EventEmitter events (this is what you watch)

`topic0 = keccak256(canonical signature)`. The `EventLogData` tuple is GMX's `EventUtils.EventLogData` expanded to its full canonical form: 7 item-categories (`address, uint256, int256, bool, bytes32, bytes, string`), each a struct of `( (string,X)[] items , (string,X[])[] arrayItems )`. Getting the parenthesisation right matters — the tuple is a **single** parameter `(cat0,cat1,…,cat6)`, not double-wrapped.

| topic0 | Event | Indexed topics layout |
|--------|-------|------|
| `0x137a44067c8961cd7e1d876f4754a5a3a75989b4552f1843fc69c3b372def160` | **`EventLog1`** — the workhorse; ~80% of all GMX V2 logs | `topics[1] = keccak256(eventName)`, `topics[2] = topic1` (an event-specific key, e.g. the market address or account) |
| `0x468a25a7ba624ceea6e540ad6f49171b52495b648417ae91bca21676d8a24dc5` | **`EventLog2`** | `topics[1] = keccak256(eventName)`, `topics[2] = topic1`, `topics[3] = topic2` |
| `0x7e3bde2ba7aca4a8499608ca57f3b0c1c1c93ace63ffd3741a9fab204146fc9a` | **`EventLog`** (no extra indexed key; rarely used) | `topics[1] = keccak256(eventName)` only |

Full canonical signature used for `EventLog1` (the others differ only by the leading indexed `bytes32`/`bytes32,bytes32`):
```
EventLog1(address,string,string,bytes32,(((string,address)[],(string,address[])[]),((string,uint256)[],(string,uint256[])[]),((string,int256)[],(string,int256[])[]),((string,bool)[],(string,bool[])[]),((string,bytes32)[],(string,bytes32[])[]),((string,bytes)[],(string,bytes[])[]),((string,string)[],(string,string[])[])))
```
`EventLog1` and `EventLog2` topic0s were matched byte-for-byte against live `eth_getLogs` on both chains' `EventEmitter`. `EventLog` (the keyless variant) is computed from the same deployed ABI but is essentially unused in practice (no occurrences observed in sampled windows).

## Proxies

**All GMX V2 contracts are immutable (non-proxied).** The EIP-1967 implementation slot (`0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc`) reads `0x0` on the `EventEmitter`, `DataStore`, and `ExchangeRouter` (verified on Arbitrum) — none are upgradeable proxies. Upgrades are performed by **deploying brand-new contracts and re-wiring them** through the `RoleStore` (granting the new handler/router the `CONTROLLER` role) and the `DataStore`. Consequences for monitoring:
- The `EventEmitter` and `DataStore` are long-lived anchors — they have not changed across upgrades, so pinning them is safe.
- The `ExchangeRouter`, `OrderHandler`, `DepositHandler`, etc. **do rotate** on upgrades; old versions keep their bytecode but lose their roles. Re-resolve handler/router addresses from the repo `deployments/` (or by which contract currently holds CONTROLLER in the RoleStore) periodically.

## Detection invariants & gotchas

- **There are no typed events. Watch the `EventEmitter`, filter by topic0 ∈ {`0x137a4406…` (EventLog1), `0x468a25a7…` (EventLog2), `0x7e3bde2b…` (EventLog)}, then discriminate on `topics[1] = keccak256(eventName)`.** Do not look for a `PositionIncrease(...)` topic0 — it does not exist. The eventName string is *also* in the (non-indexed) data, but you must filter on the indexed hash for efficiency.
- **`topics[1]` is the action; `topics[2]`/`topics[3]` are event-specific keys.** For `EventLog1`/`EventLog2` the extra indexed `topic1`/`topic2` are chosen per-event (commonly the market address, the account, or the order/position key) — they are NOT a fixed schema; decode the `eventName` first, then interpret. To pull the full payload you must ABI-decode the `EventLogData` tuple from the log `data` (key/value arrays keyed by human-readable strings like `"sizeDeltaUsd"`, `"market"`, `"account"`, `"executionPrice"`).
- **Decoding the payload:** `EventLogData` is 7 categories (`address,uint256,int256,bool,bytes32,bytes,string`), each `(items, arrayItems)` where each item is `(string key, value)`. Look up values by their string key. Numeric position/price fields are **30-decimal USD** (`/1e30` for dollars); token amounts use the token's own decimals.
- **Two-step async lifecycle.** A trade = `OrderCreated` (user tx, funds escrowed in a Vault) → later `OrderExecuted` *or* `OrderCancelled`/`OrderFrozen` (keeper tx). A position change emits `PositionIncrease`/`PositionDecrease` (and `PositionFeesCollected`, `SwapInfo`) *inside the keeper's execution tx*, not the user's. To attribute a fill to a user, join on the order key (`bytes32`) carried in `topics`/payload, not on `msg.sender`.
- **Liquidations** surface as `PositionDecrease` + `PositionLiquidated` from the `LiquidationHandler`/`OrderHandler`; ADL surfaces from the `AdlHandler`. All still go through the same `EventEmitter`.
- **GM tokens are per-market ERC-20s** (one token per market, not fungible across markets); deposits/withdrawals mint/burn them. Standard ERC-20 `Transfer` events on a GM token come from the market token itself, not the EventEmitter.
- **`multicall` is mandatory for creation.** A bare `createOrder` won't pull funds; integrators batch `sendWnt`(fee)+`sendTokens`(collateral)+`createOrder` in one `multicall`. Approvals go to the **`Router`**, not the ExchangeRouter.
- **Prices are keeper-supplied at execution** (Chainlink Data Streams / low-latency feeds), surfaced as `OraclePriceUpdate` logs from the `Oracle` — there is no standard Chainlink push-feed read in the hot path.
- **Addresses differ per chain and rotate on upgrade.** Pin `EventEmitter`+`DataStore` (stable); re-resolve handlers/routers. Never assume an address is portable Arbitrum↔Avalanche (it is not).
- **Only Arbitrum + Avalanche** among the seven target chains.
