# GMX V1 — Topics, Selectors, Addresses (Arbitrum + Avalanche; absent on Ethereum/Base/BNB/Optimism/Polygon)

**Status:** topic0/selectors computed with `cast keccak`/`cast sig` from the canonical `gmx-io/gmx-contracts` source, then re-verified on-chain with `cast` vs `publicnode` RPCs (2026-06). Arbitrum Vault confirmed by live reads (`router()`/`priceFeed()`/`usdg()` round-trip) and a live `eth_getLogs` whose `topic0` equals the computed Vault `Swap` topic exactly. **GMX V1 has been officially sunset** (trading/LP disabled) but the contracts remain deployed and immutable on-chain, so historical-event indexing and address attribution are still fully valid.
**Scope:** GMX **V1** — the original Vault-based perpetuals + spot DEX with single-asset-basket GLP liquidity, on **Arbitrum One (42161)** and **Avalanche C-Chain (43114)**. The newer GMX **V2** (gmx-synthetics: `ExchangeRouter`, market `GM` tokens, `OrderVault`, `EventEmitter`) is a completely separate codebase with different contracts/events/addresses — out of scope here.

GMX V1's core is the **Vault** (`0x489e…` on Arbitrum, `0x9ab2…` on Avalanche): a single pooled contract holding all whitelisted assets. Every spot swap, leverage open/close, liquidation, and GLP mint/redeem ultimately calls the Vault, which emits the canonical position/swap/fee events. Liquidity providers mint **GLP** (the basket LP token) against the Vault's assets via **GlpManager**; GLP holders are the counterparty to all traders. **USDG** is the internal USD accounting token minted/burned on `buyUSDG`/`sellUSDG`. Traders never call the Vault directly — they go through the **Router** (spot + plugin-gated leverage), the keeper-executed **PositionRouter** (two-step market orders: `Create…` then keeper `Execute…`/`Cancel…`), the **OrderBook** (limit/trigger orders), and **PositionManager** (keeper liquidations + order execution).

GMX V1 is **acutely price-feed dependent**: the Vault reads marks from **VaultPriceFeed**, which combines Chainlink with a keeper-pushed **FastPriceFeed** (the `secondaryPriceFeed`). All `IncreasePosition`/`DecreasePosition`/`LiquidatePosition`/`Swap` prices and every PositionRouter/OrderBook execution price flow from this oracle stack; a stale or manipulated FastPriceFeed is the central monitoring risk.

## Proxies

- **GMX V1 contracts are immutable, NOT proxies.** EIP-1967 implementation slot `0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc` reads `0x0` on the Vault and PositionRouter (confirmed on Arbitrum); they are plain non-upgradeable contracts. There is no `delegatecall`-to-implementation pattern and no `Upgraded(address)` event to watch.
- **Upgrade path is governance address-swapping, not in-place upgrade.** Each contract has a `gov` (a **Timelock** governor) that can re-point one contract at a *new* deployment of another (e.g. `vault.setPriceFeed(...)`, `router.addPlugin(...)`, GlpManager handler changes). To "upgrade" a component, gov deploys a fresh contract and rewires references — so the **set of canonical addresses can change over time via gov txs**, even though no individual address's bytecode ever changes. Vault `gov` = `0x5Cf5E6a4e430034836C9999a09781fC464615F42` (Arbitrum), `0x10194465FF07C67c00cB7f4bc1fa33A8F05c58e4` (Avalanche). Monitor these gov/Timelock addresses for `setPriceFeed`/`setGov`/handler-change calls if you need to track contract-set drift.
- Tokens (GMX/esGMX/GLP/bnGMX/USDG) and RewardTrackers are also non-proxy; treat as immutable unless re-verified.

## Detection invariants & gotchas

1. **The Vault is the single source of truth.** Every spot swap, leverage action, liquidation, and GLP mint/redeem emits from the Vault (`0x489e…` Arb / `0x9ab2…` Avax). Filter `(address = Vault, topic0)` — do NOT try to filter Vault events by account/token in topics: **no Vault event indexes any field**; all data is in `data`.
2. **A user "trade" spans multiple contracts in one tx.** A market long is: trader → PositionRouter `CreateIncreasePosition` (tx 1), then keeper → PositionRouter `executeIncreasePosition` which calls the Vault → `IncreasePosition` + `UpdatePosition` + `CollectMarginFees` + `IncreaseReservedAmount` (tx 2). Join `Create…`/`Execute…` by the position **key** (`bytes32`) and the indexed `account`.
3. **Two-step model → watch for `Cancel…`.** PositionRouter and OrderBook orders can be cancelled (or auto-cancelled by the keeper on price/slippage failure). `Create…` without a matching `Execute…` is NOT a fill; reconcile against `Cancel…`.
4. **GLP mint/redeem = `BuyUSDG`/`SellUSDG` at the Vault.** GLP supply changes route through RewardRouterV2 `mintAndStakeGlp`/`unstakeAndRedeemGlp` → GlpManager `addLiquidity`/`removeLiquidity` → Vault `buyUSDG`/`sellUSDG` (emits `BuyUSDG`/`SellUSDG` + pool/USDG deltas). Use these, not raw GLP `Transfer`s, for net LP flow (staked GLP also moves as fsGLP/fGLP tracker transfers).
5. **Position key derivation:** `key = keccak256(abi.encodePacked(account, collateralToken, indexToken, isLong))`. The same key appears in Vault `IncreasePosition`/`DecreasePosition`/`LiquidatePosition`/`UpdatePosition`/`ClosePosition` — use it to thread a position's full lifecycle.
6. **Price-feed dependency is the core risk.** All marks come from VaultPriceFeed + the keeper-pushed FastPriceFeed (`secondaryPriceFeed`). A stale/halted FastPriceFeed or a large Chainlink/Fast divergence is the highest-signal anomaly for GMX V1 (and was central to historical exploits). Monitor FastPriceFeed price-push cadence and VaultPriceFeed reads alongside trade events.
7. **Liquidations come from keepers (PositionManager), not the trader.** `LiquidatePosition` is emitted by the Vault when PositionManager/liquidator calls `vault.liquidatePosition(...)`; `markPrice` and `realisedPnl` (int256, signed) are in the event data.
8. **Fees emit separately.** `CollectMarginFees` (open/close/borrow fees on leverage) and `CollectSwapFees` (spot/GLP fees) carry `feeUsd`/`feeTokens`; the per-action `fee` field in `IncreasePosition`/`DecreasePosition`/`Swap.feeBasisPoints` is the trade-level view. Don't double-count.
9. **Avalanche tracker addresses overlap Arbitrum's at different roles** (see Addresses note). Attribute every tracker by `(chainId, role)` read from RewardRouterV2 — address alone is ambiguous across the two chains.
10. **Cross-chain address reuse false positives.** The Arbitrum and Avalanche Vault addresses each carry unrelated code on the *other* chain — they revert on GMX views. Confirm any Vault by a GMX-specific view read, never by code presence.
11. **GMX V1 is sunset.** Expect little/no new V1 trade activity; the contracts persist for historical reads. New GMX flow is V2 (gmx-synthetics) — a separate codebase; do not mix V1 and V2 topics/addresses.
