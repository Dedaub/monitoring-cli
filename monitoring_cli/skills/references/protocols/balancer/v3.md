# Balancer V3 — Topics, Selectors, Addresses (Ethereum, Arbitrum, Optimism, Base, Gnosis, Avalanche)

**Status:** event topic0 hashes and function selectors computed locally with `cast keccak` / `cast sig` from canonical signatures read out of `balancer/balancer-v3-monorepo` (verified — V3 `Swap` and `LiquidityAdded` match known anchors). Addresses from the official `balancer/balancer-deployments` registry, then **independently re-verified on-chain** via `cast` against `publicnode` RPCs (2026-05). See §9.
**Scope:** Balancer **V3** — the redesigned Vault and the Router/BatchRouter entry points. Balancer **V2** (the original Vault + pools) is documented in the sibling [`v2.md`](v2.md). The BAL/veBAL/gauge tokenomics stack is **version-agnostic and lives in [`v2.md`](v2.md) §1.3 / §4.2** — V3 pools earn BAL through the same gauges; it is not repeated here.

**V3 keeps the Vault-centric model** — a single `Vault` (`0xbA13…9bA9`, same on every chain where deployed) custodies all assets and emits the swap/liquidity events — but it is a ground-up rewrite: transient accounting (EIP-1153), a **Router/BatchRouter** front end (users call routers, which call the Vault), native **ERC-4626 buffers** (`Wrap`/`Unwrap`), and per-pool **hooks**. Monitor V3 by watching the one Vault address per chain.

> **V3 is NOT deployed on Polygon PoS or BSC.** It is on Ethereum, Arbitrum, Optimism, Base, Gnosis, and Avalanche (V3 Vault verified at 24,538 bytes of code on each; 0 bytes on Polygon and BSC).
>
> The `Swap` event also exists in V2 but hashes to a **different** `topic0` and carries different fields — never share a constant between versions. See [`v2.md`](v2.md).

## 7. Proxies

- **The V3 Vault uses a delegatecall split.** The `Vault` entry point (`0xbA13…9bA9`) holds the hot-path logic and **delegatecalls into `VaultExtension` and `VaultAdmin`** for everything else (their addresses are fixed at deploy — this is *not* an upgradeable proxy slot). For tracing: a single call to the Vault may execute code at the VaultExtension/VaultAdmin addresses via `DELEGATECALL`.
- **Hooks:** V3 pools may register per-pool external **hook** contracts called around swaps/liquidity ops. A swap can therefore touch a hook contract address in addition to the Vault and pool.
- **ERC-4626 buffers:** the Vault natively wraps/unwraps yield-bearing tokens (`Wrap`/`Unwrap`), so a swap path can include a buffer leg without a separate AMM pool.

See `references/proxies.md` for general proxy detection, and [`v2.md`](v2.md) §11 for the V2 Vault (a non-proxy singleton).

## 8. Detection invariants & gotchas

1. **Watch the Vault, not the pools.** `Swap`/`LiquidityAdded`/`LiquidityRemoved`/`Wrap`/`Unwrap` all emit from `0xbA13…9bA9`. Filtering by `(vault address, topic0)` captures every V3 pool at once.
2. **V2 vs V3 `Swap` are different events.** V2 `Swap(bytes32,address,address,uint256,uint256)` topic0 `0x2170c741…` (keyed by `poolId`); V3 `Swap(address,address,address,uint256,uint256,uint256,uint256)` topic0 `0x0874b2d5…` (keyed by `pool` address, carries `swapFeePercentage`+`swapFeeAmount`). Never reuse a constant between versions.
3. **V3 keys events by `pool` address, not `poolId`.** Unlike V2 (32-byte `poolId`), V3 events carry the plain pool contract address directly — no poolId decoding needed.
4. **`SwapFeePercentageChanged` differs from V2.** V3 = `(address,uint256)` topic0 `0x89d41522…`, emitted by the **Vault** with the pool as the indexed arg (V2 = `(uint256)` emitted by the pool — `v2.md`).
5. **The swap fee is in the event.** V3 surfaces `swapFeeAmount` directly in `Swap`; aggregate (protocol+creator) fees move via `AggregateSwapFeePercentageChanged`. No need to derive fees from pool params.
6. **`LiquidityAdded`/`Removed` cover all add/remove kinds.** The indexed `kind` enum distinguishes proportional / unbalanced / single-token — there is no separate event per shape. Don't assume "add" = proportional.
7. **Buffer legs (`Wrap`/`Unwrap`) are not AMM swaps.** A V3 trade routed through an ERC-4626 buffer emits `Wrap`/`Unwrap`, not `Swap`. A "swap volume" metric should not count buffer wraps as trades.
8. **No V3 on Polygon PoS or BSC.** Don't expect the V3 Vault there (`cast code` = 0 bytes on both); Polygon and BSC Balancer are V2-only (`v2.md`).
9. **Governance is shared with V2.** BAL/veBAL/gauge events/addresses are in [`v2.md`](v2.md) §1.3 / §4.2 — V3 pools earn BAL through the same gauge stack.
10. **Bytea hex literals must have an even digit count:** 40 hex chars for addresses, 64 for 32-byte topic values.
