## Cross-chain summary
- Live on **Avalanche, Arbitrum, BNB Chain, Ethereum**. Avax/Arb/BSC share one deterministic stack (factory `0x8e42…`); **Ethereum has its own** (`0xDC8d…`) down to a different LBPair impl. Re-`eth_getCode` before trusting an address on a new chain.
- The **same `0x8e42f2F4…` factory address is EMPTY on Base/OP/Polygon** — LB 2.1 is genuinely absent there (no decoy contract).

## Proxies
- **Factory / Router / Quoter: immutable** (no EIP-1967 impl slot).
- **LBPair instances: clone-with-immutable-args** (97-byte clone embedding `tokenX`,`tokenY`,`binStep` as immutable calldata; prefix `0x363d3d373d3d3d3d61…3d73<impl>5af4…`) delegating to the LBPair impl. A pair is **not individually upgradeable**, but the factory can repoint *future* pairs to a new impl → watch **`LBPairImplementationSet` `0x900d0e3d…`** on the factory.
- **`LimitOrderManager`: not EIP-1967** (15,595 B direct contract).

## Detection invariants & gotchas
1. **LB 2.1 and 2.2 are topic0-identical** for every shared event (Swap `0xad7d6f97…`, DepositedToBins `0x87f1f9dc…`, etc.). You **cannot** tell 2.1 from 2.2 by topic0 — only by which factory/pair address emitted it (or by the presence of `HooksParametersSet` `0x4d24d81a…`, which only 2.2 has).
2. **LB `Swap` ≠ V1/V2.0 `Swap`.** This `0xad7d6f97…` (uint24 id + bytes32 amounts) is distinct from Joe V1's `0xd78ad95f…` and LB 2.0's `0xc528cda9…`.
3. **`TransferBatch`/`ApprovalForAll` collide with ERC-1155 topic0s.** LB position moves appear in any ERC-1155 transfer monitor; disambiguate by the LBPair address. In 2.1/2.2 there is **no `TransferSingle`** — singles emit a 1-element `TransferBatch`.
4. **Amounts are packed `bytes32`**, not `uint256`: in `Swap`/`DepositedToBins`/`FlashLoan`, decode token-X from the low 128 bits and token-Y from the high 128 bits of each `bytes32`.
5. `getLBPairInformation` returns a 4-field struct — the **pair address is the 2nd word** (after `binStep`).
